## 2. UVP Statement aprobado (0.4.2)

> **Para los agentes de IA y el equipo humano del ecosistema VTT, Memory Service es el sistema centralizado de memoria que permite acceder al historial completo de conversaciones, timeline de actividad y reporte de costos de tokens en menos de 500 milisegundos, a diferencia de VTM legacy y el módulo 5F, que solo procesan una fuente de datos, no persisten contenido entre sesiones y no entregan contexto en tiempo real.**

**Estado:** ✅ APROBADO por PM — 2026-05-05

**Evaluación:**

La frase sigue correctamente la estructura UVP (Para quién → Qué es → Beneficio → A diferencia de → Gaps que resuelve). Los componentes son verificables:

| Componente | Verificación |
|------------|-------------|
| "Para los agentes de IA y el equipo humano" | Los 2 segmentos primarios de 0.4.4 |
| "sistema centralizado de memoria" | D-MEM-01: sistema independiente, BD propia |
| "historial completo de conversaciones" | SPEC §4.1 Conversation + §13 Storage |
| "timeline de actividad" | SPEC §8 `GET /agents/:id/timeline` |
| "reporte de costos de tokens" | SPEC §8 `GET /cost-report` |
| "en menos de 500 milisegundos" | D-MEM-07, D-MEM-37: `GET /context` <500ms |
| "VTM legacy y módulo 5F" | Las 2 soluciones existentes analizadas en 0.3.3 |
| "solo procesan una fuente" | VTM: solo CLI; 5F: solo CLI JSONL (0.3.3 §1-§2) |
| "no persisten contenido entre sesiones" | VTM: local; 5F: solo metadata (0.3.3 §2.3) |
| "no entregan contexto en tiempo real" | Ninguna tiene `GET /context` (0.3.3 §4 gap table) |

**Observación menor aceptada:** La frase "en menos de 500 milisegundos" podría leerse como si aplicara a todos los endpoints, cuando el SLA de <500ms es contractual solo para `GET /context`. Sin embargo, las notas del documento (0.4.2 §Notas) aclaran esto correctamente, y el UVP incluye el SLA porque es el diferenciador que convierte a Memory Service de repositorio pasivo a infraestructura runtime. La ambigüedad es aceptable en la frase UVP porque la clarificación existe en el documento.

## 5. Evaluación de hipótesis (0.4.5)

| Hipótesis | Estado | Comentario PM |
|-----------|--------|---------------|
| H1 — Contexto reduce inconsistencias (≥30% reducción) | ✅ Aceptada | Métrica clara, criterio ambicioso pero razonable. La mitigación si falla (ajustar Prompt Builder, no Memory Service) es correcta. |
| H2 — GET /context ≤500ms p95 @ 10 concurrent | ✅ Aceptada | Es la validación operativa del SLA contractual D-MEM-07. Los índices (SPEC §6.1) y el fail-fast están diseñados para cumplirlo. Criterio verificable en producción. |
| H3 — Cost-report habilita ≥2 optimizaciones en 1 mes | ✅ Aceptada | Conservador. Con los datos de PP-02 (corridas de $0.82-$2.53, cache ratios variables), encontrar 2 oportunidades debería ser directo. |
| H4 — Clasificación determinística ≥90% precisión | ✅ Aceptada | Muestra de 50 conversaciones es metodológicamente razonable para R1. La estrategia de fallback a LLM en R2 si no alcanza 90% tras 3 iteraciones es prudente. |
| H5 — Centralización reduce tiempo de review ≥20% | ⚠️ Aceptada con nota | La métrica depende de auto-reporte, que tiene sesgo inherente. Sugiero complementar con datos objetivos: número de búsquedas en Memory Service por review. No bloquea aceptación. |
| H6 — Import automático ≥95% tasa de éxito | ✅ Aceptada | El cleanup job (D-MEM-35: cada 5 min, max 3 retries) y la idempotencia (D-MEM-05/42) están diseñados para cumplir esto. 95% es un umbral razonable para R1. |

**Resultado:** 6/6 hipótesis aceptadas. 1 con nota menor (H5: complementar auto-reporte con datos objetivos). Todas son medibles y tienen criterios de éxito específicos.
