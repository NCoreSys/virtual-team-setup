## Objetivos SMART — Memory Service R1

### G1 — Contexto runtime en tiempo real

| Componente | Detalle |
|------------|---------|
| **Enunciado** | Entregar contexto histórico de agentes via `GET /context` con latencia p95 ≤500ms bajo carga de 10 requests concurrentes, con ≥100,000 conversaciones en base de datos. |
| **Specific** | Endpoint `GET /context` con 6 queries paralelas (`Promise.all`) y fail-fast (`Promise.race` 500ms). |
| **Measurable** | Latencia p95 medida con logging en middleware del endpoint. |
| **Achievable** | Arquitectura validada por AR Review §3.3. Índices GIN y parciales diseñados en SPEC §6.1. |
| **Relevant** | North Star Metric del producto. Contrato con Runtime v1.1 y Prompt Builder v1.3 (ADDENDUM §4). Resuelve PP-01 y PP-07. |
| **Time-bound** | Operativo al deploy de R1. |
| **Baseline** | Sin servicio de contexto. Los agentes arrancan sin historial (0ms porque no existe). |
| **Target** | p95 ≤ 500ms @ 10 concurrent @ 100K conversaciones |
| **Hipótesis** | H2 |

---

### G2 — Importación automática y confiable

| Componente | Detalle |
|------------|---------|
| **Enunciado** | Importar automáticamente ≥95% de las conversaciones generadas por Runtime v1.1 via `POST /import`, con idempotencia garantizada y recovery automático de fallos. |
| **Specific** | `POST /import` con constraint `@@unique([sourceId, externalSessionId])`. Cleanup job cada 5 minutos, max 3 retries. |
| **Measurable** | (Conversaciones IMPORTED / Rondas completadas en Runtime) × 100. |
| **Achievable** | Idempotencia por constraint de BD (D-MEM-05/42). Recovery por cron job (D-MEM-35). |
| **Relevant** | Sin import confiable, `GET /context` no tiene datos que servir. Es upstream del North Star. Resuelve PP-03 y PP-07. |
| **Time-bound** | Validado en las primeras 2 semanas post-deploy R1. |
| **Baseline** | 0% — no existe import automático. Runtime descarta conversaciones post-corrida. |
| **Target** | ≥95% tasa de import exitoso |
| **Hipótesis** | H6 |

---

### G3 — Visibilidad de costos por agente y proyecto

| Componente | Detalle |
|------------|---------|
| **Enunciado** | Proveer cost-reports que permitan al PM identificar al menos 2 oportunidades de optimización de costos de tokens en el primer mes de operación. |
| **Specific** | `GET /projects/:id/cost-report` y `GET /agents/:id/cost-report` con desglose por modelo y tipo de token (input, output, cache_read, cache_creation). |
| **Measurable** | Número de acciones de optimización documentadas por el PM basadas en datos del cost-report. |
| **Achievable** | ConversationUsage (SPEC §4.1) persiste tokens + USD con 6 decimales. Campo `primaryAgentRole` desnormalizado (D-MEM-38) habilita breakdown sin consultar VTT. |
| **Relevant** | Resuelve PP-02 y PP-05. Datos reales muestran corridas de $0.82-$2.53 USD con cache ratios variables — hay margen de optimización. |
| **Time-bound** | ≥2 optimizaciones en el primer mes post-deploy. |
| **Baseline** | 0 optimizaciones — no hay datos consolidados para tomar decisiones. |
| **Target** | ≥2 acciones de optimización documentadas |
| **Hipótesis** | H3 |

---

### G4 — Clasificación determinística precisa

| Componente | Detalle |
|------------|---------|
| **Enunciado** | Clasificar automáticamente las conversaciones importadas por topics y workTypes con ≥90% de precisión, usando reglas determinísticas sin ML. |
| **Specific** | Clasificador por reglas (D-MEM-08) con catálogos en BD (D-MEM-20). Topics: ARCHITECTURE, CODE_REVIEW, TESTING, etc. WorkTypes: TASK_EXECUTION, AGENT_REVIEW, AGENT_CLARIFICATION. |
| **Measurable** | Precisión = (correctamente clasificadas / muestra) × 100, evaluada sobre muestra aleatoria de 50 conversaciones. |
| **Achievable** | Las reglas operan sobre contenido estructurado (no texto libre). Los workTypes son inferibles desde metadata (sourceCode, número de participantes). |
| **Relevant** | Resuelve PP-04 y PP-06. Habilita búsqueda por tema y auditoría de decisiones. |
| **Time-bound** | Evaluado en las primeras 4 semanas post-deploy. Hasta 3 iteraciones de ajuste si no alcanza 90%. |
| **Baseline** | 0% — no existe clasificación automática en producción. |
| **Target** | ≥90% precisión en muestra de 50 |
| **Hipótesis** | H4 |

---

### G5 — Disponibilidad del servicio

| Componente | Detalle |
|------------|---------|
| **Enunciado** | Mantener Memory Service operativo con uptime ≥99.5% medido mensualmente, incluyendo API (puerto 3002) y UI (puerto 3003). |
| **Specific** | Health check via `GET /health` (liveness probe). Docker-compose con restart policies. PostgreSQL con connection_limit=20. |
| **Measurable** | Uptime = (minutos_operativos / minutos_totales_mes) × 100. Medido con health check periódico. |
| **Achievable** | VM Hetzner dedicada. Docker-compose con auto-restart. Sin dependencias externas en R1 (sin MinIO, sin servicios cloud). |
| **Relevant** | Memory Service es infraestructura runtime — si cae, Runtime v1.1 pierde la capacidad de persistir conversaciones y Prompt Builder pierde contexto. |
| **Time-bound** | Cada mes calendario post-deploy. |
| **Baseline** | N/A — servicio nuevo. |
| **Target** | ≥99.5% uptime mensual (≤3.6h downtime/mes) |
| **Hipótesis** | N/A (requisito operativo, no hipótesis de valor) |
