## KPIs de Éxito — Memory Service R1

### KPI-1 · Latencia de Contexto Runtime (OBLIGATORIO)

| Campo | Detalle |
|-------|---------|
| **Nombre** | Context Delivery Latency p95 |
| **Definición** | Percentil 95 del tiempo de respuesta de `GET /context`, medido desde recepción de la request HTTP hasta envío de la response completa. |
| **Target** | ≤ 500ms |
| **Alerta** | > 400ms (80% del umbral) |
| **Violación** | > 500ms |
| **Condiciones** | Con ≥10 requests concurrentes, ≥100K conversaciones en BD |
| **Cómo medirlo** | Middleware de logging en el endpoint que registre `{timestamp, latency_ms, status_code, conversation_count}`. Cálculo p95 sobre ventana móvil de 7 días. |
| **Frecuencia** | Continua (cada request). Reporte semanal. |
| **Responsable** | TL (instrumentación), PM (monitoreo del KPI) |
| **Referencia** | D-MEM-07, D-MEM-37, North Star Metric (1.1.5), Goal G1 |

---

### KPI-2 · Tasa de Import Exitoso (OBLIGATORIO)

| Campo | Detalle |
|-------|---------|
| **Nombre** | Import Success Rate |
| **Definición** | Porcentaje de conversaciones importadas con status final IMPORTED respecto al total de rondas completadas en Runtime v1.1. |
| **Target** | ≥ 95% |
| **Alerta** | < 97% |
| **Violación** | < 95% |
| **Fórmula** | `(COUNT(status=IMPORTED) / total_rondas_runtime) × 100` |
| **Cómo medirlo** | Query sobre tabla `conversations` agrupada por status. Correlacionar con logs de Runtime v1.1 para total de rondas. Las conversaciones en ERROR con `retryCount ≥ 3` se cuentan como fallos permanentes. |
| **Frecuencia** | Diaria. Reporte semanal. |
| **Responsable** | BE (instrumentación), QA (validación), PM (monitoreo) |
| **Referencia** | D-MEM-05, D-MEM-42, D-MEM-35 (cleanup job), Goal G2, Hipótesis H6 |

---

### KPI-3 · Precisión de Clasificación (OBLIGATORIO)

| Campo | Detalle |
|-------|---------|
| **Nombre** | Classification Accuracy |
| **Definición** | Porcentaje de conversaciones correctamente clasificadas por topics y workTypes, evaluado contra revisión manual de una muestra aleatoria. |
| **Target** | ≥ 90% |
| **Alerta** | < 92% |
| **Violación** | < 90% |
| **Fórmula** | `(correctamente_clasificadas / tamaño_muestra) × 100` |
| **Cómo medirlo** | Muestreo aleatorio de 50 conversaciones por evaluación. Un revisor (SA o TL) verifica manualmente si los topics y workTypes asignados son correctos. Evaluar topics y workTypes por separado y en conjunto. |
| **Frecuencia** | Cada 4 semanas. Hasta 3 iteraciones de ajuste de reglas si no alcanza target. |
| **Responsable** | SA (revisión de muestra), BE (ajuste de reglas), PM (decisión go/no-go) |
| **Referencia** | D-MEM-08, D-MEM-20, D-MEM-21, Goal G4, Hipótesis H4 |

---

### KPI-4 · Disponibilidad del Servicio (OBLIGATORIO)

| Campo | Detalle |
|-------|---------|
| **Nombre** | Service Uptime |
| **Definición** | Porcentaje de tiempo que Memory Service (API + UI) responde exitosamente a health checks durante un mes calendario. |
| **Target** | ≥ 99.5% |
| **Alerta** | < 99.7% |
| **Violación** | < 99.5% (>3.6h downtime/mes) |
| **Fórmula** | `(checks_exitosos / checks_totales) × 100` |
| **Cómo medirlo** | Health check via `GET /health` cada 60 segundos. Registrar resultado (200 OK / error / timeout). Calcular uptime al cierre de cada mes. |
| **Frecuencia** | Continua (cada 60s). Reporte mensual. |
| **Responsable** | DO (instrumentación y alertas), PM (monitoreo) |
| **Referencia** | KICKOFF_ACTA §3 (DO: monitoreo), Goal G5 |

---

### KPI-5 · Cobertura de Fuentes Importadas

| Campo | Detalle |
|-------|---------|
| **Nombre** | Source Coverage |
| **Definición** | Número de fuentes de conversación con al menos 1 import exitoso en las primeras 4 semanas post-deploy. |
| **Target** | 5 de 5 fuentes (CLAUDE_SDK, CLAUDE_CLI, CLAUDE_WEB, CHATGPT, VTT_CHANNEL) |
| **Fórmula** | `COUNT(DISTINCT sourceCode) WHERE status = 'IMPORTED'` |
| **Cómo medirlo** | Query directa sobre tabla `conversations`. |
| **Frecuencia** | Semanal durante las primeras 4 semanas. |
| **Responsable** | BE (soporte de adapters), QA (test por fuente), PM (tracking) |
| **Referencia** | D-MEM-14, PROJECT_MEMORY §1, Goal G2 |

---

### KPI-6 · Adopción de Cost-Report

| Campo | Detalle |
|-------|---------|
| **Nombre** | Cost Report Utilization |
| **Definición** | Número de acciones de optimización documentadas por el PM basadas en datos del cost-report en el primer mes de operación. |
| **Target** | ≥ 2 acciones documentadas |
| **Cómo medirlo** | El PM registra cada acción de optimización (cambio de modelo, ajuste de prompt, reducción de contexto) con referencia al dato del cost-report que la motivó. |
| **Frecuencia** | Mensual. |
| **Responsable** | PM (consumo y documentación), TL (implementación de optimizaciones) |
| **Referencia** | Goal G3, Hipótesis H3 |
