## North Star Metric

**Latencia p95 de `GET /context` ≤ 500ms con ≥100,000 conversaciones en base de datos.**

---

## Definición

La latencia p95 mide el tiempo de respuesta del endpoint `GET /context` en el percentil 95 — es decir, el 95% de las requests responden en ese tiempo o menos. Se mide desde que Memory Service recibe la request HTTP hasta que envía la response completa, incluyendo las 6 queries paralelas internas (`Promise.all`) y el assembly del response body.

El umbral de 100,000 conversaciones establece la escala mínima a la cual el SLA debe sostenerse. Con el ecosistema VTT operando 3+ proyectos con 7-10 agentes cada uno, las conversaciones acumuladas crecen continuamente. El North Star no es "funcionar rápido con 100 conversaciones" — es "funcionar rápido cuando la base de datos tiene la escala real de producción".

---

## Cómo medirla

| Componente | Implementación |
|------------|---------------|
| **Instrumentación** | Middleware de logging en el endpoint `GET /context` que registre timestamp_inicio, timestamp_fin y latencia_ms por request |
| **Cálculo p95** | Ordenar latencias del período de medición, tomar el valor en el percentil 95 |
| **Período** | Ventana móvil de 7 días (para suavizar picos anómalos) |
| **Carga baseline** | 10 requests concurrentes (consistente con SPEC v1.9 §11 y criterio de éxito del Kickoff) |
| **Volumen de datos** | Medido cuando `COUNT(conversations)` ≥ 100,000 en `memory_service_db` |
| **Alerta** | Si p95 > 400ms (80% del umbral), se activa alerta de degradación antes de violación |

**Fórmula:**

```
North Star = p95(latency_ms) de GET /context
             en ventana de 7 días
             con COUNT(conversations) ≥ 100,000
             con ≥ 10 requests concurrentes

Objetivo: ≤ 500ms
Alerta: > 400ms
Violación: > 500ms
```

---

## Por qué esta es la North Star

### Es el punto de convergencia de todo el valor

`GET /context` es el endpoint donde converge la cadena de valor completa de Memory Service:

```
Runtime v1.1 (POST /import) → Memory Service (persiste + clasifica) → GET /context → Prompt Builder v1.3 → Agente con contexto
```

Si `GET /context` responde en <500ms, significa que:
- Las conversaciones se importaron correctamente (la BD tiene datos)
- La clasificación por reglas funciona (los filtros de topics/workTypes operan)
- Los índices están optimizados (GIN en platformRefs, parciales para queries frecuentes)
- El storage no está degradado (las queries paralelas completan a tiempo)
- El fail-fast funciona (Promise.race con timeout de 500ms)

Si `GET /context` no responde en <500ms, toda la propuesta de valor se degrada: Prompt Builder ensambla prompts sin contexto, los agentes vuelven al arranque en frío, y Memory Service se reduce a un repositorio pasivo.

### Es contractual, no aspiracional

El SLA de <500ms no es un objetivo interno — es un contrato con Runtime v1.1 y Prompt Builder v1.3, documentado en el ADDENDUM v1.1 §4 y respaldado por las decisiones D-MEM-07 y D-MEM-37. El fail-fast (`MEM-ERR-504`) existe porque los consumidores necesitan una garantía: "o me das contexto en 500ms, o me dices que no puedes, y yo ensamblo sin contexto". El North Star mide si Memory Service cumple ese contrato.

### Es medible en R1

No requiere infraestructura adicional. Se mide con logging estándar en el endpoint. Los datos se generan naturalmente cada vez que Prompt Builder consulta contexto. El volumen de 100,000 conversaciones se alcanzará conforme los proyectos operen con Memory Service activo.

### Es una métrica, no un dashboard

A diferencia de "tasa de import exitoso" o "cost savings" que son métricas complementarias, la latencia de `GET /context` es la única que captura el contrato de valor del producto en un solo número. Si tuvieras que mirar un solo indicador para saber si Memory Service está funcionando, este es el que miras.

---

## Alternativas evaluadas y descartadas

| Candidata | Por qué no es la North Star |
|-----------|----------------------------|
| Tasa de import exitoso (≥95%) | Mide confiabilidad de ingesta, no entrega de valor. Puedes importar el 100% y que `GET /context` sea lento. Es métrica operativa (H6), no North Star. |
| Conversaciones totales importadas | Vanity metric. Más conversaciones no significa más valor si no se entregan como contexto. |
| Reducción de inconsistencias en entregables (H1) | Depende de Prompt Builder y de la calidad de los prompts, no solo de Memory Service. Atribución difusa. |
| Cost savings identificados (H3) | Depende del PM y de que actúe sobre los datos. Memory Service habilita la visibilidad, no la acción. |
| Precisión de clasificación (H4) | Métrica de calidad interna. No capta el valor para el usuario final (contexto entregado). |

---

## Relación con las hipótesis de valor (0.4.5)

| Hipótesis | Relación con North Star |
|-----------|------------------------|
| H1 (contexto reduce inconsistencias) | Si North Star se cumple, H1 es testeable — los agentes reciben contexto |
| **H2 (GET /context ≤500ms p95)** | **Es la North Star directamente** |
| H3 (cost-report habilita optimización) | Independiente — cost-report tiene su propio SLA |
| H4 (clasificación ≥90%) | Contribuye — clasificación precisa mejora la relevancia del contexto entregado |
| H5 (reducción tiempo de review) | Downstream — si el contexto se entrega rápido, los reviews tienen más información |
| H6 (import ≥95%) | Upstream — sin imports exitosos no hay datos para servir en GET /context |

La North Star es H2 elevada a métrica de producto. H6 (import) y H4 (clasificación) son inputs. H1, H3 y H5 son outcomes downstream.
