## OKRs — Memory Service R1

### Objective 1 — Los agentes del ecosistema VTT operan con memoria histórica

**Alineación:** Vision Statement (1.1.1), Pain Points PP-01 y PP-07, Hipótesis H1 y H2.

Este objective captura el valor central del producto: los agentes dejan de arrancar en frío. La cadena completa funciona: Runtime importa → Memory Service persiste → Prompt Builder consulta → el agente recibe contexto.

| KR | Enunciado | Métrica | Target | Medición |
|----|-----------|---------|--------|----------|
| KR-1.1 | `GET /context` responde en ≤500ms p95 bajo carga de producción | Latencia p95 en ventana de 7 días | ≤ 500ms @ 10 concurrent @ 100K conversaciones | Logging en middleware del endpoint. KPI-1. |
| KR-1.2 | ≥95% de las conversaciones generadas por Runtime v1.1 se importan exitosamente | Tasa import = IMPORTED / rondas Runtime | ≥ 95% | Query status IMPORTED vs logs Runtime. KPI-2. |
| KR-1.3 | Las 5 fuentes de conversación tienen al menos 1 import exitoso en las primeras 4 semanas | COUNT(DISTINCT sourceCode) con status IMPORTED | 5 de 5 | Query directa en BD. KPI-5. |

---

### Objective 2 — El equipo tiene visibilidad completa sobre costos y actividad de agentes

**Alineación:** Vision Statement (1.1.1), Pain Points PP-02 y PP-05, Hipótesis H3.

Este objective captura el segundo eje de valor: el PM y el TL pueden tomar decisiones informadas sobre costos y eficiencia porque los datos existen y son consultables.

| KR | Enunciado | Métrica | Target | Medición |
|----|-----------|---------|--------|----------|
| KR-2.1 | Los endpoints de cost-report devuelven datos reales de tokens y USD por agente y proyecto | Endpoints funcionales con datos | `GET /projects/:id/cost-report` y `GET /agents/:id/cost-report` devuelven datos no vacíos | Test de integración + verificación manual. |
| KR-2.2 | El PM identifica ≥2 oportunidades de optimización de costos basándose en datos del cost-report | Acciones documentadas por PM | ≥ 2 en primer mes | Registro del PM con referencia al dato. KPI-6. |
| KR-2.3 | El dashboard muestra stats globales actualizados (conversaciones por estado, por fuente, totales) | `GET /dashboard/stats` operativo | Response con datos reales, sin placeholders | Test de integración. |

---

### Objective 3 — Las conversaciones se clasifican automáticamente y el servicio es confiable

**Alineación:** Vision Statement (1.1.1), Pain Points PP-04 y PP-06, Hipótesis H4.

Este objective captura el tercer eje: la clasificación habilita auditoría y búsqueda, y el servicio es estable como infraestructura.

| KR | Enunciado | Métrica | Target | Medición |
|----|-----------|---------|--------|----------|
| KR-3.1 | La clasificación determinística asigna topics y workTypes con ≥90% de precisión | Precisión en muestra de 50 conversaciones | ≥ 90% | Revisión manual por SA/TL. KPI-3. |
| KR-3.2 | Memory Service mantiene uptime ≥99.5% mensual | (checks OK / checks totales) × 100 | ≥ 99.5% | Health check cada 60s via GET /health. KPI-4. |
| KR-3.3 | El cleanup job recupera conversaciones stuck sin intervención manual | Conversaciones PENDING/PROCESSING >15min que pasan a IMPORTED o ERROR | 100% procesadas (retry o error explícito, nunca stuck indefinidamente) | Query: COUNT WHERE status IN (PENDING, PROCESSING) AND updatedAt < NOW() - 15min = 0. |
