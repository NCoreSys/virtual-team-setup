## Resumen

Este documento describe el proceso end-to-end para llevar un proyecto desde input a Gantt funcional en VTT. **Sigue los 9 pasos del estándar `09_PROCESO_CIERRE_PM_HANDOFF_PJM.md`** y agrega las lecciones operativas validadas con Memory Service R1 (post-carga: corrección de dependencias, generación de HO por fase, fix de Gantt).

> **Principio del estándar:** el PM no se considera "cerrado" hasta que el PJM puede ejecutar un script y cargar 100% del proyecto a VTT en una sola pasada, sin repetir análisis.

## Reglas de Determinación de Tareas y Dependencias

Estas reglas son las **heurísticas operativas** que convierten la SPEC técnica en tareas concretas y dependencias correctas. Son el "cómo decidimos" detrás de cada tarea y cada flecha del Gantt.

---

### A. Reglas para Determinar Tareas

#### A.1 — Atomicidad de la tarea

| Regla | Aplicación |
|-------|-----------|
| **1 tarea = 1 output concreto** | La tarea debe terminar con un entregable verificable (código, doc, config aplicada) |
| **Horas por tarea: 2–8h** | Si una tarea requiere >8h, se divide. Menos de 2h se fusiona con otra del mismo delivery |
| **1 tarea = 1 rol responsable** | No hay tareas compartidas entre roles. Si dos roles colaboran, son 2 tareas con una dependencia |
| **1 tarea puede producir N deliverables SDLC** | Si los deliverables son del mismo tema. Ej: "Problem Definition" (3h) produce 4 docs relacionados |

#### A.2 — Asignación de rol

| Tipo de trabajo | Rol por defecto |
|-----------------|-----------------|
| Infraestructura, Docker, CI/CD, deploy | **DO** (DevOps) |
| Schema BD, migraciones, seeds, queries | **DB** (Database) |
| API endpoints, lógica de negocio backend | **BE** (Backend) |
| UI components, React, integración API desde UI | **FE** (Frontend) |
| Wireframes, mockups, design system | **UX** / **DL** |
| Tests (unit, integration, E2E, performance) | **QA** |
| Decisiones arquitectónicas, ADRs, patrones técnicos | **AR** (Architect) |
| Análisis funcional, FR/NFR, user stories, flows | **SA** (Solution Analyst) |
| Gestión de proyecto, scope, timeline, coordinación | **PM** / **PJM** |
| Revisión técnica, code review, aprobación de arquitectura | **TL** (Tech Lead) |

#### A.3 — Categoría VTT (valores válidos)

```
development · design · testing · documentation · review · bugfix · deployment
```

**Mapeo de términos comunes (que NO son categorías válidas en VTT):**

| Término común | Mapear a |
|---------------|----------|
| `chore` (infra, Docker, CI) | `deployment` |
| `chore` (docs, README, templates) | `documentation` |
| `chore` (setup dev env) | `development` |
| `research` / `spike` | `documentation` |
| `analysis` | `documentation` |
| `planning` | `documentation` |

#### A.4 — Complejidad

| Complejidad | Criterio |
|-------------|----------|
| **LOW** | Patrón estándar, spec clara, sin integraciones nuevas. Ej: CRUD básico, docs con template |
| **MEDIUM** | Integración con 1–2 servicios existentes, lógica de negocio no trivial pero patrón conocido |
| **HIGH** | Integraciones múltiples, patrón nuevo, SLA crítico, alta concurrencia, trabajo de investigación |

> ⚠️ Usar MAYÚSCULAS siempre. `medium` minúscula es rechazado por VTT.

#### A.5 — Prioridad

| Prioridad | Cuándo aplica |
|-----------|---------------|
| **C (Critical)** | Bloquea el release. Ej: Production Deploy, Design Handoff Final, Performance SLA validation |
| **H (High)** | Importante pero no bloquea el release. Ej: Integration tests, monitoring setup |
| **M (Medium)** | Por defecto para la mayoría de tareas |
| **L (Low)** | Nice-to-have, polish, tareas de documentación secundaria |

**Regla:** Por defecto `M`. Solo subir a `H` o `C` si la tarea es un **gate crítico** en la cadena de dependencias del proyecto.

#### A.6 — Descripción de la tarea

| Regla | Valor |
|-------|-------|
| Longitud mínima | 150 chars |
| Longitud recomendada | 150–300 chars |
| Longitud máxima | **2000 chars (límite VTT)** |
| Contenido | QUÉ hacer + en qué archivo/lugar + output esperado |
| Evitar | Copiar la descripción literal del catálogo sin contexto del proyecto |

### B. Reglas para Determinar Dependencias

Hay **tres niveles** de dependencias. Cada uno tiene su propia regla:

```
Nivel 1: Dependencias CRÍTICAS (inter-fase, gates del Gantt)
Nivel 2: Dependencias INTRA-FASE (entre deliveries de la misma fase)
Nivel 3: Dependencias INTRA-SPRINT (dentro del mismo delivery)
```

---

#### B.1 — Dependencias Críticas (Nivel 1: Inter-Fase)

Son los "gates" que conectan una fase con la siguiente en el Gantt. Típicamente **10–15 por proyecto**.

##### B.1.1 — Fases consecutivas

**Regla:** `ÚLTIMA_TAREA_FASE_N → PRIMERA_TAREA_FASE_N+1`

```
Fase N: ... → Tarea X (última)
                  │
                  └──→ Tarea Y (primera de Fase N+1)
```

**Ejemplo Memory Service R1:**
- `MEM-017` (última Planning) → `MEM-018` (primera Analysis)
- `MEM-103` (última Testing) → `MEM-104` (primera Deploy)

##### B.1.2 — Fases paralelas (divergencia)

Cuando una fase alimenta **dos fases simultáneas** (ej. Analysis → Design UX/UI + Design Technical):

**Regla:** `ÚLTIMA_TAREA_FASE_N → PRIMERA_TAREA_DE_CADA_FASE_PARALELA`

```
                   ┌──→ Primera tarea Design UX/UI
                   │
Última Analysis ───┤
                   │
                   └──→ Primera tarea Design Technical
```

##### B.1.3 — Fases paralelas que convergen en una común

Cuando **dos fases paralelas** alimentan una fase común (ej. Design UX/UI + Design Technical → Development, pero cada una gatea un punto de entrada distinto):

**Regla:** `ÚLTIMA_TAREA_DE_CADA_FASE_PARALELA → PUNTO_DE_ENTRADA_ESPECÍFICO_EN_LA_FASE_CONVERGENTE`

```
Última Design UX/UI ──→ Primera tarea FE en Development (ej. MEM-081 UI-01)
Última Design Tech  ──→ Primera tarea BE en Development (ej. MEM-048 S01)
```

**NO hacer:** `Última Design UX/UI → Primera tarea BE`. Eso fuerza a BE a esperar UX innecesariamente.

##### B.1.4 — Convergencia total al final de una fase

Cuando **dos cadenas paralelas dentro de una fase** deben ambas completarse antes de la siguiente fase (ej. BE + FE → Testing):

**Regla:** `LA_ÚLTIMA_TAREA_DE_CADA_RAMA → PRIMERA_TAREA_DE_LA_FASE_SIGUIENTE`

```
Última BE  (MEM-080) ───┐
                         ├──→ Primera Testing (MEM-094)
Última FE  (MEM-093) ───┘
```

**Validación:** si en el Gantt la fase siguiente empieza antes de que ambas ramas terminen, falta una dependencia. Agregarla.

##### B.1.5 — Identificar "gate tasks" dentro de una fase

Hay tareas que por su naturaleza son **bloqueantes críticos** aunque no sean la última de la fase. Deben marcarse prioridad `C` y ser gates explícitos:

| Tipo de tarea | Por qué es gate crítico |
|---------------|-------------------------|
| Design Handoff Final | Bloquea todo el trabajo FE posterior |
| Tech Design Complete | Bloquea todo el trabajo BE posterior |
| Schema BD aprobado | Bloquea migrations, seeds, endpoints |
| Production Deploy | Gate crítico de release |
| Performance Testing validado | Gate crítico de SLA |
| Approval de arquitectura | Bloquea implementación |

---

#### B.2 — Dependencias Intra-Fase (Nivel 2: Entre Deliveries de la Misma Fase)

**Regla:** Si un delivery produce un output que otro delivery de la misma fase consume, el segundo depende del primero.

```
Delivery A (produce schema) ──→ Delivery B (consume schema para API)
                                        │
                                        └──→ Delivery C (consume API para UI)
```

**Ejemplo Memory Service R1 — Development:**
- Sprint S01 (Schema + Seeds) → Sprint S02 (Import endpoints usan el schema)
- Sprint S02 (Import) → Sprint S03 (Content endpoints usan datos importados)

**Patrón estándar en Development:**
```
S01 → S02 → S03 → S04 → S05 → S06  (cadena BE)
UI-01 → UI-02 → UI-03 → UI-04      (cadena FE)
```

---

#### B.3 — Dependencias Intra-Sprint (Nivel 3: Dentro del Mismo Delivery)

Estas son las que **más fácilmente se olvidan** y causan el problema clásico: "el Gantt muestra todas las tareas del sprint desde el día 1".

**Regla básica:** Cada tarea dentro de un sprint depende de la anterior en secuencia lógica.

```
Gate inter-sprint
    │
    ▼
Tarea 1 ──→ Tarea 2 ──→ Tarea 3 ──→ Tarea 4 (fin sprint)
```

**Reglas específicas de dependencia intra-sprint:**

| Tarea tipo | Depende de |
|-----------|-----------|
| Migration | Schema Prisma |
| Seed | Migration aplicada |
| Endpoint API | Schema + middleware/setup de Express |
| Test del endpoint | Endpoint implementado |
| UI Component | Design tokens + API endpoint |
| E2E test | Toda la feature implementada |
| Integration test | Todos los endpoints del módulo |

**Tareas paralelas dentro de un sprint (excepción a la cadena lineal):**

Si dos tareas **no dependen una de otra** pero ambas dependen del gate inter-sprint, ambas arrancan en paralelo:

```
Gate ──┬──→ Tarea A (puede correr sola)
       └──→ Tarea B (puede correr sola)
                          │
                          └──→ Tarea C (depende de ambas)
```

**Ejemplo Memory Service R1:**
- S01: `MEM-051` (Express setup) y `MEM-050` (Seeds) son paralelas después de `MEM-048` (Schema). Pero `MEM-052` (catalog cache) depende de ambas.

---

### C. Reglas para Tareas de Cierre por Fase (Opcional pero Recomendado)

**Propuesta:** Agregar una tarea de **sign-off** al final de cada fase que:
1. Dependa de la última tarea real de la fase
2. Sea la única dependencia de entrada de la siguiente fase (reemplaza los gates directos tarea→tarea)
3. Represente el sign-off formal del HO de esa fase

#### C.1 — Fases consecutivas simples

```
Última tarea real ──→ CIERRE_FASE ──→ Primera tarea siguiente fase
```

#### C.2 — Fases paralelas que alimentan una común (por rama)

Cada fase paralela tiene **su propio cierre**, que gatea **solo su punto de entrada específico**:

```
Última UX/UI ──→ CIERRE_DISEÑO_UXUI ──→ Primera tarea FE
Última Tech  ──→ CIERRE_DISEÑO_TECH  ──→ Primera tarea BE
```

#### C.3 — Fase con dos ramas paralelas que convergen

Una sola tarea de cierre que depende de **ambas ramas**:

```
Última BE ──┐
            ├──→ CIERRE_DEVELOPMENT ──→ Primera Testing
Última FE ──┘
```

#### C.4 — Ventajas de usar tareas de cierre

| Ventaja | Descripción |
|---------|-------------|
| **Sign-off explícito** | Cada fase termina con una tarea formal de aprobación del HO |
| **Reserva espacio** | Crea el slot para futuras tareas de revisión (code review, QA, integration testing) sin cambiar deps |
| **Simplifica el grafo** | 1 dep por gate inter-fase en vez de N |
| **Visibilidad en Gantt** | Cada fase termina con una barra identificable |

---

### D. Validación Post-Carga (Reglas para Verificar el Gantt)

Después de cargar todo en VTT, validar estos criterios en el Gantt:

| Regla | Qué verificar |
|-------|---------------|
| **R.1 — No todas desde día 1** | Ninguna fase (excepto la primera) arranca en el día 1 |
| **R.2 — Paralelismo correcto** | Fases paralelas se solapan; fases secuenciales NO se solapan |
| **R.3 — Convergencia completa** | Testing arranca solo cuando BE **Y** FE terminaron (no antes) |
| **R.4 — Secuencia Testing → Deploy → Operations** | Deploy no arranca hasta que Testing cerró. Operations no hasta que Deploy completó |
| **R.5 — Intra-sprint coherente** | Dentro de un sprint, las tareas están en cadena (no todas en paralelo desde el gate) |
| **R.6 — Gates críticos visibles** | Las tareas con prioridad `C` son identificables en el Gantt y bloquean lo que deben bloquear |

**Si alguna regla falla:** diagnosticar con los patrones de la sección "Lecciones Operativas Post-Carga" (más abajo) y generar script de fix.

## Los 9 Pasos del Proceso

```
  Paso 1: Leer análisis de feature
      ↓
  Paso 2: Cerrar docs PM (freeze)
      ↓
  Paso 3: Filtrar deliverables aplicables
      ↓
  Paso 4: Definir fase de iniciación (pre-SDLC)
      ↓
  Paso 5: Consolidar plan maestro
      ↓
  Paso 6: Generar Cierre PM + Handoff operativo
      ↓
  Paso 7: Generar Task Index Seed (metadata + UUIDs)
      ↓
  Paso 8: Generar script ejecutable de carga
      ↓
  Paso 9: Generar HO de ejecución para PJM
      ↓
  [FIN: PJM ejecuta · PM firma sign-off]
```

Cada paso tiene **inputs · acciones · outputs · gate de calidad**.

---

### PASO 1 — Leer Análisis de Feature

**Inputs:**
- SPEC técnica (versión más reciente aprobada)
- Metodología funcional
- Addendum(s) de integración cross-service
- Reviews de roles técnicos (AR, DB, TL, BE)
- Documento metodológico conceptual (si existe)

**Acciones del PM:**
1. Leer cada documento completo (no resumen)
2. Identificar decisiones ya tomadas (congeladas)
3. Identificar ambigüedades o preguntas abiertas
4. Verificar que las reviews de roles técnicos están integradas al SPEC

**Outputs:**
- Entendimiento consolidado del alcance (el PM debe explicar el producto en 1 párrafo sin consultar docs)
- Lista de documentos consumidos
- Lista de decisiones cerradas (D-XX con estado FROZEN)
- Lista de correcciones incorporadas (por origen: AR-OBS-XX, DB-OBS-XX, TL-OBS-XX)

**Gate de calidad:**
- [ ] PM puede explicar el producto en 1 párrafo sin consultar docs
- [ ] PM identifica mínimo 30 decisiones cerradas (proyecto promedio)
- [ ] Reviews técnicas sin observaciones abiertas

---

### PASO 2 — Cerrar Docs PM (Freeze)

**Inputs:** Outputs del Paso 1

**Acciones del PM:**
1. Validar cada doc: estado final o pendiente cierre
2. Integrar addendums al SPEC base (si son compatibles)
3. Marcar docs obsoletos con aviso explícito (no borrar)
4. Actualizar versionado (header, changelog, footer) consistente
5. Firmar cierre como PM en cada doc

**Outputs:**
- SPEC técnica en versión final con changelog claro
- Addendums integrados o explícitamente pendientes
- Docs obsoletos marcados con aviso
- Registro de decisiones cerradas actualizado

**Gate de calidad:**
- [ ] Ningún doc PM en estado `BORRADOR` o `PENDIENTE`
- [ ] Versionado consistente entre nombre de archivo, header, changelog, footer
- [ ] Addendums integrados al SPEC base o con plan de integración

---

### PASO 3 — Filtrar Deliverables Aplicables

**Inputs:**
- Catálogo SDLC completo (~438 deliverables estándar)
- Contexto del proyecto (interno vs externo, stack, restricciones)

**Acciones del PM:**
1. Recorrer las fases SDLC con sus subfases
2. Por cada deliverable decidir `✅ Aplica` o `❌ No aplica`
3. Para `❌` documentar razón breve (feature interna, desktop only, stack sin X)
4. Consolidar criterios de filtrado reutilizables
5. Calcular totales por fase + total general

**Outputs:**
- `FASES_APLICABLES_<PROYECTO>.md` con tabla por fase
- Criterios de exclusión consolidados
- Número total de deliverables aplicables (típico: 80-95%)

**Referencia:** Memory Service R1 → 390/438 (89%)

**Gate de calidad:**
- [ ] Cada exclusión tiene razón documentada
- [ ] Totales por fase suman el total original
- [ ] Criterios de filtrado consistentes

---

### PASO 4 — Definir Fase de Iniciación (Pre-SDLC)

**Inputs:**
- Outputs del Paso 3
- Contexto operativo (VM, repo, equipo, tooling)

**Acciones del PM:**
1. Identificar tareas de setup **no** presentes en el catálogo SDLC (VTT, repo, VM, onboarding)
2. Agruparlas en 7 categorías: A) VTT Setup · B) Repo · C) VM · D) Team · E) Tooling · F) Documentation · G) Kickoff
3. Desglosar tareas umbrella (ej. MEM-001..005) en sub-tareas concretas
4. Validar que las horas reflejan el trabajo real (típicamente ~32h vs 11h de la umbrella)

**Outputs:**
- `PRE_HANDOFF_INICIACION_<PROYECTO>.md` con 20-30 sub-tareas
- Mapeo INIT-* → tareas umbrella
- Estimación realista de horas

**Gate de calidad:**
- [ ] Cada sub-tarea tiene rol + horas + output claro
- [ ] Total de horas realista para el setup
- [ ] Ninguna sub-tarea duplica una del catálogo SDLC

---

### PASO 5 — Consolidar Plan Maestro

**Inputs:**
- Outputs de Pasos 3 y 4
- Listado de tareas VTT existentes

**Acciones del PM:**
1. Combinar: iniciación + fases SDLC + tareas específicas
2. Por cada tarea definir: **Título · Delivery VTT · Rol · Horas · Complexity · Produce (deliverables)**
3. Mapear explícitamente **tarea → deliverables individuales** (una tarea puede producir N documentos)
4. Calcular totales por rol, por fase, por sprint
5. Identificar dependencias críticas entre tareas

**Outputs:**
- `CONSOLIDADO_<PROYECTO>.md` con todas las tareas organizadas por fase
- Resumen por rol (horas totales)
- Lista de dependencias críticas (típico: 10-15)

**Lección aprendida (crítica):**

> Una tarea puede producir múltiples deliverables SDLC. **Explicitar la columna `Produce`** evita que se repita el análisis. Ejemplo: MEM-006 Problem Definition (3h) produce 4 documentos (0.3.1, 0.3.2, 0.3.3, 0.3.4).

**Gate de calidad:**
- [ ] Cada tarea tiene mapeo explícito `tarea → deliverables que produce`
- [ ] Suma de deliverables producidos = total de deliverables aplicables
- [ ] Dependencias críticas identificadas con impacto

---

### PASO 6 — Generar Cierre PM + Handoff Operativo

**Inputs:** Todos los outputs anteriores

**Acciones del PM:**
Estructurar `CIERRE_PM_HANDOFF_PJM_<PROYECTO>.md` en 2 partes:

- **PARTE I — CIERRE PM DEL ANÁLISIS**
  1. Documentos consumidos
  2. Decisiones PM finales (D-XX FROZEN)
  3. Correcciones incorporadas (por origen)
  4. Limitaciones documentadas (LIM-XX)
  5. Reassignments aprobados (si aplica)
  6. Veredicto PM (caja ASCII)

### PASO 8 — Generar Script Ejecutable de Carga

**Inputs:** Output del Paso 7

**Acciones del PM:**
1. Crear script Python 3.8+ con tuplas expandidas
2. Secuencia de 6 bloques:
   - `POST /api/projects` (NO wizard)
   - `POST /api/projects/:id/phases` × N
   - `POST /api/deliveries` × N
   - `POST /api/phases/:id/tasks` × N (con metadata completa)
   - Asociar tareas a deliveries
   - `POST /api/tasks/:id/dependencies` × N
3. Helpers: `auth_token()`, `post()`, `extract_id()`, `log()`
4. Manejo de errores: auth fail aborta, delivery/dep fail solo warn
5. Output: `VTT_UUIDS_<PROYECTO>.json` con IDs capturados

**Outputs:** `scripts/create_<proyecto>_vtt.py`

**Template:**

```python
#!/usr/bin/env python3
"""Script de creación VTT — <Proyecto>"""
import os, json, urllib.request, urllib.error, sys, time

API_URL = os.environ.get("VTT_API_URL", "http://77.42.88.106:3000")
SERVICE_KEY = os.environ.get("VTT_SERVICE_KEY")
if not SERVICE_KEY:
    sys.exit("ERROR: Define VTT_SERVICE_KEY")

# Constantes: USERS, STATUS_PENDING, PRIORITY
# Data: PROJECT, PHASES, DELIVERIES, TASKS, DEPENDENCIES

def auth_token():
    req = urllib.request.Request(
        f"{API_URL}/api/auth/service-token",
        data=json.dumps({"userId": PJM_UUID, "serviceKey": SERVICE_KEY}).encode(),
        headers={"Content-Type": "application/json"}, method="POST")
    with urllib.request.urlopen(req, timeout=15) as r:
        return json.loads(r.read())["data"]["token"]

def main():
    token = auth_token()
    uuids = {"projectId": None, "phases": {}, "deliveries": {}, "tasks": {}}
    # 6 bloques secuenciales con logging
    with open(OUTPUT_FILE, "w", encoding="utf-8") as f:
        json.dump(uuids, f, indent=2)

if __name__ == "__main__":
    main()
```

**Gate de calidad:**
- [ ] Script se ejecuta sin errores de sintaxis
- [ ] Todas las tuplas con número correcto de elementos
- [ ] Variables de entorno documentadas en header
- [ ] Output JSON estructurado correctamente

## Lecciones Operativas Post-Carga (Memory Service R1)

Estas son **correcciones post-ejecución** que no están en el estándar 09 porque se descubrieron al validar el Gantt después de la carga.

### Lección 1 — Gantt Muestra Tareas Intra-Sprint desde el Día 1

**Síntoma:** Las dependencias críticas (gates inter-fase) están cargadas, pero dentro de cada sprint solo la primera tarea tiene predecesor. Las demás aparecen en paralelo desde el día 1.

**Causa:** El Paso 7 documenta solo las ~15 deps críticas inter-fase. Las deps intra-sprint (secuencia lógica dentro del sprint) no se cargan.

**Solución:** Generar `scripts/fix_dev_deps_<proyecto>.py` con las deps intra-sprint:

```python
NEW_DEPS = [
    # Sprint 1
    ("MEM-049", "MEM-048", "S01: Migrations after Schema Prisma"),
    ("MEM-050", "MEM-049", "S01: Seed after Migrations applied"),
    # ... una por cada pareja tarea→tarea dentro de cada sprint
]
```

**Regla para generar estas deps:**
- La primera tarea del sprint ya tiene el gate inter-sprint (inter-fase)
- Cada tarea subsiguiente depende de la anterior dentro del sprint
- Si hay paralelismo dentro del sprint, todas dependen del gate

---

### Lección 2 — Fases Superpuestas Incorrectamente

**Síntoma:** Operations aparece en medio de Deploy, o Testing se superpone con Development.

**Diagnóstico:** La primera tarea de la fase posterior depende de una tarea que completa temprano en la fase anterior (no la última).

**Solución:** Verificar la cadena de deps y forzar que el gate de la fase siguiente dependa de la ÚLTIMA tarea real de la fase anterior:

```python
# Ejemplo Memory Service R1:
# Operations gate (MEM-111) dependía de MEM-110 (Rollback Plan),
# pero MEM-110 dependía de MEM-105 (CI/CD) y completaba temprano.
# Fix: agregar MEM-110 → MEM-109 (Post-Deploy Monitoring)
# Así Operations no arranca hasta que Deploy termina completamente.
```

---

### Lección 3 — Convergencia de Ramas Paralelas

**Síntoma:** Testing empieza cuando FE termina, pero BE todavía está en sprint.

**Causa:** Development tiene dos cadenas paralelas (BE + FE). El gate de Testing solo esperaba a una.

**Solución:** El gate de la fase convergente debe depender de la **última tarea de cada rama**:

```python
# Testing gate debe esperar tanto a BE como a FE:
("MEM-094", "MEM-080", "Testing gate espera BE end"),
("MEM-094", "MEM-093", "Testing gate espera FE end"),
```

---

### Lección 4 — Tareas de Cierre por Fase (Sign-Off)

**Propuesta validada (opcional):** Agregar una tarea de cierre al final de cada fase que:
- Depende de la última tarea real de la fase
- Es la dependencia de entrada de la siguiente fase
- Representa el sign-off formal del HO de esa fase

**Para fases paralelas** (Design UX/UI + Design Technical → Development):
- Cada fase de diseño tiene su propia tarea de cierre
- Cada cierre gatea solo SU punto de entrada en Development:
  - `CIERRE_DISEÑO_UXUI` → primera tarea FE
  - `CIERRE_DISEÑO_TECH` → primera tarea BE

**Para fases convergentes** (BE+FE → Testing):
- Una tarea de cierre que depende de la última de cada rama
- Gatea la primera tarea de la siguiente fase

## Errores Comunes a Evitar

| Error | Consecuencia | Corrección |
|-------|--------------|-----------|
| Asumir UUIDs en VTT cuando se hizo cleanup | Script falla con 404 | Confirmar estado real de VTT antes de scriptear |

| 1 HO por rol por sprint | Exceso de HOs (22 en un proyecto) | 1 HO por fase con líder de área |
| Omitir descripción en tareas | PJM repite el análisis al cargar | 150–300 chars por tarea |
| Hardcodear UUIDs en docs PM | Stale data cuando se recrea el proyecto | Usar `TASK_INDEX_SEED` como source of truth |
| Usar wizard de creación de proyecto | Complica el control de la carga | POST simples separados |
| `complexity: "medium"` | 400 Validation Error | MAYÚSCULAS: `LOW/MEDIUM/HIGH` |
| `assignedTo` en vez de `assignedToId` | Campo ignorado | Siempre `assignedToId` |
| `category: "chore"` | 400 — no existe | Mapear a `deployment`/`documentation`/`development` |
| Cargar solo deps inter-fase (skipping intra-sprint) | Gantt: tareas del sprint desde día 1 | Script fix post-carga con deps intra-sprint |
| Gate de fase N+1 depende de tarea que termina temprano en fase N | Fases solapadas incorrectamente | Gate debe depender de la ÚLTIMA tarea real de la fase N |
