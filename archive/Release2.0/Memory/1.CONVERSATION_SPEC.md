# Spec: Estructura de Conversación SDK

## Fuente de verdad
Basado en datos reales observados en `agents-state/fe-agent/conversation.jsonl`
y en el stream del SDK (`agent-runtime_06.py`).

---

## 1. Qué es conversation.jsonl

**Una línea por corrida** — no por mensaje individual.
Cada línea representa el resultado final de una ejecución completa del agente.

> El historial de mensajes turno a turno NO se guarda aquí.
> Aquí va el output final consolidado + metadata de la corrida.

---

## 2. Schema real observado

```json
{
  "timestamp":  "2026-03-16T07:02:14.756273+00:00",
  "runId":      "experimental_fe_001",
  "taskId":     "FE-001",
  "sessionId":  "4070041e-bbd2-4d99-9436-ba5730d09a12",
  "summary":    "## Confirmación de entrega...\n[output completo del agente en markdown]",
  "result":     "success"
}
```

### Campos y fuentes:

| Campo | Tipo | Fuente en el SDK | Notas |
|-------|------|-----------------|-------|
| `timestamp` | ISO string | `datetime.now(timezone.utc).isoformat()` | Al momento de finalizar la corrida |
| `runId` | string | Constante del script | Ej: `experimental_fe_001` |
| `taskId` | string | Constante del script | Ej: `FE-001` |
| `sessionId` | UUID string | `AssistantMessage.session_id` o `ResultMessage.session_id` | ID de la sesión generada por Claude |
| `summary` | markdown string | `assistantText` — concatenación de todos los `TextBlock` de `AssistantMessage` | En la práctica es el output completo final |
| `result` | `"success"` \| `"error"` | `"error"` si `ResultMessage.is_error == True`, si no `"success"` | |

### Observación importante sobre `summary`:

El campo `summary` actualmente almacena **el output completo del agente** (puede ser markdown largo con tablas, secciones, etc.), no un resumen corto. Esto es lo que produce el agente como confirmación de entrega.

## 3. Tipos de mensajes del SDK (stream)

El SDK entrega mensajes en streaming via `async for message in query(...)`.
Los tipos relevantes son:

### AssistantMessage
Llega en cada turno donde el modelo produce texto o usa tools.

```python
isinstance(message, AssistantMessage)

# Campos útiles:
message.session_id          # UUID de la sesión
message.content             # lista de bloques

# Tipos de bloques dentro de content:
block.type == "text"        # texto generado por el modelo
block.type == "tool_use"    # el modelo quiere usar una herramienta
block.type == "tool_result" # resultado de herramienta (en algunos contextos)
```

### ResultMessage
Llega UNA VEZ al final de la corrida completa.

```python
isinstance(message, ResultMessage)

# Campos útiles:
message.session_id          # UUID de la sesión
message.total_cost_usd      # costo total de la corrida
message.usage               # dict con tokens (ver USAGE_TRACKER_SPEC)
message.duration_ms         # duración total en ms
message.duration_api_ms     # duración solo de llamadas API
message.num_turns           # número de turnos del agente
message.is_error            # True si la corrida terminó en error
message.result              # texto del resultado final (si lo hay)
```

## 4. Clasificación de eventos del stream

Basado en el análisis de arquitectura (virtual_teams_conversacion_Part1.md, sección 2.5),
los eventos del stream se pueden clasificar así:

| Tipo de evento | Origen en SDK | Descripción |
|----------------|--------------|-------------|
| `user_input` | El prompt que se pasa a `query()` | La tarea/instrucción enviada al agente |
| `assistant_thought` | `AssistantMessage` con `block.type == "text"` | Texto generado por el modelo |
| `tool_call` | `AssistantMessage` con `block.type == "tool_use"` | El agente solicita ejecutar una herramienta |
| `tool_result` | Bloque de resultado de tool en stream parcial | Resultado de la ejecución de la herramienta |
| `agent_result` | `ResultMessage` | Resultado final de la corrida |
| `agent_error` | `ResultMessage` con `is_error=True` | Corrida terminada en error |
| `system_status` | Stderr / mensajes del proceso | Logs del sistema |

> Esta clasificación aplica si en el futuro se quiere guardar el transcript turno a turno
> en lugar de solo el output final.

---

## 5. Lo que conversation.jsonl NO guarda hoy

Por diseño actual, conversation.jsonl guarda solo el **resultado final por corrida**.
Lo siguiente NO está aquí (va en otros archivos):

| Dato | Dónde va |
|------|----------|
| Tokens y costo | `usage.jsonl` |
| Archivos creados | `artifacts.jsonl` |
| Errores | `errors.jsonl` |
| Estado de sesión activa | `session-map.json` |
| Transcript completo turno a turno | `runs/agent2-last-run.json` (campo `rawMessages`) |

---

## 6. Evolución futura: transcript por turno

Si se necesita guardar el historial completo (no solo el output final),
el schema por entrada sería:

```json
{
  "timestamp":   "2026-03-16T07:02:14Z",
  "runId":       "experimental_fe_001",
  "taskId":      "FE-001",
  "sessionId":   "4070041e-...",
  "eventType":   "assistant_thought | tool_call | tool_result | agent_result | agent_error",
  "turn":        3,
  "content":     "texto o dict del evento",
  "result":      null
}
```

Una línea por evento, múltiples líneas por corrida.
El `result` solo se llena en la línea de tipo `agent_result`.

> **No implementar hasta que haya necesidad real de auditar turnos individuales.**
> El output final en `summary` es suficiente para la fase actual.
