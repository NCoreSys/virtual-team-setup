# Spec: Agent Metrics & Cost Analytics

## Objetivo
Entender cuánto cuesta cada agente, cada tarea y cada proyecto — y encontrar dónde reducir tokens.

**Fuente de datos**: `agents-state/{agentId}/usage.jsonl` (ver USAGE_TRACKER_SPEC.md)

---

## 1. Métricas de Costo

### 1.1 Costo por agente (total acumulado)

| Métrica | Fórmula | Para qué sirve |
|---------|---------|----------------|
| `total_cost_usd` | `sum(costUsd)` por agente | Ver cuánto costó cada agente en total |
| `avg_cost_per_run` | `total_cost / num_runs` | Baseline de costo por corrida |
| `avg_cost_per_task` | `total_cost / unique_taskIds` | Costo real por tarea (excluye reintentos) |
| `cost_per_artifact` | `total_cost / total_archivos_generados` | Eficiencia: cuánto cuesta producir un archivo |

### 1.2 Costo por tarea / tipo de tarea

Agrupar por `taskId` para ver el costo real de cada tarea incluyendo reintentos:

```
taskId: FE-002
  runs: 2  (experimental_fe_002 apareció 2 veces → hubo reintento)
  total_cost: $0.XX
  total_turns: N
  resultado_final: success
```

### 1.3 Costo por proyecto

Cuando haya múltiples proyectos, agrupar por prefijo del `taskId`:
- `FE-*` → Frontend tasks
- `BE-*` → Backend tasks
- `QA-*` → QA tasks

```
Proyecto FE:  $X.XX  (N tareas, N runs)
Proyecto BE:  $X.XX
Total:        $X.XX
```

## 2. Métricas de Tokens

### 2.1 Desglose de tokens por corrida

| Campo | Descripción | Peso en costo |
|-------|-------------|---------------|
| `input_tokens` | Tokens frescos de entrada | Alto |
| `output_tokens` | Tokens generados | Muy alto |
| `cache_creation_input_tokens` | Primera vez que se cachea contexto | Bajo (se paga una vez) |
| `cache_read_input_tokens` | Lecturas desde cache | Muy bajo (5-10% del precio normal) |

> **Insight de tus datos reales**: en las corridas de agent2, `cache_read` supera 900K tokens en algunas corridas. Ese cache es lo que hace viable el análisis de repos grandes sin quebrar en costo.

### 2.2 Ratio de eficiencia de cache

```
cache_ratio = cache_read / (input + cache_read + cache_creation)
```

| Rango | Interpretación |
|-------|----------------|
| > 80% | Excelente — el contexto se está reutilizando bien |
| 50-80% | Aceptable |
| < 50% | El agente está pagando mucho por tokens frescos — revisar contexto |

### 2.3 Tokens por turno

```
tokens_per_turn = (input + output) / num_turns
```

Indicador de verbosidad del agente. Si sube mucho entre versiones de workflow → el prompt se infló.

### 2.4 Ratio output/input

```
output_ratio = output_tokens / input_tokens
```

| Rango | Interpretación |
|-------|----------------|
| > 2.0 | El agente escribe mucho — tareas de generación de contenido |
| 0.5-2.0 | Balanceado |
| < 0.5 | El agente consume mucho más de lo que produce — posible contexto inflado |

## 3. Métricas de Complejidad por Tarea

Proxy de complejidad usando datos disponibles sin ML:

| Indicador | Cálculo | Significado |
|-----------|---------|-------------|
| `turn_complexity` | `num_turns` | Más turns = tarea más compleja o ambigua |
| `token_complexity` | `input + output` total | Volumen total procesado |
| `retry_count` | apariciones del mismo `taskId` | Cuántas veces falló antes de completarse |
| `artifacts_count` | archivos en `artifacts.jsonl` para ese run | Alcance de la tarea |
| `had_errors` | si `errors.jsonl` tiene entradas para ese run | Fricción durante ejecución |

### Clasificación automática de complejidad:

```
LOW:    num_turns <= 5  AND  costUsd < $0.50
MEDIUM: num_turns 6-15  OR  costUsd $0.50-$2.00
HIGH:   num_turns > 15  OR  costUsd > $2.00  OR  retry_count > 1
```

## 4. Métricas de Reducción de Tokens (el objetivo principal)

Para saber **dónde reducir**, hay que identificar qué está inflando los tokens:

### 4.1 Comparativa entre versiones de workflow

Si tienes `workflow-agent2.md` v1 y v2, comparar:

```
v1 → avg_input_tokens: X
v2 → avg_input_tokens: Y
delta: Y - X  (negativo = mejoró)
```

Medir el impacto de cada cambio al workflow en tokens consumidos.

### 4.2 Tareas que consumen desproporcionado

```
cost_per_artifact > avg_cost_per_artifact * 1.5  → candidata a optimizar
num_turns > avg_turns * 2                         → handoff ambiguo
retry_count > 1                                   → problema estructural
```

### 4.3 Señales de contexto inflado

- `input_tokens` alto pero `cache_read` bajo → se está mandando contexto fresco innecesario
- `num_turns` alto pero pocos archivos en `artifacts.jsonl` → el agente está "pensando mucho"
- `output_tokens` muy alto → el agente está siendo demasiado verboso en sus respuestas

### 4.4 Estrategias de reducción (para documentar resultados)

| Estrategia | Cómo medir el impacto |
|------------|----------------------|
| Reducir tamaño del system prompt | `avg_input_tokens` antes vs después |
| Aumentar uso de cache (contexto estático al inicio) | `cache_ratio` antes vs después |
| Handoffs más concretos | `avg_num_turns` antes vs después |
| Limitar `max_turns` | `costUsd` cap por corrida |
| Separar tareas grandes en subtareas | `retry_count` antes vs después |
