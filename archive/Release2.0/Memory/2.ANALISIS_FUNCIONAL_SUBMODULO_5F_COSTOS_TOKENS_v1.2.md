# 1. CASOS DE USO

## CU-5F-01: Cargar Archivo JSONL

**Actor:** Admin, PM

**Precondiciones:**
- Usuario autenticado con rol admin o PM
- Archivo JSONL exportado de Claude Code CLI

**Flujo Principal:**
1. Usuario selecciona archivo JSONL desde su máquina local
2. Usuario envía archivo a `POST /api/costs/sync/upload`
3. Sistema valida formato y tamaño (máx 50MB)
4. Sistema procesa línea por línea
5. Sistema infiere proyecto de cada línea via CWD
6. Sistema extrae taskIds de cada línea via content
7. Sistema calcula costo de cada línea
8. Sistema inserta líneas nuevas (idempotencia por uuid)
9. Sistema actualiza summaries afectados (usando upsert)
10. Sistema retorna reporte de sync

**Flujos Alternativos:**

| # | Condición | Comportamiento |
|---|-----------|----------------|
| 3a | Archivo > 50MB | Retornar 400 con `FILE_TOO_LARGE` |
| 3b | Extensión no .jsonl/.txt | Retornar 400 con `INVALID_FILE_TYPE` |
| 4a | Línea con JSON inválido | Registrar en errors, continuar con siguiente |
| 5a | CWD no matchea proyecto | `projectCode = null`, continuar |
| 7a | Modelo sin precio | Usar fallback de BD (lanza error si no existe) |
| 7b | No hay fallback en BD | Error fatal, abortar sync |
| 8a | uuid ya existe | Skip línea, incrementar `linesSkipped` |

**Postcondiciones:**
- Líneas nuevas insertadas en `agent_session_lines`
- Summaries actualizados en `agent_session_summaries` y `agent_cost_summaries`
- Response con conteo de procesadas/insertadas/skipped/errores

## RF-5F-01: Validación de Upload

**Descripción:** Validar archivo JSONL antes de procesar.

**Reglas:**
- Tamaño máximo: 50MB
- Extensiones permitidas: `.jsonl`, `.txt`
- Content-Type: `multipart/form-data`

**Validaciones:**
```typescript
interface UploadValidation {
  maxSizeBytes: 52_428_800; // 50MB
  allowedExtensions: ['.jsonl', '.txt'];
  allowedMimeTypes: ['application/json', 'text/plain', 'application/octet-stream'];
}
```

**Nota Técnica:** Crear middleware dedicado `uploadJsonl`. NO modificar el middleware existente `upload.ts`.

## RF-5F-02: Parsing Línea por Línea

**Descripción:** Procesar archivo JSONL línea por línea sin cargar todo en memoria.

**Algoritmo:**
1. Leer archivo en stream
2. Por cada línea:
   - Intentar `JSON.parse()`
   - Si falla: registrar error con número de línea, continuar
   - Si éxito: extraer campos y continuar procesamiento
3. Usar batch inserts (chunks de 100-500 líneas)

**Campos extraídos directamente del JSONL:**

| Campo JSONL | Campo BD | Transformación |
|-------------|----------|----------------|
| `uuid` | `uuid` | Directo (UNIQUE) |
| `sessionId` | `sessionId` | Directo |
| `parentUuid` | `parentUuid` | Directo |
| `timestamp` | `timestamp` | Parse ISO-8601 |
| `message.role` o `type` | `role` | Directo |
| `cwd` | `cwd` | Directo |
| `message.model` | `model` | Directo |
| `gitBranch` | `gitBranch` | Directo |
| `message.usage.input_tokens` | `inputTokens` | Directo, default 0 |
| `message.usage.output_tokens` | `outputTokens` | Directo, default 0 |
| `message.usage.cache_creation_input_tokens` | `cacheCreationTokens` | Directo, default 0 |
| `message.usage.cache_read_input_tokens` | `cacheReadTokens` | Directo, default 0 |
| `message.usage.service_tier` | `serviceTier` | Directo |

**Campo calculado:**
- `totalTokens = inputTokens + outputTokens`

> **Nota:** Los cache tokens (`cacheCreationTokens`, `cacheReadTokens`) NO se incluyen en `totalTokens`. Los tokens de cache son derivados de input; incluirlos duplicaría el conteo.

## RF-5F-03: Inferencia de Proyecto desde CWD

**Descripción:** Inferir código de proyecto desde el path del directorio de trabajo.

**Algoritmo:**
```typescript
const PROJECT_MAPPINGS: Record<string, string> = {
  'designmine': 'DSM',
  'virtual-teams-tracking': 'VTT',
  'vtt': 'VTT',
  'dsm': 'DSM',
};

function inferProjectCode(cwd: string | null): string | null {
  if (!cwd) return null;
  
  const cwdLower = cwd.toLowerCase();
  
  for (const [pathFragment, code] of Object.entries(PROJECT_MAPPINGS)) {
    if (cwdLower.includes(pathFragment)) {
      return code;
    }
  }
  
  return null;
}
```

**Comportamiento:**

| Situación | Acción | Valores en BD |
|-----------|--------|---------------|
| CWD vacío o null | Continuar sin proyecto | `projectCode = null`, `projectId = null` |
| CWD no matchea | Continuar sin proyecto | `projectCode = null`, `projectId = null` |
| CWD matchea | Asignar código | `projectCode = 'DSM'`, buscar `projectId` |
| `projectCodeHint` enviado | Usar hint | `projectCode = hint`, buscar `projectId` |

**Importante:** Las líneas sin proyecto **SÍ se insertan**. Se filtran después.

## RF-5F-05: Extracción de TaskIds desde Content

**Descripción:** Buscar patrones de task ID en el contenido del mensaje.

**Algoritmo:**
```typescript
function extractTaskIds(content: any[], projectCode: string | null): string[] {
  const taskIds = new Set<string>();
  
  // Concatenar todo el texto de content
  const text = content
    ?.filter(c => c.type === 'text')
    ?.map(c => c.text)
    ?.join(' ') || '';
  
  if (!text) return [];
  
  // Patrón 1: Código de proyecto + número (DSM-266, VTT-001)
  if (projectCode) {
    const projectPattern = new RegExp(`${projectCode}-\\d+`, 'gi');
    const matches = text.match(projectPattern) || [];
    matches.forEach(m => taskIds.add(m.toUpperCase()));
  }
  
  // Patrón 2: Sprint + Agente + número (S9-BE-01, S7-FE-03)
  const sprintPattern = /S\d+-[A-Z]{2}-\d+/gi;
  const sprintMatches = text.match(sprintPattern) || [];
  sprintMatches.forEach(m => taskIds.add(m.toUpperCase()));
  
  // Patrón 3: Agente + número (FE-001, BE-042, DB-003)
  const agentPattern = /\b(?:FE|BE|DB|QA|TL|DO|SA|AR)-\d{3,4}\b/gi;
  const agentMatches = text.match(agentPattern) || [];
  agentMatches.forEach(m => taskIds.add(m.toUpperCase()));
  
  return [...taskIds];
}
```

**Comportamiento:**

| Situación | Valores en BD |
|-----------|---------------|
| No se detectan task IDs | `taskIds = []`, `hasTaskId = false` |
| Se detecta 1 task ID | `taskIds = ['DSM-266']`, `hasTaskId = true` |
| Se detectan múltiples | `taskIds = ['DSM-266', 'DSM-267', 'S9-BE-01']`, `hasTaskId = true` |
| Task ID no existe en BD | **Se guarda igual**, `taskIdsValidated = false` |

**Importante:** Los task IDs **NO se validan** contra tabla `tasks` en tiempo de sync (performance).

## RF-5F-06: Cálculo de Costo con Fallback

**Descripción:** Calcular costo estimado, usando precio default si el modelo no tiene precio configurado.

> ⚠️ **PREREQUISITO:** El seed de `model_pricing` (tarea 5F-DB-005) DEBE ejecutarse ANTES de implementar este service.

**Algoritmo:**
```typescript
async function calculateLineCost(
  model: string,
  inputTokens: number,
  outputTokens: number,
  cacheReadTokens: number,
  cacheCreationTokens: number
): Promise<CostCalculationResult> {
  
  // 1. Buscar precio del modelo exacto con lógica temporal
  let pricing = await getActivePrice('anthropic', model);
  let usedDefaultPricing = false;
  let pricingModelUsed = model;
  
  // 2. Si no existe, usar precio default de BD
  if (!pricing) {
    pricing = await getDefaultFallbackPrice();  // ⚠️ Lanza error si no existe
    usedDefaultPricing = true;
    pricingModelUsed = pricing.modelName;
  }
  
  // 3. Calcular costos
  const inputCost = (inputTokens / 1_000_000) * Number(pricing.inputCostPerMillion);
  const outputCost = (outputTokens / 1_000_000) * Number(pricing.outputCostPerMillion);
  
  // Cache read: usar precio específico o 10% del input
  const cacheReadPrice = pricing.cacheReadCostPerMillion 
    ? Number(pricing.cacheReadCostPerMillion)
    : Number(pricing.inputCostPerMillion) * 0.1;
  const cacheReadCost = (cacheReadTokens / 1_000_000) * cacheReadPrice;
  
  // Cache creation: usar precio específico o 25% del input
  const cacheCreationPrice = pricing.cacheWriteCostPerMillion
    ? Number(pricing.cacheWriteCostPerMillion)
    : Number(pricing.inputCostPerMillion) * 0.25;
  const cacheCreationCost = (cacheCreationTokens / 1_000_000) * cacheCreationPrice;
  
  const totalCost = inputCost + outputCost + cacheReadCost + cacheCreationCost;
  
  return {
    estimatedCost: round6(totalCost),
    usedDefaultPricing,
    pricingModelUsed
  };
}

/**
 * Obtener precio activo con lógica temporal correcta.
 * Retorna el precio más reciente vigente.
 */
async function getActivePrice(provider: string, modelName: string): Promise<ModelPricing | null> {
  return prisma.modelPricing.findFirst({
    where: {
      provider,
      modelName,
      isActive: true,
      effectiveFrom: { lte: new Date() },
      OR: [
        { effectiveTo: null },
        { effectiveTo: { gte: new Date() } }
      ]
    },
    orderBy: { effectiveFrom: 'desc' }  // ⚠️ CRÍTICO: precio más reciente
  });
}

/**
 * Obtener precio fallback de BD.
 * ⚠️ NUNCA usa valores hardcodeados — lanza error si no existe.
 */
async function getDefaultFallbackPrice(): Promise<ModelPricing> {
  const fallback = await prisma.modelPricing.findFirst({
    where: { isDefaultFallback: true, isActive: true },
    orderBy: { effectiveFrom: 'desc' }
  });
  
  if (!fallback) {
    throw new Error(
      'FATAL: No default fallback pricing configured. Run seed 5F-DB-005.'
    );
  }
  
  return fallback;
}
```

**Campos guardados:**

| Campo | Descripción |
|-------|-------------|
| `estimatedCost` | Costo calculado (6 decimales) |
| `usedDefaultPricing` | `true` si usó fallback |
| `pricingModelUsed` | Modelo usado para pricing (puede diferir de `model`) |

## RF-5F-07: Actualización de Summaries

**Descripción:** Actualizar tablas de agregación después de cada sync usando **upsert**.

**Tablas afectadas:**
- `agent_session_summaries` — por sessionId
- `agent_cost_summaries` — por projectCode

**Algoritmo:**
1. Obtener lista de `sessionIds` únicos de líneas insertadas
2. Para cada sessionId:
   - Recalcular totales desde `agent_session_lines`
   - **Upsert** en `agent_session_summaries`
3. Obtener lista de `projectCodes` únicos afectados
4. Para cada projectCode:
   - Recalcular totales desde `agent_session_summaries`
   - **Upsert** en `agent_cost_summaries`

**Implementación:**
```typescript
await prisma.agentSessionSummary.upsert({
  where: { sessionId },
  create: { sessionId, ...aggregation },
  update: { ...aggregation }
});
```

**Campos calculados en session_summaries:**

| Campo | Fórmula |
|-------|---------|
| `lineCount` | `COUNT(*)` |
| `totalTokens` | `SUM(totalTokens)` |
| `totalCost` | `SUM(estimatedCost)` |
| `linesWithTask` | `COUNT(*) WHERE hasTaskId = true` |
| `costWithTask` | `SUM(estimatedCost) WHERE hasTaskId = true` |
| `linesNoTask` | `COUNT(*) WHERE hasTaskId = false` |
| `costNoTask` | `SUM(estimatedCost) WHERE hasTaskId = false` |
| `linesWithDefaultPricing` | `COUNT(*) WHERE usedDefaultPricing = true` |
| `taskIds` | `ARRAY_AGG(DISTINCT unnest(taskIds))` |
| `hasTaskIds` | `BOOL_OR(hasTaskId)` |
| `firstLineAt` | `MIN(timestamp)` |
| `lastLineAt` | `MAX(timestamp)` |
| `primaryModel` | Modelo más frecuente |

# 3. REGLAS DE NEGOCIO

## RN-5F-01: Idempotencia por UUID

**Regla:** El campo `uuid` del JSONL es único. Si una línea con el mismo uuid ya existe, se salta sin error.

**Implementación:**
```sql
INSERT INTO agent_session_lines (...) 
ON CONFLICT (uuid) DO NOTHING
```

**Métricas de sync:**
- `linesInserted`: nuevas líneas insertadas
- `linesSkipped`: líneas con uuid ya existente

---

## RN-5F-02: Líneas sin Proyecto

**Regla:** Las líneas sin proyecto identificado **SÍ se insertan** con `projectCode = null`.

**Comportamiento:**
- NO es error, es warning
- Se contabilizan en `warnings.linesWithoutProject`
- Se pueden consultar via `GET /api/costs/unassigned`
- Se excluyen de summaries por proyecto (no tienen projectCode)

---

## RN-5F-03: TaskIds Inexistentes

**Regla:** Los taskIds extraídos **NO se validan** contra tabla `tasks` en tiempo de sync.

**Razón:** Performance — el sync puede tener miles de líneas.

**Comportamiento:**
- `taskIds` se guardan como array de strings
- `taskIdsValidated = false` indica que no se verificó
- La validación se puede hacer después bajo demanda
- En consulta por tarea, se indica `existsInTasksTable`

---

## RN-5F-04: Múltiples TaskIds en una Línea

**Regla:** Si una línea menciona múltiples taskIds, se guardan **todos** en el array.

**Ejemplo:**
```json
{
  "content": "...trabajando en DSM-266 y DSM-267, también S9-BE-01..."
}
```
**Resultado:** `taskIds = ['DSM-266', 'DSM-267', 'S9-BE-01']`

**Implicación en costos:**
- El costo de la línea se reporta en **cada** tarea mencionada
- Al sumar costos por proyecto, la línea cuenta **una sola vez**
- Al sumar costos por tarea individual, la línea aparece en **cada una**

---

## RN-5F-05: Fallback de Pricing — Solo BD

**Regla:** El fallback de pricing **SIEMPRE** viene de la base de datos. **NUNCA** se usan valores hardcodeados.

**Implementación:**
```typescript
async function getDefaultFallbackPrice(): Promise<ModelPricing> {
  const fallback = await prisma.modelPricing.findFirst({
    where: { isDefaultFallback: true, isActive: true },
    orderBy: { effectiveFrom: 'desc' }
  });
  
  if (!fallback) {
    throw new Error('FATAL: No default fallback pricing configured. Run seed 5F-DB-005.');
  }
  
  return fallback;
}
```

**Trazabilidad cuando se usa fallback:**
1. `usedDefaultPricing = true`
2. `pricingModelUsed` = modelo usado para calcular (ej: 'claude-sonnet-4-5')
3. El modelo original queda en campo `model`

---

## RN-5F-06: Límites de Archivo

**Regla:** Límites para proteger el sistema.

| Límite | Valor |
|--------|-------|
| Tamaño máximo archivo | 50MB |
| Extensiones permitidas | `.jsonl`, `.txt` |
| Batch insert size | 100-500 líneas |

---

## RN-5F-07: Warnings vs Errores

**Regla:** Distinguir entre situaciones que detienen el proceso y situaciones que se registran pero no detienen.

| Situación | Tipo | Comportamiento |
|-----------|------|----------------|
| Archivo > 50MB | **Error** | Rechazar upload |
| Extensión inválida | **Error** | Rechazar upload |
| **No hay fallback en BD** | **Error Fatal** | Abortar sync |
| Línea con JSON inválido | **Warning** | Logear, continuar |
| CWD no matchea proyecto | **Warning** | Guardar con null |
| Modelo sin precio | **Warning** | Usar fallback |
| uuid duplicado | **Info** | Skip línea |

## RN-5F-08: Exclusión Memory / Hook Manager

**Regla:** Este módulo **SOLO** procesa datos de Claude Code CLI via JSONL.

**NO se procesan:**
- Sesiones del Memory del sistema de agentes
- Datos del Hook Manager
- Llamadas via API SDK
- Cualquier fuente que no sea upload manual de JSONL

## RN-5F-09: Autorización — Solo PM y Admin

**Regla:** Todos los endpoints del módulo 5F requieren autenticación con rol **PM** o **admin**.

**Implementación:**
```typescript
router.use(authenticate);
router.use(authorize(['pm', 'admin']));
```

## RN-5F-10: BigInt Serialization

**Regla:** Los campos `totalTokens` en summaries son `BigInt`. Deben convertirse a `Number` antes de retornar en JSON.

**Razón:** `JSON.stringify(BigInt)` lanza `TypeError`.

**Implementación:**
```typescript
return {
  ...summary,
  totalTokens: Number(summary.totalTokens)
};
```

---

## RN-5F-11: Lógica Temporal de Precios

**Regla:** Al buscar precio de un modelo, usar el precio más reciente vigente.

**Implementación:**
```typescript
await prisma.modelPricing.findFirst({
  where: {
    provider, modelName, isActive: true,
    effectiveFrom: { lte: new Date() },
    OR: [
      { effectiveTo: null },
      { effectiveTo: { gte: new Date() } }
    ]
  },
  orderBy: { effectiveFrom: 'desc' }  // ⚠️ CRÍTICO
});
```

**Sin el `orderBy: desc`:** Si hay dos precios activos para el mismo modelo, el resultado es no-determinístico.

---

## RN-5F-12: Body Limit para Sync Text

**Regla:** El endpoint `POST /api/costs/sync/text` requiere configurar body limit de 50MB.

**Implementación:**
```typescript
router.post(
  '/sync/text',
  express.json({ limit: '50mb' }),
  authenticate,
  authorize(['pm', 'admin']),
  costsController.syncText
);
```

**Razón:** Express.js tiene límite default de 100KB. Sin esta configuración, archivos grandes retornan 413.

---

## RN-5F-13: Upsert en Summaries

**Regla:** Los summaries (`agent_session_summaries`, `agent_cost_summaries`) se actualizan usando **upsert**, no update.

**Razón:** Garantiza idempotencia. Si el summary no existe, se crea; si existe, se actualiza.

---

## RN-5F-14: Fallback Único

**Regla:** Solo puede existir **UN** registro con `isDefaultFallback = true AND isActive = true`.

**Implementación (en controller):**
```typescript
if (body.isDefaultFallback === true) {
  await prisma.modelPricing.updateMany({
    where: { isDefaultFallback: true, isActive: true },
    data: { isDefaultFallback: false }
  });
}
```
