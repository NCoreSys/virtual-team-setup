## 1. Resumen Ejecutivo

### 1.1 Qué es

El Memory Service es el módulo responsable de almacenar, clasificar e indexar las conversaciones y resultados de las corridas de agentes. Provee contexto histórico para que los agentes puedan ejecutar tareas con conocimiento de trabajo previo.

### 1.2 Consumidor primario

El **TL/DL-Agent** es el consumidor principal. Cuando necesita armar el contexto para asignar una tarea a un agente, consulta Memory Service para obtener información relevante de corridas anteriores.

### 1.3 Decisión clave

El Memory Service vive como un **módulo separado** con su propia base de datos (`memory_db`), independiente de VTT (`vtt_db`). Esto permite que sea reutilizable para múltiples proyectos.

---

## 2. Contexto y Antecedentes

### 2.1 VTM (Virtual Teams Memory)

Existe un módulo previo llamado VTM que:

- Procesa archivos `.jsonl` de Claude Code
- Clasifica por reglas determinísticas
- Tiene 56 tags en 4 categorías: topic/(13), tipo/(10), agente/(7), fase/(6)
- Corre en puerto 3002 local
- Fases A-D completadas (21 tareas, 26 PRs)

**Limitación:** VTM fue diseñado para conversaciones históricas de Claude Code, no para Agent Runtime.

### 2.2 Por qué un nuevo Memory Service

El Agent Runtime genera datos en un formato diferente:

- `run-request.json` con metadata VTT
- Conversaciones estructuradas
- Resultados con sessionId, costos, duración
- Relación directa con tareas/fases/proyectos

Memory Service unifica todo esto en un formato propio.

## 3. Arquitectura

### 3.1 Vista general

```
VM (Hetzner)
├── VTT Backend (API:3000) ─── PostgreSQL vtt_db
├── Hook Manager (orquesta)
└── Memory Service ─────────── PostgreSQL memory_db (separada)
    │
    ├── Worker/Watcher (detecta corridas terminadas)
    ├── Clasificador (reglas determinísticas)
    ├── Indexador (índice invertido)
    └── API de consulta
```

### 3.2 Separación de bases de datos

| Base de datos | Contenido |
|---------------|-----------|
| **vtt_db** | Proyectos, fases, tareas, usuarios, permisos |
| **memory_db** | Corridas, conversaciones, clasificaciones, índices |

**Beneficio:** Memory Service es reutilizable. Puede servir a múltiples proyectos VTT o incluso a otros sistemas.

## 4. Decisiones de Diseño

### 4.1 Tabla de decisiones

| Aspecto | Decisión | Justificación |
|---------|----------|---------------|
| **Consumidor primario** | TL/DL-Agent | Arma contexto para asignaciones |
| **Modelo de contexto** | Híbrido: memoria filtrada + resumen base + referencias | Balancear relevancia y completitud |
| **Clasificación** | Dinámica, automática con reglas, ajuste humano cuando falla | Escalable y corregible |
| **Relevancia cruzada** | Explícita + por reglas + por dependencias | Múltiples señales de relevancia |
| **Cuándo clasifica** | Inmediato post-corrida (no batch) | Contexto disponible de inmediato |
| **Quién clasifica** | Memory Service (worker separado) | No bloquea el flujo principal |
| **Dónde vive** | Módulo separado, BD propia | Reutilizable |
| **Relación con VTT** | IDs en payload (projectId, phaseId, taskId) | Trazabilidad sin acoplamiento |
| **Formato almacenamiento** | Unificado: request + conversación + resultado + clasificación | Todo en un lugar |
| **Niveles de búsqueda** | Por corrida + por turn | Granularidad flexible |

### 4.2 Modelo de contexto híbrido

Cuando TL-Agent consulta contexto, recibe:

1. **Memoria filtrada** (prioridad): Corridas relevantes al tema
2. **Resumen base** (complemento): Contexto general del proyecto
3. **Referencias a docs**: Rutas a DevLogs, Code Logic, Lessons Learned

El agente recibe tanto datos como instrucciones de dónde encontrar más información.

## 6. Flujos Principales

### 6.1 Flujo post-corrida (clasificación)

```
Agent Runtime termina
    │
    │ Guarda last-run.json
    │ Notifica a Memory Service
    │
    ▼
Memory Service (watcher) detecta
    │
    ▼
┌─────────────────────────────────────────┐
│ 1. Lee conversación completa            │
│ 2. Consolida con formato unificado      │
│ 3. Clasifica (reglas determinísticas)   │
│ 4. Extrae entidades, temas, relaciones  │
│ 5. Actualiza índice invertido           │
│ 6. Guarda en memory_db                  │
│ 7. Marca como disponible                │
└─────────────────────────────────────────┘
    │
    ▼
Hook Manager puede consultar contexto actualizado
```

### 6.2 Flujo pre-corrida (consulta de contexto)

```
TL-Agent analiza tarea nueva
    │
    │ Consulta Memory Service:
    │ "Dame contexto relevante para BE-004 (authentication)"
    │
    ▼
Memory Service
    │
    ├─ Busca por topic: "authentication"
    ├─ Busca por task relacionada: "BE-003" (dependencia)
    ├─ Busca por entidad: "AuthService"
    │
    ▼
Retorna:
    │
    ├─ Corridas relevantes (resumen)
    ├─ Turns específicos (detalle cuando aplica)
    └─ Referencias a docs:
         - "Lee: /devlogs/2026-03-16_BE-003.md"
         - "Referencia: /knowledge/code-logic/auth.service.LOGIC.md"
    │
    ▼
TL-Agent arma contexto completo
    │
    │ Memoria filtrada + referencias
    │
    ▼
Hook Manager lanza agente con contexto
```

## 8. Búsqueda y Clasificación

### 8.1 Clasificación por reglas (v1)

La clasificación inicial usa reglas determinísticas:

```javascript
// Ejemplo de reglas
const rules = [
  {
    name: "detect-authentication",
    condition: (run) => {
      const text = run.conversation.map(t => t.content).join(" ");
      return text.match(/auth|login|jwt|token|session/i);
    },
    action: (classification) => {
      classification.topics.push("authentication");
    }
  },
  {
    name: "detect-database",
    condition: (run) => {
      const files = run.result.artifacts || [];
      return files.some(f => f.includes(".prisma") || f.includes("migration"));
    },
    action: (classification) => {
      classification.topics.push("database");
      classification.type = "migration";
    }
  }
];
```

### 8.2 Evolución de clasificación

| Capa | Estado | Descripción |
|------|--------|-------------|
| **1. Reglas determinísticas** | Por implementar | Keywords, patrones, heurísticas |
| **2. LLM pequeño (3B params)** | Futuro | Clasificación más inteligente |
| **3. Graph Search** | Futuro | Relaciones entre entidades |

### 8.3 Búsqueda futura (RAG híbrido)

Para la evolución, se considera:

- **BM25**: Búsqueda por keywords (rápida)
- **Embeddings BGE-M3** (768 dims): Búsqueda semántica
- **Graph**: Navegación por relaciones

```
Query: "cómo se implementó el refresh token?"
    │
    ├─ BM25: busca "refresh token" → runs con esas palabras
    ├─ Embeddings: busca conceptos similares → runs semánticamente relacionados
    └─ Graph: busca entidades conectadas → AuthService → JWT → RefreshToken
    │
    ▼
Combina resultados, rankea, retorna top-K
```

### 10.2 Riesgos identificados

| Riesgo | Impacto | Mitigación |
|--------|---------|------------|
| **Clasificación incorrecta** | Agente recibe contexto irrelevante | Ajuste humano cuando falla + mejora de reglas |
| **Latencia en clasificación** | Agente espera contexto | Clasificación en paralelo, cache de índices |
| **Crecimiento de BD** | Memoria crece sin límite | Políticas de retención, compactación |
| **Contexto muy grande** | Supera límite de tokens | Filtrado más agresivo, priorización |
