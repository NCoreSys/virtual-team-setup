## 1. RESUMEN EJECUTIVO

### 1.1 Objetivo

Implementar tracking completo del consumo de tokens y costos de los agentes LLM para:
- Conocer el costo por tarea, por sesión, por proyecto
- Analizar eficiencia de tokens
- Diferenciar costo de trabajo en tareas vs conversación general
- Establecer presupuestos y alertas
- Proyectar costos futuros

### 1.2 Alcance

| Incluido | Excluido |
|----------|----------|
| Track CLI (Claude Code JSONL) | Track SDK (Python Runtime) — fase posterior |
| Sync manual via endpoint (archivo subido) | Sync automático / cron |
| Inferencia de proyecto desde CWD | Memory / Hook Manager |
| Inferencia de taskIds desde content | Integración con otros providers (solo Anthropic MVP) |
| API de consulta de costos | Dashboard UI (solo API en este módulo) |
| Catálogo de precios de modelos | Agentes manuales |

### 1.3 Exclusiones Explícitas

> ⚠️ **IMPORTANTE:** Este módulo es **SOLO CLI / JSONL**. NO incluye:
> - Memory del sistema de agentes
> - Hook Manager
> - Agentes ejecutados via API
> - Cualquier otra fuente de datos que no sea el JSONL de Claude Code CLI

### 3.5 Nota sobre totalTokens

> ⚠️ **DECISIÓN SEMÁNTICA:** El campo `totalTokens` en `agent_session_lines` se calcula como:
> ```
> totalTokens = inputTokens + outputTokens
> ```
> 
> Los tokens de cache (`cacheCreationTokens`, `cacheReadTokens`) **NO se incluyen** en `totalTokens`.
> 
> **Razón:** Los cache tokens son derivados de input, incluirlos duplicaría el conteo.
> 
> **Impacto:** Las métricas F7 (Total Tokens) y F8 (Tokens per Session) reportan tokens de procesamiento activo, no cache.
> 
> El costo SÍ incluye todos los tipos de tokens (la fórmula de §5.5 es correcta).

### 3.6 Nota sobre BigInt en Responses

> ⚠️ **REQUISITO TÉCNICO:** Los campos `totalTokens` en summaries son `BigInt`.
> 
> JavaScript `BigInt` no es serializable por `JSON.stringify()`. Los controllers DEBEN convertir a `Number` antes de retornar:
> ```typescript
> return {
>   ...summary,
>   totalTokens: Number(summary.totalTokens)
> };
> ```

### 5.2 RF-COST-002: Inferencia de Proyecto desde CWD

**Descripción:** Inferir el código de proyecto desde el path del directorio.

**Lógica:**
```typescript
function inferProjectCode(cwd: string): { code: string | null; confidence: 'high' | 'low' | 'none' } {
  if (!cwd) return { code: null, confidence: 'none' };
  
  const cwdLower = cwd.toLowerCase();
  
  // Mapeo de paths a códigos de proyecto (configurable)
  const projectMappings: Record<string, string> = {
    'designmine': 'DSM',
    'virtual-teams-tracking': 'VTT',
    // Agregar más proyectos según se necesite
  };
  
  for (const [pathFragment, code] of Object.entries(projectMappings)) {
    if (cwdLower.includes(pathFragment)) {
      return { code, confidence: 'high' };
    }
  }
  
  return { code: null, confidence: 'none' };
}
```

**Comportamiento cuando NO se puede identificar el proyecto:**

| Situación | Acción | Valores en BD |
|-----------|--------|---------------|
| CWD vacío o null | Continuar sin proyecto | `projectCode = null`, `projectId = null` |
| CWD no matchea ningún proyecto | Continuar sin proyecto | `projectCode = null`, `projectId = null` |
| Usuario envía `projectCodeHint` | Usar el hint | `projectCode = hint`, buscar `projectId` en BD |

**Importante:** Las líneas sin proyecto identificado **SÍ se insertan** en la BD. Se pueden filtrar después por `projectCode IS NULL`.

### 5.4 RF-COST-004: Extracción de Task IDs desde Content

**Descripción:** Buscar patrones de task ID en el contenido del mensaje.

**Lógica:**
```typescript
function extractTaskIds(text: string, projectCode: string | null): string[] {
  if (!text) return [];
  
  const taskIds = new Set<string>();
  
  // Patrón 1: Código de proyecto + número (DSM-266, VTT-001)
  if (projectCode) {
    const projectPattern = new RegExp(`${projectCode}-\\d+`, 'gi');
    const matches = text.match(projectPattern) || [];
    matches.forEach(m => taskIds.add(m.toUpperCase()));
  }
  
  // Patrón 2: Sprint + Agente + número (S9-BE-01, S7-FE-03)
  const sprintPattern = /S\d+-[A-Z]{2}-\d+/gi;
  const sprintMatches = text.match(sprintPattern) || [];
  sprintMatches.forEach(m => taskIds.add(m.toUpperCase()));
  
  // Patrón 3: Agente + número (FE-001, BE-042, DB-003)
  const agentPattern = /\b(?:FE|BE|DB|QA|TL|DO|SA|AR)-\d{3,4}\b/gi;
  const agentMatches = text.match(agentPattern) || [];
  agentMatches.forEach(m => taskIds.add(m.toUpperCase()));
  
  return [...taskIds];
}
```

**Comportamiento cuando se detectan Task IDs:**

| Situación | Acción | Valores en BD |
|-----------|--------|---------------|
| No se detectan task IDs | Continuar | `taskIds = []`, `hasTaskId = false` |
| Se detecta 1 task ID | Guardar | `taskIds = ['DSM-266']`, `hasTaskId = true` |
| Se detectan múltiples task IDs | **Guardar TODOS** | `taskIds = ['DSM-266', 'DSM-267', 'S9-BE-01']`, `hasTaskId = true` |
| Task ID no existe en tabla `tasks` | **Guardar igual** | `taskIds = ['DSM-999']`, `hasTaskId = true`, `taskIdsValidated = false` |

**Importante:** 
- Los task IDs **NO se validan** contra la tabla `tasks` en tiempo de sync (performance)
- La validación es opcional y puede hacerse después con un endpoint separado
- El campo `taskIdsValidated` indica si ya se verificó la existencia

### 5.5 RF-COST-005: Cálculo de Costo con Precio Default

**Descripción:** Calcular el costo estimado de una línea, usando precio default si no existe el modelo.

**Lógica:**
```typescript
interface CostCalculationResult {
  estimatedCost: number;
  usedDefaultPricing: boolean;
  pricingModelUsed: string;
  breakdown: {
    inputCost: number;
    outputCost: number;
    cacheReadCost: number;
    cacheCreationCost: number;
  };
}

async function calculateLineCost(
  model: string,
  inputTokens: number,
  outputTokens: number,
  cacheReadTokens: number,
  cacheCreationTokens: number
): Promise<CostCalculationResult> {
  
  // 1. Buscar precio del modelo exacto (con lógica temporal)
  let pricing = await getActivePrice('anthropic', model);
  let usedDefaultPricing = false;
  let pricingModelUsed = model;
  
  // 2. Si no existe, usar precio default de BD
  if (!pricing) {
    pricing = await getDefaultFallbackPrice();
    usedDefaultPricing = true;
    pricingModelUsed = pricing.modelName;
    
    // Log warning
    console.warn(`No pricing found for model "${model}", using default: ${pricing.modelName}`);
  }
  
  // 3. Calcular costos
  const inputCost = (inputTokens / 1_000_000) * Number(pricing.inputCostPerMillion);
  const outputCost = (outputTokens / 1_000_000) * Number(pricing.outputCostPerMillion);
  
  const cacheReadPrice = pricing.cacheReadCostPerMillion 
    ? Number(pricing.cacheReadCostPerMillion)
    : Number(pricing.inputCostPerMillion) * 0.1;
  const cacheReadCost = (cacheReadTokens / 1_000_000) * cacheReadPrice;
  
  const cacheCreationPrice = pricing.cacheWriteCostPerMillion
    ? Number(pricing.cacheWriteCostPerMillion)
    : Number(pricing.inputCostPerMillion) * 0.25;
  const cacheCreationCost = (cacheCreationTokens / 1_000_000) * cacheCreationPrice;
  
  const totalCost = inputCost + outputCost + cacheReadCost + cacheCreationCost;
  
  return {
    estimatedCost: round6(totalCost),
    usedDefaultPricing,
    pricingModelUsed,
    breakdown: {
      inputCost: round6(inputCost),
      outputCost: round6(outputCost),
      cacheReadCost: round6(cacheReadCost),
      cacheCreationCost: round6(cacheCreationCost)
    }
  };
}

/**
 * Obtener precio activo para un modelo con lógica temporal correcta.
 * Retorna el precio más reciente vigente.
 */
async function getActivePrice(provider: string, modelName: string): Promise<ModelPricing | null> {
  return prisma.modelPricing.findFirst({
    where: {
      provider,
      modelName,
      isActive: true,
      effectiveFrom: { lte: new Date() },
      OR: [
        { effectiveTo: null },
        { effectiveTo: { gte: new Date() } }
      ]
    },
    orderBy: { effectiveFrom: 'desc' }  // ⚠️ CRÍTICO: precio más reciente vigente
  });
}

/**
 * Obtener precio fallback de BD.
 * ⚠️ NUNCA usa valores hardcodeados — lanza error si no hay fallback configurado.
 */
async function getDefaultFallbackPrice(): Promise<ModelPricing> {
  const fallback = await prisma.modelPricing.findFirst({
    where: { isDefaultFallback: true, isActive: true },
    orderBy: { effectiveFrom: 'desc' }
  });
  
  if (!fallback) {
    // ⚠️ ERROR FATAL: No hay fallback configurado
    throw new Error(
      'FATAL: No default fallback pricing configured in model_pricing table. ' +
      'Run seed task 5F-DB-005 or insert a record with isDefaultFallback=true and isActive=true.'
    );
  }
  
  return fallback;
}
```

**Campos que se guardan cuando se usa precio default:**

| Campo | Valor |
|-------|-------|
| `estimatedCost` | Costo calculado con precio default |
| `usedDefaultPricing` | `true` |
| `pricingModelUsed` | Nombre del modelo fallback (ej: `'claude-sonnet-4-5'`) |

## 10. CRITERIOS DE ACEPTACIÓN

### 10.1 Carga de JSONL

- [ ] Usuario puede subir archivo JSONL via endpoint
- [ ] Archivo de hasta 50MB se procesa correctamente
- [ ] Líneas duplicadas (mismo uuid) no se insertan dos veces
- [ ] Response incluye conteo de líneas procesadas/insertadas/skipped

### 10.2 Inferencia de Proyecto

- [ ] ProjectCode se infiere correctamente de DesignMine → DSM, VTT → VTT
- [ ] Líneas sin proyecto se insertan con projectCode = null
- [ ] Response incluye warning con cantidad de líneas sin proyecto

### 10.3 Extracción de Task IDs

- [ ] Task IDs se extraen de patrones DSM-XXX, S9-BE-XX, FE-XXX
- [ ] Múltiples task IDs en una línea se guardan todos en array
- [ ] Task IDs que no existen en BD se guardan igual (sin validación)

### 10.4 Cálculo de Costo con Default

- [ ] Costo se calcula usando precios de model_pricing
- [ ] Si no existe precio, se usa el modelo marcado como isDefaultFallback
- [ ] Campo usedDefaultPricing = true cuando se usa fallback
- [ ] Campo pricingModelUsed indica qué modelo se usó para el cálculo
- [ ] Response incluye warning con cantidad de líneas con precio default
- [ ] Si no hay fallback en BD, el sistema lanza error (no usa hardcoded)

### 10.5 API de Consultas

- [ ] GET /api/projects/:projectCode/costs retorna datos correctos
- [ ] GET /api/tasks/:taskId/costs retorna líneas asociadas a esa tarea
- [ ] GET /api/costs/unassigned retorna líneas sin proyecto
- [ ] GET /api/costs/default-pricing retorna líneas con precio default
- [ ] Solo PM y admin pueden acceder a todos los endpoints

### 10.6 Model Pricing

- [ ] Solo puede existir un registro con isDefaultFallback=true AND isActive=true
- [ ] Al crear/actualizar un fallback, se desactiva el anterior automáticamente

## 11. NOTAS TÉCNICAS

### 11.1 Mecanismo de Upload

El archivo JSONL NO se lee del filesystem del usuario. Opciones de carga:

1. **Upload de archivo:** `POST /api/costs/sync/upload` con multipart/form-data
2. **Pegar contenido:** `POST /api/costs/sync/text` con JSON body (requiere `limit: '50mb'`)

### 11.2 Consideraciones de Performance

- El JSONL puede tener miles de líneas
- Usar batch inserts (chunks de 100-500 líneas)
- Índices en uuid, sessionId, projectCode, timestamp, usedDefaultPricing
- **Índice GIN en taskIds** para consultas por tarea
- Los summaries se recalculan solo para sessionIds/projectCodes afectados
- Usar `upsert()` para summaries (idempotencia)

### 11.3 Manejo de Errores

- Si una línea tiene JSON inválido, logear y continuar
- Si no se puede inferir proyecto, projectCode = null (no error)
- Si no hay precio para el modelo, usar fallback de BD
- **Si no hay fallback en BD, lanzar error fatal** (no usar hardcoded)
- Retornar lista de warnings y errors en response del sync

### 11.4 Seguridad

- El endpoint de sync debe ser protegido (solo PM/admin)
- No exponer paths del filesystem en responses
- Validar tamaño máximo de archivo (50MB)
- Sanitizar nombres de archivo

### 11.5 BigInt Serialization

Los campos `totalTokens` en summaries son `BigInt`. Convertir a `Number` antes de retornar en JSON:
```typescript
totalTokens: Number(summary.totalTokens)
```
