# rt-api

Ejecutor de workflows HTTP por pasos. Corre como servidor HTTP en `localhost:3005`.
Los agentes lo usan vía `POST /run` o vía Anthropic SDK apuntando el `base_url` al servidor.

---

## Modo Servidor — HTTP API (recomendado para agentes)

El runtime puede correr como servidor HTTP. Los agentes lo llaman via `POST /run`
sin necesidad de subprocess, paths, ni copiar el proyecto.

### Levantar el servidor

Ejecutar desde la carpeta del runtime:

```bash
cd "C:\Users\Martin\Documents\proyectos\Agente - Orquestador\RunTime\rt-api"
npm run serve
# → Escuchando en http://localhost:3005
```

Puerto por defecto: **3005** (para no colisionar con proyectos en 3000/3001).
Cambiar puerto: `PORT=3010 npm run serve`

### Endpoints

| Método | Ruta | Descripción |
|---|---|---|
| `GET` | `/health` | Status del servidor |
| `GET` | `/workflows` | Lista los workflows disponibles |
| `POST` | `/run` | Ejecuta un workflow |

### Llamar desde Python

```python
import requests

result = requests.post("http://localhost:3005/run", json={
    "workflowFile": "qa-inventory-export",   # nombre en workflows/ (sin extensión)
    "inputs": {
        "baseUrl":     "http://mi-api.com",
        "email":       "usuario@test.com",
        "password":    "pass123",
        "inventoryId": "uuid-del-inventory"
    }
}).json()

print(result["success"])          # True / False
print(result["outputs"])          # outputs declarados en el workflow
print(result["artifacts"])        # archivos guardados
print(result["runId"])            # ID del run para trazabilidad
```

### Llamar desde Node.js / fetch

```javascript
const res = await fetch("http://localhost:3005/run", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    workflowFile: "qa-inventory-export",
    inputs: { baseUrl: "...", email: "...", password: "...", inventoryId: "..." }
  })
});
const result = await res.json();
```

### Llamar con curl

```bash
curl -X POST http://localhost:3005/run \
  -H "Content-Type: application/json" \
  -d '{
    "workflowFile": "qa-inventory-export",
    "inputs": {
      "baseUrl": "http://77.42.88.106:8001",
      "email": "qa.e2e.s6@designmine.ai",
      "password": "QAe2eS6Test2026!",
      "inventoryId": "474d71e9-19dd-4996-b4d5-d8a85d0084e9"
    }
  }'
```

### Respuesta

```json
{
  "success": true,
  "workflowId": "wf-qa-inventory-export",
  "runId": "run-1234567890-abc12345",
  "status": "DONE",
  "durationMs": 312,
  "stepsExecuted": 2,
  "outputs": {
    "zipFile": "inventory-export-run-xxx.zip",
    "zipSizeBytes": 14823

  "artifacts": [

  ]

```

En caso de error el runtime retorna `HTTP 422` con `"success": false` y `"errorMessage"`.

## Modo Anthropic SDK — Simula la Claude API

El runtime expone `POST /v1/messages` compatible con el Anthropic SDK.
El agente **no cambia nada** — solo apunta `base_url` al runtime.

### Convención

| Campo Anthropic | Uso en el runtime |
|---|---|
| `model` | Nombre del workflow file (sin extensión) |
| `system` | JSON string con los inputs del workflow |
| `messages[last user].content` | JSON alternativo de inputs (se mergea con `system`) |

### Desde Python con Anthropic SDK

```python
import anthropic, json

client = anthropic.Anthropic(
    api_key="runtime",   # cualquier valor, no se valida
    base_url="http://localhost:3005"
)

msg = client.messages.create(
    model="qa-inventory-export",
    max_tokens=1024,
    system=json.dumps({
        "baseUrl":     "http://77.42.88.106:8001",
        "email":       "qa.e2e.s6@designmine.ai",
        "password":    "QAe2eS6Test2026!",
        "inventoryId": "474d71e9-19dd-4996-b4d5-d8a85d0084e9"
    }),
    messages=[{"role": "user", "content": "run"}]
)

print(msg.content[0].text)
# ✅ Workflow completado
# Run ID   : run-xxx
# Duración : 312ms
# Outputs:
#   zipFile: inventory-export-run-xxx.zip
#   zipSizeBytes: 14823
```

El campo `_runtime` en la respuesta contiene el resultado completo del runtime
por si el agente necesita acceder a detalles adicionales.

```
runs/
└── run-<timestamp>-<id>/
    ├── run.log.json          ← log completo (entries + summary)
    ├── output.json           ← outputs declarados en el workflow
    └── artifacts/
        └── inventory-export-run-xxx.zip
```

## Usar con API real

1. Crear un nuevo archivo de inputs con las credenciales reales:
```json
{
  "baseUrl":  "https://tu-api-real.com",
  "username": "tu-usuario",
  "password": "tu-password"
}
```

2. Ajustar `workflows/login-export-zip.workflow.json` si los endpoints difieren.

3. Ejecutar:
```bash
node index.js --workflow=workflows/login-export-zip.workflow.json --inputs=inputs/real.inputs.json
```

## Módulos

| Módulo | Archivo | Responsabilidad |
|---|---|---|
| WorkflowExecutor | `src/executor/workflowExecutor.js` | Orquestador principal |
| InputResolver | `src/modules/inputResolver.js` | Valida inputs, inicializa context |
| AuthHandler | `src/modules/authHandler.js` | Gestiona token de auth |
| StepRunner | `src/modules/stepRunner.js` | Ejecuta llamadas HTTP |
| ResponseParser | `src/modules/responseParser.js` | Extrae variables de respuestas |
| Validator | `src/modules/validator.js` | Evalúa condiciones de validación |
| ArtifactWriter | `src/modules/artifactWriter.js` | Persiste archivos y outputs |
| RunLogger | `src/modules/runLogger.js` | Registro de trazabilidad |

## Formato de workflow

Ver `workflows/login-export-zip.workflow.json` como referencia.

**Elementos clave:**
- `baseUrl`: puede usar `{{inputs.variable}}`
- `auth.resolveFromStep`: stepId del paso de login
- `steps[].extract`: reglas de extracción (`body.path`, `header.name`, `status`)
- `steps[].validate`: condiciones con `{{variable}} != null`, `{{count}} > 0`, etc.
- `steps[].artifact.validate: "isZip"`: valida magic bytes del ZIP
- `finalValidation.conditions`: validaciones tras completar todos los pasos
- `outputs`: mapa de `key: "{{variable}}"` para el output estructurado

## Limitaciones v0

- Flujo lineal únicamente (no branching, no paralelo)
- Sin retries automáticos
- Auth tipo `bearer` (login-step) y `apikey` estático
- Validación de artifacts: solo magic bytes ZIP
- Sin scheduler, sin UI, sin base de datos
- Condiciones de validación: expresiones JS simples (`==`, `!=`, `>`, `<`, `null`)
