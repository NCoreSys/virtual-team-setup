# server.js — Code Logic

## Archivo
`rt-api/server.js`

## Propósito
Expone el runtime de workflows como un servidor HTTP (Express) en el puerto 3005. Permite que cualquier agente o proceso externo ejecute workflows HTTP mediante llamadas REST, sin necesidad de subprocess ni acceso al sistema de archivos local.

---

## Dependencias
- `express` — HTTP server
- `WorkflowExecutor` (`./src/executor/workflowExecutor`) — motor de ejecución de workflows
- `fs`, `path` — lectura de archivos de workflow desde disco

---

## Endpoints

### GET /health
Verificación de estado del servidor. Retorna `{ status: "ok", runtime: "rt-api", port }`.

### GET /workflows
Lee el directorio `workflows/` y lista los archivos `.workflow.json` disponibles, sin extensión.

### POST /run
Endpoint principal. Recibe un workflow (por nombre de archivo o inline) y sus inputs, ejecuta con `WorkflowExecutor`, y retorna el resultado completo del runtime.

**Flujo:**
1. Validar que venga `workflowFile` o `workflow` inline
2. Si es `workflowFile`: buscar en `workflows/[nombre].workflow.json`
3. Parsear el JSON del workflow
4. Instanciar `WorkflowExecutor` y llamar `.execute(workflowDef, inputs)`
5. Retornar resultado con status 200 (éxito) o 422 (workflow falló pero ejecutó)

### POST /v1/messages
Endpoint compatible con Anthropic Messages API. Permite que agentes que usan el SDK de Anthropic llamen al runtime simplemente cambiando `base_url`, sin modificar su código.

**Convención de mapeo:**
- `model` → nombre del workflow file
- `system` → JSON string con los inputs del workflow
- `messages[last user].content` → JSON alternativo de inputs (se mergea sobre system)

**Flujo:**
1. Extraer inputs de `system` (JSON) y opcionalmente del último mensaje usuario (JSON, mergeado)
2. Buscar y parsear el workflow por nombre (campo `model`)
3. Ejecutar con `WorkflowExecutor`
4. Formatear resultado como respuesta de Claude: `{ role: "assistant", content: [{ type: "text", text }] }`
5. Incluir campo extra `_runtime` con el resultado completo del runtime

---

## Función auxiliar: `_buildResultText(result)`
Convierte el objeto resultado del runtime en texto plano para el campo `content[0].text` de la respuesta de Anthropic Messages API.

- Si falló: muestra error, step fallido, run ID
- Si tuvo éxito: muestra run ID, duración, steps, outputs y artifacts

---

## Decisiones de diseño

| Decisión | Razón |
|----------|-------|
| Puerto 3005 | Evitar conflicto con otros proyectos que usan 3000, 3001, 3100 |
| `workflowFile` acepta nombre sin extensión | Más ergonómico para agentes |
| Status 422 en fallo de workflow | El request fue válido, pero el workflow no completó — 500 se reserva para errores inesperados del server |
| `_runtime` en respuesta de /v1/messages | El SDK de Anthropic solo expone `content[0].text`, pero la app puede acceder al resultado completo si necesita los artifacts o outputs |
| Inline workflow en POST /run | Permite testing rápido sin tener que crear un archivo en disco |
