## Feature: Import

### RF-001 · Import de conversación TASK_EXECUTION

| Campo | Valor |
|-------|-------|
| **Feature** | Import |
| **Endpoint** | `POST /api/conversations/import` |
| **MoSCoW** | Must |
| **Ref. SPEC** | §9.1, D-MEM-14 |

**Descripción:** El sistema debe recibir un payload JSON con los datos de una conversación de tipo TASK_EXECUTION (fuentes: CLAUDE_SDK, CLAUDE_CLI, CLAUDE_WEB, CHATGPT), validarlo con schema Zod, crear el registro en BD con status PENDING, escribir el archivo JSONL en `/storage/`, ejecutar la clasificación determinística, y mover a status IMPORTED. Si cualquier paso falla después de la creación en BD, el catch NO mueve a ERROR — delega al cleanup job (AMB-07 cerrada, SPEC v1.9 §9.1).

---

### RF-002 · Idempotencia de import

| Campo | Valor |
|-------|-------|
| **Feature** | Import |
| **Endpoint** | `POST /api/conversations/import` |
| **MoSCoW** | Must |
| **Ref. SPEC** | D-MEM-05, D-MEM-42, DB-OBS-08 |

**Descripción:** El sistema debe verificar idempotencia mediante el constraint `@@unique([sourceId, externalSessionId])`. Si la combinación ya existe con status IMPORTED, retorna `{ status: "ALREADY_INDEXED", conversationId }` sin duplicar datos. En caso de race condition (error Prisma P2002), el sistema debe capturar el error, consultar el registro existente y retornar ALREADY_INDEXED.

---

### RF-003 · Validación de payload de import

| Campo | Valor |
|-------|-------|
| **Feature** | Import |
| **Endpoint** | `POST /api/conversations/import` |
| **MoSCoW** | Must |
| **Ref. SPEC** | §9.1 |

**Descripción:** El sistema debe validar el payload de entrada con schema Zod. Campos obligatorios: `sourceCode`, `externalSessionId`, `jsonlContent` (o referencia al archivo). Campos opcionales con defaults: `projectId`, `taskId`, `primaryAgentId`, `agentRole`, `platformRefs`. Si la validación falla, retorna HTTP 400 con detalle del error Zod.

---

### RF-004 · Escritura de archivo JSONL en storage

| Campo | Valor |
|-------|-------|
| **Feature** | Import |
| **Endpoint** | `POST /api/conversations/import`, `POST /api/conversations/upload` |
| **MoSCoW** | Must |
| **Ref. SPEC** | §13, D-MEM-18, D-MEM-25 |

**Descripción:** El sistema debe escribir el contenido JSONL completo en el filesystem bajo `/storage/{sourceCode}/{conversationId}.jsonl`. La BD almacena solo metadata y `contentPreview` (500 chars del primer turno de asistente). El archivo en storage es la fuente de verdad para el contenido completo.

---

### RF-005 · Clasificación determinística post-import

| Campo | Valor |
|-------|-------|
| **Feature** | Import |
| **Endpoint** | `POST /api/conversations/import` |
| **MoSCoW** | Must |
| **Ref. SPEC** | §12, D-MEM-08, D-MEM-20 |

**Descripción:** Después de persistir la conversación, el sistema debe clasificarla por reglas determinísticas (pattern matching en contenido). El clasificador asigna: `workType` (via tabla Classification), `topics` (via tabla ConversationTopic, relación N:M) y `entities` (via tabla ConversationEntity con constraint `@@unique([conversationId, entityName])` — DB-OBS-01). Sin ML, sin embeddings, sin falsos positivos impredecibles.

---

### RF-006 · Import de conversación AGENT_REVIEW/CLARIFICATION

| Campo | Valor |
|-------|-------|
| **Feature** | Import |
| **Endpoint** | `POST /api/conversations/import-review` |
| **MoSCoW** | Must |
| **Ref. SPEC** | §9.2, D-MEM-14, D-MEM-15, D-MEM-29 |

**Descripción:** El sistema debe recibir un payload JSON con los datos de una conversación de tipo AGENT_REVIEW o AGENT_CLARIFICATION (fuente: VTT_CHANNEL). El payload incluye un array de `participants` (ConversationParticipant con agentId, role y platformId) y un array de `messages` (AgentMessage con messageType, messageStatus, priority, content, timestamp). El `primaryAgentId` es NULL para estos tipos. El archivo se almacena en `/storage/_reviews/`.

---

### RF-007 · Import incremental de VTT_CHANNEL

| Campo | Valor |
|-------|-------|
| **Feature** | Import |
| **Endpoint** | `POST /api/conversations/import-review` |
| **MoSCoW** | Must |
| **Ref. SPEC** | D-MEM-16, AMB-02 (cerrada en SRS) |

**Descripción:** Si la conversación VTT_CHANNEL ya existe (mismo `externalSessionId`), el sistema debe ejecutar import incremental: append-only para mensajes nuevos, upsert para participantes (agregar nuevos, no modificar existentes), actualización de campos de conversación (`title`, `summary`) con valores del último import. Retorna `{ status: "UPDATED", conversationId, newMessages: N }`.

---

### RF-008 · Resolución de catálogos desde Maps pre-cargados

| Campo | Valor |
|-------|-------|
| **Feature** | Import |
| **Endpoint** | `POST /api/conversations/import`, `POST /api/conversations/import-review` |
| **MoSCoW** | Must |
| **Ref. SPEC** | §3.3, §9.2, DEC-01, DB-OBS-03 |

**Descripción:** Los flujos de import deben resolver IDs de catálogo (sourceCode→sourceId, messageType→messageTypeId, etc.) desde Maps pre-cargados en memoria, no con queries individuales dentro de loops de procesamiento. Esto elimina el problema N+1 documentado en DB-OBS-03. Si un código de catálogo no existe en el Map, el import falla con error descriptivo.

---

### RF-009 · Upload manual de conversaciones

| Campo | Valor |
|-------|-------|
| **Feature** | Import |
| **Endpoint** | `POST /api/conversations/upload` |
| **MoSCoW** | Should |
| **Ref. SPEC** | §1.2 |

**Descripción:** El sistema debe permitir upload manual de archivos de conversación desde la UI. Soporta formatos de exportación de claude.ai (CLAUDE_WEB) y ChatGPT (CHATGPT). El flujo es: validar formato → crear conversación con status PENDING → escribir archivo → clasificar → mover a IMPORTED. Comparte la lógica de idempotencia de RF-002.

---

### RF-010 · Autenticación SERVICE_KEY en endpoints de import

| Campo | Valor |
|-------|-------|
| **Feature** | Import |
| **Endpoint** | `POST /api/conversations/import`, `POST /api/conversations/import-review` |
| **MoSCoW** | Must |
| **Ref. SPEC** | §7, D-MEM-26 |

**Descripción:** Los endpoints de import (`POST /import`, `POST /import-review`) deben requerir autenticación via header `Authorization: Bearer {SERVICE_KEY}`. Si la key es inválida o ausente, retorna HTTP 401. `POST /upload` y los endpoints GET son públicos en R1 (KICKOFF §5.4).

## Feature: Context

### RF-011 · Provisión de contexto runtime

| Campo | Valor |
|-------|-------|
| **Feature** | Context |
| **Endpoint** | `GET /api/context` |
| **MoSCoW** | Must |
| **Ref. SPEC** | §11, D-MEM-07, D-MEM-37, ADDENDUM §4 |

**Descripción:** El sistema debe retornar un objeto JSON de contexto estructurado para un agente en un proyecto. Query parameters obligatorios: `projectId`, `agentId`. El response incluye 6 secciones: `recentSessions` (últimas N conversaciones del agente en el proyecto), `recentDecisions` (topics frecuentes), `frequentFiles` (archivos más usados via TurnBlock), `recentReviews` (últimas revisiones donde el agente participó), `costSummary` (tokens totales del agente) y `projectContext` (stats del proyecto).

---

### RF-012 · SLA <500ms fail-fast

| Campo | Valor |
|-------|-------|
| **Feature** | Context |
| **Endpoint** | `GET /api/context` |
| **MoSCoW** | Must |
| **Ref. SPEC** | §11, D-MEM-07, D-MEM-37, AR Review §3.3 |

**Descripción:** `GET /context` DEBE responder en menos de 500ms (p95). La implementación ejecuta las 6 queries en paralelo con `Promise.all` envuelto en `Promise.race` con timeout de 500ms. Si el timeout se alcanza, retorna error `MEM-ERR-504` inmediatamente. Es fail-fast, no degradado: no retorna resultados parciales.

---

### RF-013 · Autenticación SERVICE_KEY en contexto

| Campo | Valor |
|-------|-------|
| **Feature** | Context |
| **Endpoint** | `GET /api/context` |
| **MoSCoW** | Must |
| **Ref. SPEC** | §7, D-MEM-26 |

**Descripción:** `GET /context` requiere autenticación via SERVICE_KEY porque es consumido por Prompt Builder v1.3 (sistema, no usuario humano). Si la key es inválida, retorna HTTP 401.

---

### RF-014 · Filtrado de conversaciones IMPORTED en contexto

| Campo | Valor |
|-------|-------|
| **Feature** | Context |
| **Endpoint** | `GET /api/context` |
| **MoSCoW** | Must |
| **Ref. SPEC** | §11 |

**Descripción:** Todas las queries de contexto deben filtrar solo conversaciones con status IMPORTED. Las conversaciones en PENDING, PROCESSING o ERROR no aparecen en el contexto runtime.

---

### RF-015 · Índice compuesto para contexto

| Campo | Valor |
|-------|-------|
| **Feature** | Context |
| **Endpoint** | `GET /api/context` |
| **MoSCoW** | Must |
| **Ref. SPEC** | §6.1, DB-OBS-05 |

**Descripción:** El sistema debe disponer del índice compuesto `(primaryAgentId, projectId, startedAt DESC)` en la tabla Conversation para que las queries de `recentSessions` y `frequentFiles` cumplan el SLA <500ms. Este índice es requisito de performance validado por DB Review como impacto directo en el cumplimiento del SLA.

## Feature: Content & Search

### RF-016 · Lectura de contenido completo de conversación

| Campo | Valor |
|-------|-------|
| **Feature** | Content & Search |
| **Endpoint** | `GET /api/conversations/:id/content` |
| **MoSCoW** | Must |
| **Ref. SPEC** | §1.2, D-MEM-43, AMB-01 (cerrada) |

**Descripción:** El sistema debe retornar el contenido completo de una conversación leyendo el archivo JSONL desde `/storage/{sourceCode}/{conversationId}.jsonl`. NO lee el contenido de la BD — la BD solo tiene `contentPreview`. Si el archivo no existe, retorna HTTP 404 con error `MEM-ERR-404`. Para conversaciones VTT_CHANNEL, lee desde `/storage/_reviews/`.

---

### RF-017 · Listado de conversaciones con filtros

| Campo | Valor |
|-------|-------|
| **Feature** | Content & Search |
| **Endpoint** | `GET /api/conversations` |
| **MoSCoW** | Must |
| **Ref. SPEC** | §1.2 |

**Descripción:** El sistema debe retornar una lista paginada de conversaciones con filtros opcionales: `projectId`, `primaryAgentId`, `sourceCode`, `statusCode`, `conversationTypeCode`, `taskId`. Paginación por cursor (`cursor` + `limit`, default 20, max 100). Cada item incluye metadata (id, title, source, status, type, startedAt, contentPreview) pero NO el contenido completo.

---

### RF-018 · Retorno de metadata de conversación

| Campo | Valor |
|-------|-------|

**Descripción:** Cada conversación en el listado debe incluir: turnCount, totalTokens (input+output de ConversationUsage), classification (workType), topics (array de codes), y contentPreview (500 chars). Estos datos provienen de las relaciones en BD, no del archivo en storage.

### RF-020 · Filtrado por status IMPORTED por defecto

### RF-030 · Health check

| Campo | Valor |
|-------|-------|
| **Feature** | Dashboard & Ops |

### RF-031 · Cleanup job de recovery

| Campo | Valor |
|-------|-------|
| **Feature** | Dashboard & Ops |
| **Endpoint** | Job interno (no endpoint) |
| **MoSCoW** | Must |
| **Ref. SPEC** | §10.2, D-MEM-35, AMB-04 (cerrada), DB-OBS-02 |

**Descripción:** El sistema debe ejecutar un job cron cada 5 minutos que: busca conversaciones en PENDING o PROCESSING con `updatedAt` > 10 minutos de antigüedad, usando `statusId` directo (no `status.code` — DB-OBS-02). Si retryCount < 3, resetea a PENDING e incrementa retryCount. Si retryCount >= 3, mueve a ERROR con `errorMessage: "Max retries exceeded"`. Si la conversación está en PROCESSING pero el archivo no existe en `/storage/`, mueve a ERROR directamente sin retry (AMB-04 cerrada).

---

### RF-032 · Pre-carga de catálogos en startup

| Campo | Valor |
|-------|-------|
| **Feature** | Dashboard & Ops |
| **Endpoint** | Inicialización del sistema |
| **MoSCoW** | Must |
| **Ref. SPEC** | §3.3, DEC-01, DB-OBS-03 |

**Descripción:** Al iniciar el servicio, el sistema debe pre-cargar todos los catálogos (10 tablas) en memoria como Maps `(code → id)`. Estos Maps se usan en los flujos de import para resolver IDs sin queries individuales. Si un catálogo no puede cargarse, el servicio debe fallar en startup con error descriptivo (fail-fast).

---

### RF-033 · Validación de constraint único en ConversationEntity

| Campo | Valor |
|-------|-------|
| **Feature** | Dashboard & Ops |
| **Endpoint** | Clasificación post-import |
| **MoSCoW** | Must |
| **Ref. SPEC** | §4.1, DB-OBS-01 |

**Descripción:** La tabla ConversationEntity debe tener el constraint `@@unique([conversationId, entityName])` para prevenir duplicados en caso de retry (cleanup resetea PROCESSING → PENDING → reimport). El clasificador debe usar upsert o ignorar errores de duplicado al insertar entidades.
