## Should Have (7 RFs)

Requisitos importantes que mejoran significativamente el sistema pero cuya ausencia no lo hace inoperante.

| RF | Endpoint / Componente | Descripción | Justificación Should |
|----|-----------------------|-------------|---------------------|
| RF-009 | POST /upload | Upload manual | Flujo de baja frecuencia; import automático (RF-001) cubre el caso principal |
| RF-018 | GET /conversations | Metadata enriquecida (turnCount, tokens, topics) | Mejora UX pero el listado básico funciona sin ella |
| RF-019 | GET /conversations | Ordenamiento por startedAt DESC | Default razonable pero no bloquea funcionalidad |
| RF-022 | GET /timeline | Filtros de timeline (proyecto, tipo, fechas) | Timeline básico funciona sin filtros |
| RF-023 | GET /timeline | Metadata de timeline entry | Mejora visualización pero no bloquea |
| RF-029 | GET /dashboard/stats | Índice en importedAt | Performance, no funcionalidad |
| RF-032 | Inicialización | Pre-carga de catálogos en startup | Must conceptualmente pero clasificado como Should porque el sistema puede funcionar con queries individuales (lento pero funcional) |

> **Nota sobre RF-032:** La pre-carga de catálogos es técnicamente Must para performance, pero funcionalmente el sistema opera (lentamente) sin ella. Se clasifica como Should para reflejar que no es un blocker funcional absoluto. Sin embargo, la implementación DEBE incluirlo para cumplir DB-OBS-03.
