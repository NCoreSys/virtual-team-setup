## 1. Scope

Este modelo de casos de uso describe el comportamiento observable de Memory Service R1 desde la perspectiva de sus 5 actores. Cubre los 11 endpoints REST del scope fijo (SPEC v1.9 §1.2) más el cleanup job interno, traduciendo los 33 RFs de MS-018 en interacciones actor-sistema.

El modelo NO cubre componentes internos de implementación (arquitectura de módulos, estructura de código) ni sistemas fuera del boundary de Memory Service (cómo Runtime orquesta agentes, cómo PB ensambla prompts).

## 4. Metodología

Los casos de uso siguen el formato de Cockburn simplificado:

- **ID:** UC-XXX (numeración secuencial)
- **Nombre:** Verbo + objeto
- **Actor principal:** Quién inicia la interacción
- **Precondiciones:** Estado del sistema antes del UC
- **Flujo principal:** Secuencia numerada de pasos (happy path)
- **Flujos alternativos:** Ramificaciones con ID (A, B, C...) referenciando el paso donde divergen
- **Postcondiciones:** Estado del sistema después del UC
- **RFs relacionados:** Trazabilidad directa a requisitos funcionales de MS-018

Las relaciones entre UCs usan notación UML estándar: `<<include>>` (siempre se ejecuta), `<<extend>>` (se ejecuta condicionalmente), generalización (herencia de comportamiento).

## 5. Resolución de Ambigüedades

### AMB-UC-01 — Actores de endpoints públicos

**Resolución:** Los endpoints GET sin SERVICE_KEY (`GET /conversations`, `GET /content`, `GET /timeline`, `GET /cost-report`, `GET /dashboard/stats`) tienen como actor primario **UI User** en R1. Sistemas externos no están autorizados a consumirlos directamente en R1 — si un sistema necesita datos de conversaciones, usa `GET /context` (protegido con SERVICE_KEY). En R2, con JWT, estos endpoints podrán tener actores diferenciados.

### AMB-UC-02 — Catch en import delega siempre al cleanup

**Resolución:** Consistente con MS-018 SRS §2 y TL Review AMB-07: el catch del import NO mueve directamente a ERROR. El estado queda en PROCESSING y el cleanup cron reintenta en <10 minutos. ERROR solo ocurre cuando se excede MAX_RETRIES (3) o cuando el archivo no existe en storage (AMB-04 de MS-018).

### AMB-UC-03 — Import incremental actualiza participantes

**Resolución:** Consistente con MS-018 SRS §2 (AMB-02): import incremental de VTT_CHANNEL hace upsert de participantes (agrega nuevos, no modifica existentes) y append-only de mensajes.

### AMB-UC-04 — GET /content lee de storage

**Resolución:** Consistente con MS-018 SRS §2 (AMB-01): GET /content parsea el JSONL completo desde `/storage/`. La BD solo tiene `contentPreview` (500 chars). D-MEM-43.

### AMB-UC-05 — Ventana de "agente activo"

**Resolución:** Consistente con MS-018 SRS §2 (AMB-03): 30 días. Confirmado por PM en sesión de MS-020.
