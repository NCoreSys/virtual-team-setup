## UC-003 — Importar conversación CLAUDE_SDK

| Campo | Valor |
|-------|-------|
| **Actor principal** | Runtime v1.1 (A1) |
| **Endpoint** | `POST /api/conversations/import` |
| **RFs** | RF-001, RF-002, RF-003, RF-004, RF-005, RF-008, RF-010 |
| **Includes** | UC-005 (idempotencia) |
| **Extends** | UC-021 (race condition P2002) |

**Precondiciones:**
- SERVICE_KEY válida configurada en Runtime
- `externalSessionId` con formato `{run_id}:r{N}:{agentRole}` (D-INT-01)
- Payload con `sourceCode: CLAUDE_SDK`, JSONL content y `platformRefs`

**Flujo principal:**
1. Runtime envía `POST /import` con payload JSON, header `Authorization: Bearer {SERVICE_KEY}`
2. Sistema valida SERVICE_KEY → válida
3. Sistema valida payload con schema Zod → válido
4. Sistema resuelve `sourceCode` → `sourceId` desde Maps pre-cargados (RF-008)
5. Sistema verifica idempotencia: `findUnique` con `[sourceId, externalSessionId]` → no existe
6. Sistema crea registro Conversation con status PENDING
7. Sistema transiciona a PROCESSING
8. Sistema escribe JSONL completo en `/storage/CLAUDE_SDK/{conversationId}.jsonl`
9. Sistema parsea JSONL: extrae turns, blocks, calcula `contentPreview` (500 chars del primer turno asistente)
10. Sistema persiste ConversationTurn, TurnBlock, ConversationUsage en transacción
11. Sistema ejecuta clasificación determinística: asigna workType (Classification), topics (ConversationTopic), entities (ConversationEntity con upsert — DB-OBS-01)
12. Sistema transiciona a IMPORTED
13. Sistema retorna `{ status: "IMPORTED", conversationId }` con HTTP 201

**Flujo alternativo A — Idempotencia (paso 5):**
5a. `findUnique` encuentra registro existente con status IMPORTED
5b. Sistema retorna `{ status: "ALREADY_INDEXED", conversationId }` con HTTP 200
5c. No se crea registro ni se escribe archivo

**Flujo alternativo B — Race condition P2002 (paso 6):**
6a. Dos requests concurrentes pasan el check de idempotencia simultáneamente
6b. El segundo `create` falla con error Prisma P2002 (unique constraint violation)
6c. Sistema captura P2002, consulta registro existente via `findUnique`
6d. Retorna `{ status: "ALREADY_INDEXED", conversationId }` con HTTP 200

**Flujo alternativo C — Error de procesamiento (pasos 8-11):**
8c. Error durante escritura, parseo o clasificación
8d. Estado queda en PROCESSING (NO se mueve a ERROR directamente — AMB-UC-02)
8e. El cleanup cron (UC-020) detecta el registro stuck en <10 minutos y reintenta
8f. Si falla después de 3 reintentos → ERROR con `errorMessage: "Max retries exceeded"`

**Flujo alternativo D — SERVICE_KEY inválida (paso 2):**
2d. SERVICE_KEY ausente o inválida → HTTP 401

**Flujo alternativo E — Payload inválido (paso 3):**
3e. Validación Zod falla → HTTP 400 con detalle del error

**Postcondiciones (éxito):**
- Conversation con status IMPORTED en BD
- Archivo JSONL en `/storage/CLAUDE_SDK/{conversationId}.jsonl`
- Classification, ConversationTopic, ConversationEntity creados
- ConversationTurn, TurnBlock, ConversationUsage persistidos

## UC-006 — Importar conversación VTT_CHANNEL

| Campo | Valor |
|-------|-------|
| **Actor principal** | Hook Manager VTT (A3) |
| **Endpoint** | `POST /api/conversations/import-review` |
| **RFs** | RF-006, RF-007, RF-008, RF-010 |

**Precondiciones:**
- SERVICE_KEY válida
- Canal VTT_CHANNEL con `conversationType: AGENT_REVIEW` o `AGENT_CLARIFICATION`
- Payload con `participants[]` y `messages[]`

**Flujo principal (primer import):**
1. Hook Manager envía `POST /import-review` con payload, header `Authorization: Bearer {SERVICE_KEY}`
2. Sistema valida SERVICE_KEY → válida
3. Sistema valida payload con Zod
4. Sistema resuelve catálogos desde Maps pre-cargados (messageType, messageStatus, priority, platform — RF-008, DB-OBS-03)
5. Sistema verifica idempotencia: `[sourceId=VTT_CHANNEL, externalSessionId]` → no existe
6. Sistema crea Conversation con `primaryAgentId: null`, `conversationType: AGENT_REVIEW`, status PENDING
7. Sistema crea ConversationParticipant para cada participante
8. Sistema crea AgentMessage para cada mensaje
9. Sistema escribe archivo en `/storage/_reviews/{conversationId}.jsonl`
10. Sistema transiciona a IMPORTED
11. Retorna `{ status: "IMPORTED", conversationId }`

**Flujo alternativo A — Import incremental (paso 5, D-MEM-16, AMB-UC-03):**
5a. Registro existe → import incremental
5b. Sistema hace upsert de participantes: agrega nuevos, no modifica existentes
5c. Sistema hace append-only de mensajes nuevos (AgentMessage)
5d. Sistema actualiza campos de conversación (`title`, `summary`) con valores del último import
5e. Sistema actualiza archivo en `/storage/_reviews/`
5f. Retorna `{ status: "UPDATED", conversationId, newMessages: N }`

**Flujo alternativo B — Error de procesamiento:**
(Mismo patrón que UC-003 flujo C: queda en PROCESSING, cleanup reintenta)

**Postcondiciones (primer import):**
- Conversation con status IMPORTED, primaryAgentId null
- ConversationParticipant para cada participante
- AgentMessage para cada mensaje
- Archivo en `/storage/_reviews/`

## UC-008 — Obtener contexto runtime

| Campo | Valor |
|-------|-------|
| **Actor principal** | Prompt Builder v1.3 (A2) |
| **Endpoint** | `GET /api/context` |
| **RFs** | RF-011, RF-013, RF-014 |
| **Extends** | UC-009 (timeout) |

**Precondiciones:**
- SERVICE_KEY válida
- `projectId` y `agentId` en query parameters

**Flujo principal:**
1. Prompt Builder envía `GET /context?projectId=X&agentId=Y` con SERVICE_KEY
2. Sistema valida SERVICE_KEY → válida
3. Sistema inicia timer (500ms deadline)
4. Sistema lanza 6 queries en paralelo (`Promise.all`), todas filtradas por status IMPORTED (RF-014):
   - `recentSessions`: últimas N conversaciones del agente en el proyecto (usa índice `[primaryAgentId, projectId, startedAt DESC]`)
   - `recentDecisions`: topics frecuentes del agente
   - `frequentFiles`: archivos más usados via TurnBlock groupBy
   - `recentReviews`: últimas revisiones donde el agente es participante
   - `costSummary`: tokens totales del agente en el proyecto
   - `projectContext`: estadísticas del proyecto (total conversaciones, agentes activos)
5. Todas las queries responden dentro de 500ms
6. Sistema retorna JSON estructurado con las 6 secciones → HTTP 200

**Postcondiciones:**
- PB recibe contexto completo para ensamblar prompt
- Ningún dato fue modificado (operación read-only)

## UC-009 — Obtener contexto con timeout (fail-fast)

| Campo | Valor |
|-------|-------|
| **Actor principal** | Prompt Builder v1.3 (A2) |
| **Endpoint** | `GET /api/context` |
| **RFs** | RF-011, RF-012, RF-013 |
| **Extends** | UC-008 |

**Precondiciones:**
- Mismas que UC-008
- El sistema está bajo carga o las queries son lentas

**Flujo principal:**
1. Prompt Builder envía `GET /context` con SERVICE_KEY
2. Sistema valida SERVICE_KEY → válida
3. Sistema inicia `Promise.race` entre:
   - `Promise.all([6 queries])` — las queries de contexto
   - Timeout de 500ms
4. El timeout de 500ms se alcanza antes de que las queries completen
5. Sistema retorna inmediatamente HTTP 504 con `{ error: "MEM-ERR-504", message: "Context timeout exceeded" }`

**Comportamiento explícito (D-MEM-07, D-MEM-37):**
- NO hay retry automático
- NO hay modo degradado (no retorna resultados parciales)
- NO hay cache fallback
- Es fail-fast: 504 inmediato, el caller decide qué hacer

**Postcondiciones:**
- PB recibe error 504
- PB ensambla prompt sin contexto histórico (responsabilidad de PB, no de Memory Service)
- Las queries en vuelo se cancelan o se descartan

## UC-011 — Leer contenido completo de conversación

| Campo | Valor |
|-------|-------|
| **Actor principal** | UI User (A4) |
| **Endpoint** | `GET /api/conversations/:id/content` |
| **RFs** | RF-016 |

**Precondiciones:**
- Conversación existe en BD con status IMPORTED
- Archivo JSONL existe en `/storage/`

**Flujo principal (AMB-UC-04 resuelta, D-MEM-43):**
1. UI User navega a detalle de conversación en la UI
2. UI envía `GET /conversations/{id}/content`
3. Sistema busca metadata de la conversación en BD (sourceCode, conversationId)
4. Sistema resuelve ruta del archivo: `/storage/{sourceCode}/{conversationId}.jsonl` (o `/storage/_reviews/` para VTT_CHANNEL)
5. Sistema lee el archivo JSONL completo del filesystem
6. Sistema retorna el contenido completo al cliente → HTTP 200

**Comportamiento explícito:**
- La BD solo almacena `contentPreview` (500 chars del primer turno de asistente — D-MEM-18)
- GET /content NO lee contenido de la BD — lee del filesystem (D-MEM-43)
- El response incluye el contenido completo de todos los turns, no solo el preview

**Flujo alternativo A — Archivo no existe (paso 5):**
5a. Archivo no encontrado en `/storage/`
5b. Sistema retorna HTTP 404 con `{ error: "MEM-ERR-404", message: "Content file not found" }`

**Flujo alternativo B — Conversación no existe (paso 3):**
3b. ID no encontrado en BD → HTTP 404

**Postcondiciones:**
- UI User visualiza el contenido completo de la conversación
- Ningún dato fue modificado

## UC-020 — Recuperar conversaciones stuck (cleanup)

| Campo | Valor |
|-------|-------|
| **Actor principal** | Sistema interno (A6) |
| **Trigger** | Cron job cada 5 minutos (D-MEM-35) |
| **RFs** | RF-031 |

**Precondiciones:**
- El servicio está corriendo
- El cron job está configurado (intervalo: 5 minutos)

**Flujo principal:**
1. Timer dispara el cleanup job
2. Sistema busca conversaciones con `statusId` en [PENDING, PROCESSING] y `updatedAt` > 10 minutos de antigüedad, usando `statusId` directo (DB-OBS-02, no `status.code`)
3. Para cada conversación encontrada:

   **3a. Si `retryCount` < 3 (MAX_RETRIES):**
   - Sistema resetea status a PENDING
   - Sistema incrementa `retryCount` en 1
   - Sistema limpia `errorMessage`
   - La conversación será reprocesada en el siguiente ciclo de import

   **3b. Si `retryCount` >= 3:**
   - Sistema mueve a ERROR
   - Sistema establece `errorMessage: "Max retries exceeded after 3 attempts"`

   **3c. Si status es PROCESSING pero archivo no existe en `/storage/` (AMB-04 de MS-018):**
   - Sistema mueve directamente a ERROR
   - Sistema establece `errorMessage: "Storage file missing after processing"`
   - NO incrementa retryCount (no tiene sentido reintentar sin archivo)

4. Sistema registra en logs: cantidad procesada, resets, errores

**Postcondiciones:**
- Conversaciones recuperables vuelven a PENDING para reintento
- Conversaciones irrecuperables quedan en ERROR con mensaje descriptivo
- No hay conversaciones stuck indefinidamente en PROCESSING

## UC-021 — Manejar race condition en import

| Campo | Valor |
|-------|-------|
| **Actor principal** | Runtime v1.1 (A1) |
| **Endpoint** | `POST /api/conversations/import` |
| **RFs** | RF-002 |
| **Extends** | UC-003, UC-001, UC-002, UC-004 |

**Precondiciones:**
- Dos requests concurrentes con el mismo `[sourceId, externalSessionId]`
- Ambas pasan el check de `findUnique` (paso 5 de UC-003) simultáneamente

**Flujo principal (DB-OBS-08):**
1. Request A y Request B llegan concurrentemente con mismo `externalSessionId`
2. Ambas ejecutan `findUnique` → no encuentran registro (race window)
3. Request A ejecuta `create` → éxito, Conversation creada
4. Request B ejecuta `create` → falla con error Prisma P2002 (unique constraint violation)
5. Request B captura P2002
6. Request B ejecuta `findUnique` → encuentra el registro creado por Request A
7. Request B retorna `{ status: "ALREADY_INDEXED", conversationId }` con HTTP 200

**Postcondiciones:**
- Solo existe un registro en BD (el de Request A)
- Solo existe un archivo en `/storage/`
- Ambos callers reciben una respuesta válida (ninguno falla con error)
- La idempotencia se mantiene incluso bajo concurrencia
