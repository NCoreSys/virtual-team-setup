## §1 · Runtime v1.1

| Campo | Valor |
|-------|-------|
| **ID** | A1 |
| **Tipo** | Sistema externo |
| **Autenticación** | SERVICE_KEY (Bearer) |
| **Endpoints** | `POST /api/conversations/import` |

**Descripción:** Orquestador de agentes de IA del ecosistema VTT. Ejecuta agentes (TL, BE, FE, DB, QA, SA, etc.) en rondas secuenciales. Al terminar cada ronda de cada agente, Runtime envía la conversación completa a Memory Service via `POST /import`.

**Responsabilidades:**
- Enviar la conversación en formato JSONL con `sourceCode: CLAUDE_SDK` al terminar cada ronda
- Construir `externalSessionId` con formato `{run_id}:r{N}:{agentRole}` (ADDENDUM §2, D-INT-01)
- Incluir `platformRefs` con `runtime_run_id`, `round` y `orchestrated: true` (D-INT-03)
- No reintentar si recibe ALREADY_INDEXED — la conversación ya está persistida

**Contrato:** ADDENDUM_INTEGRACION v1.1 §2

## §2 · Prompt Builder v1.3

| Campo | Valor |
|-------|-------|
| **ID** | A2 |
| **Tipo** | Sistema externo |
| **Autenticación** | SERVICE_KEY (Bearer) |
| **Endpoints** | `GET /api/context` |

**Descripción:** Ensamblador de prompts del ecosistema VTT. Antes de cada invocación de agente, consulta Memory Service para obtener contexto histórico relevante y lo transforma en texto tokenizado via su Memory Context Adapter.

**Responsabilidades:**
- Consultar `GET /context?projectId=X&agentId=Y` con SERVICE_KEY
- Aceptar respuesta en <500ms o manejar error `MEM-ERR-504` (timeout)
- Transformar el JSON de contexto a texto tokenizado (responsabilidad de PB, no de Memory Service — D-INT-05)
- No cachear el contexto externamente — cada invocación consulta datos frescos

**Contrato:** ADDENDUM_INTEGRACION v1.1 §4

## §3 · Hook Manager VTT

| Campo | Valor |
|-------|-------|
| **ID** | A3 |
| **Tipo** | Sistema externo |
| **Autenticación** | SERVICE_KEY (Bearer) |
| **Endpoints** | `POST /api/conversations/import-review` |

**Descripción:** Componente de VTT que gestiona canales de comunicación inter-agente (VTT_CHANNEL). Cuando se crea o actualiza un canal de tipo AGENT_REVIEW o AGENT_CLARIFICATION, Hook Manager envía los datos a Memory Service.

**Responsabilidades:**
- Enviar conversaciones multi-agente via `POST /import-review` con `sourceCode: VTT_CHANNEL`
- Incluir array de `participants` (ConversationParticipant) y `messages` (AgentMessage)
- Soportar import incremental: enviar solo mensajes nuevos cuando el canal se actualiza (D-MEM-16)
- El `primaryAgentId` es NULL para conversaciones multi-agente (D-MEM-15)

**Contrato:** ADDENDUM_INTEGRACION v1.1 §3

## §4 · UI User

| Campo | Valor |
|-------|-------|
| **ID** | A4 |
| **Tipo** | Humano |
| **Autenticación** | Ninguna (endpoints públicos en R1) |
| **Endpoints** | `POST /upload`, `GET /conversations`, `GET /content`, `GET /timeline`, `GET /cost-report` (x2), `GET /dashboard/stats` |

**Descripción:** Cualquier miembro del equipo VTT (PM, TL, SA, AR, QA) que interactúa con Memory Service a través de la UI Standalone (React, puerto 3003). Consulta el historial de agentes, revisa costos, sube conversaciones manuales y navega el dashboard.

**Responsabilidades:**
- Subir conversaciones históricas de claude.ai o ChatGPT via `POST /upload`
- Consultar timeline de actividad de agentes
- Revisar cost-reports por proyecto y agente
- Consultar el dashboard de estadísticas globales
- Leer el contenido completo de conversaciones individuales

**Nota (AMB-UC-01):** En R1, los endpoints GET son públicos (sin SERVICE_KEY). El actor UI User es el único actor de estos endpoints. Sistemas externos que necesiten datos usan `GET /context` (con SERVICE_KEY). En R2 con JWT, estos endpoints tendrán control de acceso diferenciado.

## §5 · Admin / DO

| Campo | Valor |
|-------|-------|
| **ID** | A5 |
| **Tipo** | Humano |
| **Autenticación** | Ninguna (endpoint público) / acceso infra directo |
| **Endpoints** | `GET /api/health` |

**Descripción:** DevOps Engineer del equipo VTT. Gestiona la infraestructura donde corre Memory Service: VM Hetzner, docker-compose, PostgreSQL, Redis, storage filesystem. Monitorea la salud del servicio y responde a alertas.

**Responsabilidades:**
- Monitorear `GET /health` como liveness probe
- Gestionar el storage filesystem (espacio en disco, bind mount)
- Verificar que el cleanup job está ejecutándose cada 5 minutos
- Responder a alertas de disco, conexiones PostgreSQL o Redis

## §6 · Sistema (interno)

| Campo | Valor |
|-------|-------|
| **ID** | A6 |
| **Tipo** | Actor interno (no externo al boundary) |
| **Autenticación** | N/A |
| **Endpoints** | Ninguno (job cron) |

**Descripción:** Representación del propio Memory Service actuando como actor en procesos automáticos. Específicamente, el cleanup job cron que se ejecuta cada 5 minutos para recuperar conversaciones stuck en PENDING/PROCESSING.

**Responsabilidades:**
- Ejecutar cleanup cada 5 minutos (D-MEM-35)
- Identificar conversaciones con updatedAt > 10 minutos en PENDING/PROCESSING
- Reintentar (retryCount < 3) o mover a ERROR (retryCount >= 3)
- Mover a ERROR directamente si el archivo no existe en storage (AMB-04 de MS-018)
