## 1. Relaciones `<<include>>` (sub-flujos obligatorios)

| UC origen | UC incluido | Justificación |
|-----------|------------|---------------|
| UC-001 (Import CLI) | UC-005 (Verificar idempotencia) | Toda importación DEBE verificar idempotencia antes de crear registros. El check `[sourceId, externalSessionId]` es paso obligatorio del flujo principal, no opcional. Ref: RF-002, D-MEM-05 |
| UC-002 (Import Web) | UC-005 (Verificar idempotencia) | Mismo motivo que UC-001 |
| UC-003 (Import SDK) | UC-005 (Verificar idempotencia) | Mismo motivo que UC-001 |
| UC-004 (Import ChatGPT) | UC-005 (Verificar idempotencia) | Mismo motivo que UC-001 |
| UC-006 (Import VTT_CHANNEL) | UC-005 (Verificar idempotencia) | Aplica también a import-review: verifica si el canal ya fue importado antes de crear, o ejecuta import incremental si ya existe. Ref: RF-007, D-MEM-16 |
| UC-007 (Upload manual) | UC-005 (Verificar idempotencia) | Upload manual comparte la lógica de idempotencia: no duplicar conversaciones que ya fueron importadas. Ref: RF-009 |

**Patrón:** UC-005 es un sub-flujo transversal a TODOS los casos de uso de ingesta. Encapsula la lógica de idempotencia (findUnique + manejo de existente) que se reutiliza en 6 UCs.

## 2. Relaciones `<<extend>>` (comportamiento condicional)

| UC base | UC extensión | Condición | Justificación |
|---------|-------------|-----------|---------------|
| UC-003 (Import SDK) | UC-021 (Race condition P2002) | Dos requests concurrentes con mismo externalSessionId pasan el check de idempotencia simultáneamente | Extensión condicional: solo ocurre bajo concurrencia. El UC base (UC-003) tiene su propio flujo completo; UC-021 se activa solo cuando falla el `create` con P2002. Ref: DB-OBS-08 |
| UC-001 (Import CLI) | UC-021 (Race condition P2002) | Misma condición que UC-003 | Aplica a cualquier import JSONL bajo concurrencia |
| UC-008 (Obtener contexto) | UC-009 (Timeout fail-fast) | Las queries de contexto exceden 500ms | Extensión condicional: solo se activa si el timeout se alcanza. El UC base tiene su flujo completo (happy path <500ms); UC-009 describe el comportamiento de fallo. Ref: D-MEM-07, D-MEM-37 |

**Patrón extend vs include:** La diferencia es que `include` siempre se ejecuta (idempotencia se verifica en CADA import) mientras que `extend` es condicional (P2002 solo ocurre bajo concurrencia, timeout solo ocurre bajo carga).

## 3. Relaciones de Generalización

| UC general | UCs específicos | Justificación |
|------------|----------------|---------------|
| *Import JSONL* (abstracto) | UC-001 (CLI), UC-002 (Web), UC-003 (SDK), UC-004 (ChatGPT) | Los 4 UCs comparten el mismo flujo principal: validar → idempotencia → crear → escribir → clasificar → IMPORTED. Solo difieren en el `sourceCode` y en los metadatos específicos de cada fuente (ej: UC-003 incluye `platformRefs.runtime_run_id`). El UC general no se instancia directamente — se usa uno de los 4 específicos. |

**Nota:** No se crea un UC-000 "Import JSONL genérico" porque la generalización se documenta aquí como relación conceptual. Cada UC específico (001-004) es auto-contenido y puede leerse sin referencia al padre abstracto.

## 4. Dependencias Funcionales (no UML, informativas)

Relaciones entre UCs que no son include/extend/generalización pero sí son dependencias funcionales del sistema:

| UC | Depende de | Tipo | Razón |
|----|-----------|------|-------|
| UC-008 (Obtener contexto) | UC-001..006 (Imports) | Datos | El contexto solo incluye conversaciones IMPORTED. Sin imports previos, GET /context retorna vacío. |
| UC-011 (Leer contenido) | UC-001..006 (Imports) | Datos | El contenido se lee del archivo en /storage/ que fue escrito durante el import. |
| UC-016, UC-017 (Cost reports) | UC-001..006 (Imports) | Datos | Los costos provienen de ConversationUsage que se crea durante el import. |
| UC-018 (Dashboard stats) | UC-001..006 (Imports) | Datos | Las estadísticas se calculan sobre conversaciones existentes. |
| UC-014 (Timeline) | UC-001..006 (Imports) | Datos | El timeline muestra conversaciones del agente que fueron importadas previamente. |
| UC-020 (Cleanup) | UC-001..006 (Imports) | Recovery | El cleanup recupera conversaciones que fallaron durante el import. |
| UC-001..006 (Imports) | UC-022 (Init catálogos) | Prerequisito | Los imports resuelven catálogos desde Maps pre-cargados. Si los Maps no están inicializados, los imports fallan. |

**Implicación para testing:** Los UCs de consulta (008, 011, 012..018) requieren que al menos un import (001..006) se haya ejecutado previamente. El orden de testing debe ser: UC-022 → UC-003 → UC-008 → demás.
