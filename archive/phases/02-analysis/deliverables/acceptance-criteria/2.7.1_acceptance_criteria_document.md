## 1. Propósito

Este documento establece el marco para los criterios de aceptación de Memory Service R1. Los ACs son el contrato formal entre análisis (SA) y testing (QA): definen qué significa "funciona correctamente" para cada User Story del backlog.

---

## 2. Metodología BDD / Gherkin

Los criterios se documentan en formato **Behavior-Driven Development (BDD)** usando notación **Gherkin**:

```gherkin
Feature: [nombre de la feature]

  Scenario: [descripción del escenario]
    Given [precondición — estado del sistema antes]
    And [precondición adicional]
    When [acción del actor]
    Then [resultado esperado]
    And [resultado adicional]
```

Cada User Story tiene al menos 1 escenario (happy path). Las US críticas tienen escenarios adicionales para flujos alternativos, errores y edge cases.

---

## 3. Convenciones

| Convención | Descripción |
|-----------|-------------|
| **AC-US-XXX-N** | ID del criterio: AC de US-XXX, escenario N |
| **Feature** | Agrupa escenarios de la misma US o grupo de US relacionadas |
| **Happy path primero** | El primer escenario de cada feature es siempre el flujo exitoso |
| **Datos concretos** | Los escenarios usan valores concretos, no genéricos (ej: `sourceCode=CLAUDE_SDK`, no `sourceCode=<valor>`) |
| **Ref. a BRs** | Cada escenario que implementa una regla de negocio referencia el BR correspondiente |
