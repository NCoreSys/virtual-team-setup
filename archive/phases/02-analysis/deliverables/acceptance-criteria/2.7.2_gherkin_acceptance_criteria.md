## EP-01 · Import & Storage

### Feature: Import CLAUDE_SDK (US-001)

```gherkin
  Scenario: AC-US-001-1 Import exitoso primera vez
    Given el Runtime tiene una sesión JSONL válida con sourceCode=CLAUDE_SDK
    And externalSessionId="{run_id}:r1:be" no existe en BD
    And la SERVICE_KEY en header Authorization es válida
    When el Runtime envía POST /api/conversations/import
    Then la respuesta es HTTP 201 con { status: "IMPORTED", conversationId: "<uuid>" }
    And la conversación queda en estado IMPORTED en BD
    And el JSONL completo está en /storage/CLAUDE_SDK/{conversationId}.jsonl
    And BD contiene solo contentPreview (primeros 500 chars del primer turno ASSISTANT)
    And ConversationTurn, TurnBlock y ConversationUsage están persistidos
    And Classification y ConversationTopic están asignados por el clasificador
    # Ref: RF-001, BR-003, BR-004, D-MEM-18

  Scenario: AC-US-001-2 Validación Zod falla
    Given el payload no tiene campo sourceCode
    When el Runtime envía POST /api/conversations/import
    Then la respuesta es HTTP 400 con detalle del error Zod
    And no se crea ningún registro en BD
```

### Feature: Import idempotente (US-002)

```gherkin
  Scenario: AC-US-002-1 Reimport retorna ALREADY_INDEXED
    Given existe una conversación con sourceId=CLAUDE_SDK y externalSessionId="run1:r1:be" en estado IMPORTED
    When el Runtime envía POST /import con el mismo sourceCode y externalSessionId
    Then la respuesta es HTTP 200 con { status: "ALREADY_INDEXED", conversationId: "<uuid-existente>" }
    And no se crea registro duplicado en BD
    And no se sobreescribe el archivo en /storage/
    # Ref: RF-002, BR-001, D-MEM-05
```

### Feature: Race condition P2002 (US-003)

```gherkin
  Scenario: AC-US-003-1 Dos imports simultáneos del mismo externalSessionId
    Given dos requests concurrentes llegan con el mismo externalSessionId
    And ambas pasan la verificación de idempotencia (findUnique) simultáneamente
    When la primera ejecuta create → éxito
    And la segunda ejecuta create → falla con Prisma P2002
    Then la segunda captura P2002, consulta el registro existente via findUnique
    And la segunda retorna HTTP 200 con { status: "ALREADY_INDEXED" }
    And solo existe un registro en BD y un archivo en /storage/
    # Ref: RF-002, BR-001, DB-OBS-08, D-MEM-42
```

### Feature: Import VTT_CHANNEL (US-004)

```gherkin
  Scenario: AC-US-004-1 Import de canal AGENT_REVIEW
    Given Hook Manager tiene un canal VTT_CHANNEL con conversationType=AGENT_REVIEW
    And el payload incluye 3 participants y 5 messages
    And la SERVICE_KEY es válida
    When Hook Manager envía POST /api/conversations/import-review
    Then la respuesta es HTTP 201 con { status: "IMPORTED" }
    And la conversación tiene primaryAgentId=null
    And 3 registros en ConversationParticipant
    And 5 registros en AgentMessage
    And archivo en /storage/_reviews/{conversationId}.jsonl
    # Ref: RF-006, BR-004, BR-021, D-MEM-15, D-MEM-29

  Scenario: AC-US-004-2 SERVICE_KEY inválida
    Given la SERVICE_KEY en el header es incorrecta
    When Hook Manager envía POST /api/conversations/import-review
    Then la respuesta es HTTP 401
    And no se crea ningún registro
    # Ref: BR-018
```

### Feature: Import incremental VTT_CHANNEL (US-005)

```gherkin
  Scenario: AC-US-005-1 Append mensajes nuevos a canal existente
    Given existe un canal VTT_CHANNEL con 5 mensajes y 2 participantes
    When Hook Manager envía import-review con 3 mensajes nuevos
    Then la respuesta contiene { status: "UPDATED", newMessages: 3 }
    And el total de mensajes en BD es 8
    And los 5 mensajes originales no se modificaron

  Scenario: AC-US-005-2 Upsert de participante nuevo
    Given existe un canal con 2 participantes (agente-A, agente-B)
    When Hook Manager envía import incremental con participante agente-C
    Then agente-C se agrega como ConversationParticipant
    And agente-A y agente-B no se modifican
    And el total de participantes es 3
    # Ref: RF-007, BR-008, D-MEM-16
```

### Feature: Upload manual (US-006)

```gherkin
  Scenario: AC-US-006-1 Upload de export claude.ai
    Given un usuario tiene un archivo JSON de exportación de claude.ai
    When envía POST /api/conversations/upload con el archivo
    Then el sistema detecta formato CLAUDE_WEB
    And la conversación se importa con el flujo estándar
    And retorna { status: "IMPORTED" }

  Scenario: AC-US-006-2 Upload duplicado
    Given el archivo ya fue subido previamente
    When envía POST /upload con el mismo archivo
    Then retorna { status: "ALREADY_INDEXED" }
    # Ref: RF-009, BR-001
```

### Feature: Import CLAUDE_CLI (US-007)

```gherkin
  Scenario: AC-US-007-1 Import sesión CLI
    Given una sesión JSONL con sourceCode=CLAUDE_CLI
    When el Runtime envía POST /api/conversations/import
    Then la conversación se importa correctamente con sourceCode=CLAUDE_CLI
    And el archivo queda en /storage/CLAUDE_CLI/{conversationId}.jsonl
```

### Feature: Import CLAUDE_WEB (US-008)

```gherkin
  Scenario: AC-US-008-1 Import exportación web
    Given una sesión con sourceCode=CLAUDE_WEB
    When se importa via POST /import
    Then ConversationUsage.totalCostUsd es null (adapter sin costo)
    # Ref: BR-014
```

### Feature: Import ChatGPT (US-009)

```gherkin
  Scenario: AC-US-009-1 Import exportación ChatGPT
    Given una sesión con sourceCode=CHATGPT
    When se importa via POST /import
    Then ConversationUsage.totalCostUsd es null
    And la conversación aparece en cost-reports con $0.00
    # Ref: BR-014
```

### Feature: Cleanup reintenta imports fallidos (US-010)

```gherkin
  Scenario: AC-US-010-1 Error en import — estado permanece PROCESSING
    Given el Runtime importa una sesión CLAUDE_SDK válida
    And ocurre un error interno durante parseo de turns
    When el catch captura el error
    Then la conversación permanece en estado PROCESSING
    And NO cambia a ERROR directamente
    And el cleanup cron será responsable de reintentar
    # Ref: BR-002, AMB-07
```

## EP-02 · Memory Context

### Feature: Contexto runtime <500ms (US-011)

```gherkin
  Scenario: AC-US-011-1 Contexto respondido dentro del SLA
    Given el Prompt Builder tiene SERVICE_KEY válida
    And hay conversaciones IMPORTED para projectId=X y agentId=Y
    When PB envía GET /api/context?projectId=X&agentId=Y
    And el sistema responde en menos de 500ms
    Then la respuesta es HTTP 200 con JSON de 6 secciones
    And contiene: recentSessions, recentDecisions, frequentFiles, recentReviews, costSummary, projectContext
    And solo incluye conversaciones con status IMPORTED
    # Ref: RF-011, BR-011, BR-012

  Scenario: AC-US-011-2 Performance bajo carga
    Given 10 virtual users concurrentes (k6)
    When cada uno llama GET /context durante 60 segundos
    Then el p95 de latencia es menor a 500ms
    # Ref: RNF-001
```

### Feature: Timeout fail-fast 504 (US-012)

```gherkin
  Scenario: AC-US-012-1 Timeout retorna 504
    Given PB tiene SERVICE_KEY válida
    When PB envía GET /api/context
    And las queries internas superan 500ms
    Then la respuesta es HTTP 504 con { error: "MEM-ERR-504" }
    And NO hay respuesta parcial
    And NO hay retry automático del servidor
    And NO hay modo degradado ni cache fallback
    # Ref: RF-012, BR-011, D-MEM-07, D-MEM-37
```

### Feature: Filtro por proyecto (US-013)

```gherkin
  Scenario: AC-US-013-1 projectId obligatorio
    Given PB envía GET /api/context sin projectId
    Then la respuesta es HTTP 400
    # Ref: BR-013, D-MEM-39

  Scenario: AC-US-013-2 Filtrado por proyecto
    Given hay conversaciones de projectId=A y projectId=B
    When PB envía GET /context?projectId=A
    Then solo retorna conversaciones de proyecto A
```

### Feature: Filtro por tarea (US-014)

```gherkin
  Scenario: AC-US-014-1 taskId filtra adicionalmente
    Given hay conversaciones de taskId=T1 y taskId=T2 en proyecto A
    When PB envía GET /context?projectId=A&taskId=T1
    Then solo retorna conversaciones de tarea T1
```

### Feature: Contexto síncrono (US-015)

```gherkin
  Scenario: AC-US-015-1 Respuesta siempre síncrona
    When PB envía GET /api/context
    Then recibe la respuesta completa en una sola llamada HTTP
    And NO requiere polling ni websocket
    # Ref: D-MEM-37
```

## EP-03 · Timeline & History

### Feature: Timeline de agente (US-016)

```gherkin
  Scenario: AC-US-016-1 Timeline incluye TASK_EXECUTION
    Given agente-BE tiene 3 conversaciones TASK_EXECUTION
    When TL consulta GET /api/agents/agente-BE/timeline
    Then retorna las 3 conversaciones ordenadas por startedAt DESC

  Scenario: AC-US-016-2 Timeline incluye participaciones multi-agente
    Given agente-BE participó en 2 AGENT_REVIEW como ConversationParticipant
    When TL consulta GET /api/agents/agente-BE/timeline
    Then las 2 reviews aparecen en el timeline junto con las TASK_EXECUTION
    # Ref: RF-021
```

### Feature: Contenido completo (US-017)

```gherkin
  Scenario: AC-US-017-1 Contenido leído desde /storage/
    Given conversación con id=X en estado IMPORTED
    And archivo existe en /storage/CLAUDE_SDK/X.jsonl
    When TL consulta GET /api/conversations/X/content
    Then el sistema lee y parsea el JSONL desde /storage/
    And retorna turns y blocks completos
    And la BD NO fue consultada para el contenido (solo para metadata)
    And la BD solo tiene contentPreview de 500 chars
    # Ref: RF-016, BR-003, D-MEM-43, AMB-AC-02

  Scenario: AC-US-017-2 Archivo no existe en storage
    Given conversación con id=X existe en BD
    And el archivo NO existe en /storage/
    When TL consulta GET /api/conversations/X/content
    Then retorna HTTP 404 con { error: "MEM-ERR-404" }
    And NO intenta leer contenido desde BD
```

### Feature: Listar conversaciones (US-018)

```gherkin
  Scenario: AC-US-018-1 Listado con filtros
    Given hay 20 conversaciones de proyecto-A y 10 de proyecto-B
    When TL consulta GET /api/conversations?projectId=A&limit=10
    Then retorna 10 conversaciones de proyecto-A
    And cada una incluye metadata (turnCount, tokens, topics, contentPreview)
    And NO incluye contenido completo
    # Ref: RF-017, RF-018

  Scenario: AC-US-018-2 Default solo IMPORTED
    Given hay conversaciones en IMPORTED y ERROR
    When TL consulta GET /api/conversations sin filtro de status
    Then retorna solo las IMPORTED
    # Ref: RF-020
```

### Feature: Timeline multi-agente (US-019)

```gherkin
  Scenario: AC-US-019-1 Reviews en timeline via ConversationParticipant
    Given agente-SA es participante en 1 AGENT_REVIEW y 1 AGENT_CLARIFICATION
    When consulta GET /api/agents/agente-SA/timeline
    Then ambas aparecen con su conversationType correspondiente
```

### Feature: ContentPreview en lista (US-020)

```gherkin
  Scenario: AC-US-020-1 Preview visible en listado
    Given conversación con contentPreview="Los primeros 500 caracteres..."
    When aparece en GET /api/conversations
    Then el campo contentPreview muestra los primeros 500 chars
```

### Feature: Contenido desde /storage/ transparente (US-021)

```gherkin
  Scenario: AC-US-021-1 UI no sabe de dónde viene el contenido
    Given el usuario hace click en una conversación en la UI
    When la UI llama GET /api/conversations/:id/content
    Then recibe el JSONL completo con todos los turns
    And la UI renderiza el contenido sin saber que vino de /storage/
    # Ref: D-MEM-43, BR-003
```

## EP-04 · Cost Tracking

### Feature: Costo por proyecto (US-022)

```gherkin
  Scenario: AC-US-022-1 Totales de costo por proyecto
    Given proyecto-A tiene 5 conversaciones IMPORTED con costos
    When PM consulta GET /api/projects/A/cost-report
    Then totalConversations=5
    And totalCostUsd es la suma de ConversationUsage.totalCostUsd
    And incluye breakdown por agente y por modelo
    # Ref: RF-024, BR-016

  Scenario: AC-US-022-2 Filtro por rango de fechas
    Given conversaciones de enero y febrero
    When PM consulta cost-report con from=2026-02-01&to=2026-02-28
    Then solo incluye conversaciones de febrero (por startedAt)
```

### Feature: Desglose por agente (US-023)

```gherkin
  Scenario: AC-US-023-1 Breakdown por agente
    Given proyecto-A tiene conversaciones de agente-BE (3) y agente-FE (2)
    When PM consulta cost-report de proyecto-A
    Then breakdownByAgent muestra agente-BE con count=3 y agente-FE con count=2
```

### Feature: Costo de agente (US-024)

```gherkin
  Scenario: AC-US-024-1 Cost report por agente
    Given agente-BE tiene conversaciones en proyecto-A y proyecto-B
    When PM consulta GET /api/agents/agente-BE/cost-report
    Then incluye breakdownByProject con ambos proyectos
    And incluye breakdownByModel
    # Ref: RF-025
```

### Feature: Cálculo USD real (US-025)

```gherkin
  Scenario: AC-US-025-1 Costo importado del SDK
    Given conversación CLAUDE_SDK con totalCostUsd=1.42 del ResultMessage
    When se consulta en cost-report
    Then totalCostUsd=1.42 (importado, no recalculado)
    # Ref: BR-014
```

### Feature: Solo SDK con costo real (US-026)

```gherkin
  Scenario: AC-US-026-1 Adapters sin costo muestran $0.00
    Given conversación CLAUDE_WEB con totalCostUsd=null
    When aparece en cost-report
    Then muestra $0.00 para esa conversación
    And la conversación NO se excluye del reporte
    And el conteo totalConversations la incluye
    # Ref: BR-014
```

### Feature: Trend semanal (US-027)

```gherkin
  Scenario: AC-US-027-1 groupBy=byWeek funciona
    Given conversaciones distribuidas en 3 semanas
    When PM consulta cost-report con groupBy=byWeek
    Then retorna trend con 3 entries en formato YYYY-W##
    And la implementación usa prisma.$queryRaw con DATE_TRUNC('week',...)
    And NO usa Prisma groupBy nativo
    # Ref: BR-017, TL-08, AMB-AC-03
```

## EP-05 · Operations & Dashboard

### Feature: Dashboard stats (US-028)

```gherkin
  Scenario: AC-US-028-1 Stats globales
    Given hay 50 conversaciones IMPORTED, 2 ERROR, 1 PROCESSING
    When PM consulta GET /api/dashboard/stats
    Then totalConversations muestra breakdown: IMPORTED=50, ERROR=2, PROCESSING=1
    And bySource muestra conteo por sourceCode
    And byType muestra conteo por conversationType
    And activeAgents cuenta agentes distintos con IMPORTED en últimos 30 días
    # Ref: RF-028, BR-015
```

### Feature: Health check (US-029)

```gherkin
  Scenario: AC-US-029-1 Servicio operativo
    Given BD, storage y Redis están disponibles
    When Admin consulta GET /api/health
    Then retorna HTTP 200 con { status: "ok" }
    # Ref: RF-030
```

### Feature: Health detalle (US-030)

```gherkin
  Scenario: AC-US-030-1 Health con componentes
    Given BD está UP, storage accesible, Redis UP
    When Admin consulta GET /api/health
    Then retorna { status: "ok", db: "up", storage: "up", redis: "up" }

  Scenario: AC-US-030-2 BD caída
    Given BD no responde
    When Admin consulta GET /api/health
    Then retorna HTTP 503 con { status: "down", db: "down" }
    # Ref: RF-030
```

### Feature: Cleanup automático (US-031)

```gherkin
  Scenario: AC-US-031-1 Cleanup ejecuta cada 5 minutos
    Given el servicio está corriendo
    When pasan 5 minutos
    Then el cleanup cron se ejecuta automáticamente
    And busca conversaciones STALE (PROCESSING + updatedAt >10min)
    # Ref: RF-031, BR-006, D-MEM-35
```

### Feature: Cleanup MAX_RETRIES (US-032)

```gherkin
  Scenario: AC-US-032-1 Reintento dentro del límite
    Given conversación en PROCESSING con updatedAt >10min y retryCount=1
    When cleanup cron ejecuta
    Then la conversación vuelve a PENDING
    And retryCount se incrementa a 2

  Scenario: AC-US-032-2 ERROR tras agotar reintentos
    Given conversación en PROCESSING con updatedAt >10min y retryCount=4 (>MAX_RETRIES)
    When cleanup cron ejecuta
    Then la conversación cambia a ERROR
    And errorMessage="Max retries exceeded after 3 attempts"
    # Ref: RF-031, BR-006, D-MEM-40

  Scenario: AC-US-032-3 Archivo faltante en storage
    Given conversación en PROCESSING y archivo NO existe en /storage/
    When cleanup cron ejecuta
    Then la conversación cambia a ERROR directamente
    And errorMessage="Storage file missing after processing"
    And retryCount NO se incrementa
    # Ref: BR-006, AMB-04
```

### Feature: Inicialización catálogos (US-033)

```gherkin
  Scenario: AC-US-033-1 Catálogos cargados al startup
    Given el servicio se inicia
    When el proceso de bootstrap se ejecuta
    Then los 10 catálogos están cargados en memoria como Maps (code→id)
    And los imports pueden resolver IDs sin queries individuales
    # Ref: RF-032, BR-009

  Scenario: AC-US-033-2 Fallo de carga aborta startup
    Given la BD no está accesible al iniciar
    When el proceso intenta cargar catálogos
    Then el servicio aborta con error descriptivo
    And NO se levanta el servidor HTTP
    # Ref: BR-009, DEC-01
```
