## Mecanismo de Autenticación R1

Memory Service R1 usa exclusivamente **SERVICE_KEY** como mecanismo de autenticación (D-MEM-26, KICKOFF §5.4). La SERVICE_KEY se envía como `Authorization: Bearer {SERVICE_KEY}` en el header HTTP. No hay JWT, no hay OAuth, no hay sesiones de usuario en R1.

La SERVICE_KEY es compartida entre los sistemas consumidores autorizados: Runtime v1.1, Prompt Builder v1.3 y Hook Manager VTT. Es una clave estática configurada como variable de entorno.

## Reglas

### AR-01 · Middleware SERVICE_KEY

El middleware de autenticación valida el header `Authorization`. Si el valor no coincide con la SERVICE_KEY configurada, retorna HTTP 401 antes de ejecutar la lógica del endpoint. El middleware se aplica a los endpoints 1, 2 y 3 de la tabla. Los endpoints 4-11 NO pasan por el middleware de auth.

### AR-02 · Público no significa anónimo (R2)

En R1, "público" significa sin autenticación. En R2, cuando se implemente JWT para usuarios finales, los endpoints hoy públicos requerirán token JWT con roles diferenciados (PM, TL, SA, etc.). La arquitectura del middleware debe permitir esta extensión sin refactor mayor.

**Ref:** KICKOFF §5.4, ADR-SA-006

### AR-03 · SERVICE_KEY no expira en R1

La SERVICE_KEY es estática y no expira durante R1. No hay rotación automática. Si se compromete, se cambia manualmente en las variables de entorno de todos los consumidores y se reinicia el servicio.

### AR-04 · Rate limiting por SERVICE_KEY

Los endpoints protegidos por SERVICE_KEY tienen rate limiting via Redis (prefix `mem`). En desarrollo, el rate limiting usa memoria. Si Redis no está disponible en producción, el rate limiting no funciona (AS-003).

**Ref:** D-MEM-22, D-MEM-23
