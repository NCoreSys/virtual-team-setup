## Términos de Dominio (20)

### ALREADY_INDEXED
Respuesta del sistema cuando se intenta importar una conversación que ya existe con status IMPORTED. No es un error — es comportamiento esperado de la idempotencia. HTTP 200 con `{ status: "ALREADY_INDEXED", conversationId }`. Ref: D-MEM-05, BR-001.

### Catalog Cache
Cache en memoria (Maps `code → id`) de los 10 catálogos del sistema, pre-cargado al inicio del servicio. Los flujos de import resuelven IDs desde estos Maps para evitar N+1 queries en loops. Si el cache no puede cargarse, el servicio aborta startup. Ref: DEC-01, DB-OBS-03, BR-009.

### Cleanup Cron
Job automático que ejecuta cada 5 minutos para recuperar conversaciones stuck en PENDING o PROCESSING con más de 10 minutos sin actualización. Reintenta hasta MAX_RETRIES (3) antes de mover a ERROR. Es el ÚNICO mecanismo que produce el estado ERROR. Ref: D-MEM-35, BR-006.

### contentPreview
Primeros 500 caracteres del contenido del primer turno de rol ASSISTANT en una conversación. Es el único fragmento de contenido almacenado en BD. Se calcula durante el import y se usa en listados para que el usuario decida si abrir el detalle. El contenido completo está en `/storage/`. Ref: D-MEM-18, D-MEM-43, BR-020.

### Conversation
Entidad central del sistema. Representa una sesión de agente (TASK_EXECUTION) o un canal de comunicación inter-agente (AGENT_REVIEW, AGENT_CLARIFICATION). Cada conversación tiene un ciclo de vida: PENDING → PROCESSING → IMPORTED/ERROR. Ref: SPEC §4.1.

### ConversationParticipant
Registro de un agente que participa en una conversación multi-agente (AGENT_REVIEW o AGENT_CLARIFICATION). Solo aplica cuando `primaryAgentId` es null. Incluye: agentId, role, platformId. Ref: D-MEM-29.

### ConversationUsage
Registro de consumo de tokens y costo USD asociado a una conversación. Campos: inputTokens, outputTokens, cacheCreationInputTokens, cacheReadInputTokens, totalCostUsd, modelId. Los datos provienen del `ResultMessage` del SDK. Ref: SPEC §4.1, BR-014.

### externalSessionId
Identificador único externo de una conversación, proporcionado por el sistema que la origina. Para CLAUDE_SDK: formato `{run_id}:r{N}:{agentRole}` donde `run_id` es el UUID de la corrida, `N` es el número de ronda (1-based) y `agentRole` es el rol del agente. Junto con `sourceId`, forma el constraint de idempotencia. Ref: D-INT-01, D-INT-02.

### Fail-fast
Patrón de diseño de `GET /context`: si la respuesta no está lista en 500ms, el sistema retorna error 504 inmediatamente sin intentar retry, modo degradado ni cache fallback. El caller (Prompt Builder) decide cómo proceder. Ref: D-MEM-07, D-MEM-37, BR-011.

### MAX_RETRIES
Límite de reintentos del cleanup cron antes de mover una conversación a ERROR. Valor: 3 en R1. Después de 3 intentos fallidos de procesamiento, el cleanup asume que el error no es transitorio y marca la conversación como ERROR definitivo. Ref: D-MEM-40, LIM-10, BR-006.

### MEM-ERR-404
Código de error retornado por `GET /content` cuando el archivo JSONL no existe en `/storage/`. Indica que la metadata existe en BD pero el archivo se perdió del filesystem. Ref: D-MEM-43, RF-016.

### MEM-ERR-504
Código de error retornado por `GET /context` cuando el timeout de 500ms se alcanza. Señal explícita al Prompt Builder de que el contexto no está disponible. NO es un error de infraestructura — es comportamiento diseñado (fail-fast). Ref: D-MEM-07, BR-011.

### platformRefs
Campo JSON nullable en Conversation que almacena referencias cruzadas con la plataforma de origen. Para CLAUDE_SDK incluye: `runtime_run_id` (UUID de la corrida del Runtime), `round` (número de ronda), `orchestrated: true`. Permite buscar conversaciones por run del Runtime. Ref: D-MEM-17, D-INT-03, D-INT-04.

### SERVICE_KEY
Token estático de autenticación para sistemas internos del ecosistema VTT. Compartida entre Runtime v1.1, Prompt Builder v1.3 y Hook Manager VTT. Se envía como `Authorization: Bearer {SERVICE_KEY}`. No expira en R1. Protege los endpoints de import y contexto. Ref: D-MEM-26, BR-018.

### sourceCode
Código que identifica el tipo de fuente de una conversación. Valores R1: `CLAUDE_SDK` (Anthropic SDK via Runtime), `CLAUDE_CLI` (Claude Code CLI), `CLAUDE_WEB` (exportación claude.ai), `CHATGPT` (exportación ChatGPT), `VTT_CHANNEL` (canal inter-agente VTT). Cada sourceCode tiene su adapter de procesamiento. Ref: D-MEM-09, SPEC §5.

### STALE
Calificación que el cleanup cron asigna a una conversación que lleva más de 10 minutos en PENDING o PROCESSING sin actualización de `updatedAt`. Las conversaciones STALE son candidatas a reintento (si retryCount <= MAX_RETRIES) o a ERROR (si se excede). Ref: D-MEM-35, BR-006.

### Storage
Sistema de archivos (filesystem bind mount en R1) donde se almacena el contenido JSONL completo de cada conversación. Path: `/storage/{sourceCode}/{conversationId}.jsonl` para TASK_EXECUTION, `/storage/_reviews/{conversationId}.jsonl` para AGENT_REVIEW/CLARIFICATION. Es la fuente de verdad para el contenido — la BD solo tiene metadata y contentPreview. Ref: D-MEM-25, BR-003, BR-004.

### Turn
Unidad de intercambio dentro de una Conversation. Cada turn tiene un rol (USER, ASSISTANT, SYSTEM), un índice ordinal (`turnIndex`) y un `contentPreview`. Los turns contienen Blocks. En BD se almacena metadata y preview; el contenido completo está en `/storage/`. Ref: SPEC §4.1.

### Block (TurnBlock)
Contenido atómico dentro de un Turn. Tipos: TEXT (texto generado), TOOL_USE (invocación de herramienta), TOOL_RESULT (resultado de herramienta), THINKING (razonamiento interno). Cada block tiene un `blockIndex` ordinal y un `blockType`. Ref: SPEC §4.1.

### workType
Clasificación del tipo de trabajo realizado en una conversación. Valores R1 (catálogo extensible): implementation, bug-fix, review, migration, deploy, exploration. Asignado por el clasificador determinístico durante el import. Almacenado en la tabla Classification. Ref: D-MEM-20, BR-007.
