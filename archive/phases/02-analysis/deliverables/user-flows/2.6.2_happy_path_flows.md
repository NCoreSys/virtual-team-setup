## 1. Propósito

```

  │
  ├─ 10:15  ● AGENT_REVIEW · VTT_CHANNEL
```

### FP-02 · Importar Conversación Manual

**Actor:** SA (cualquier rol del equipo)  
**Objetivo:** Subir un export de claude.ai que no pasó por el flujo automático del Runtime.  
**UC:** UC-007 | **US:** US-006

```
PASO 1 — Navegar a Upload
──────────────────────────
El SA abre Memory Service UI → navegación → "Upload".
La pantalla Manual Upload (UF-07) muestra el formulario.

PASO 2 — Seleccionar archivo
─────────────────────────────
El SA arrastra un archivo JSON exportado de claude.ai al dropzone.
La UI muestra:
  Archivo: "conversation_2026-05-06.json" (245 KB)
  Fuente detectada: CLAUDE_WEB (auto-detect)

PASO 3 — Confirmar y subir
───────────────────────────
El SA click "Subir conversación".

  Sistema ejecuta:
  POST /upload
  Body: { file: [archivo], sourceCode: "CLAUDE_WEB" }

La UI muestra spinner: "Procesando..."

PASO 4 — Resultado exitoso
───────────────────────────
El sistema procesa el archivo:
  validar → PENDING → PROCESSING → escribir /storage/CLAUDE_WEB/{id}.jsonl
  → parsear turns → clasificar → IMPORTED

La API responde: { status: "IMPORTED", conversationId: "abc-123" }

La UI muestra banner verde:
  ✓ Conversación importada exitosamente
  [Ver conversación]  [Subir otra]

PASO 5 — Ver la conversación importada
───────────────────────────────────────
El SA click "Ver conversación".

  → Navega a Conversation Viewer (UF-03)
  
  GET /conversations/abc-123/content
  
  La UI muestra el contenido completo de la conversación recién importada.
```

**Flujo de estados:** PENDING → PROCESSING → IMPORTED (ST-01, ST-02, ST-03)  
**Idempotencia:** Si el SA sube el mismo archivo dos veces → ALREADY_INDEXED (ver error flows 2.6.3)

### FP-04 · Verificar Health

**Actor:** Admin / DO  
**Objetivo:** Confirmar que Memory Service está operativo antes de un deploy del Runtime.  
**UC:** UC-019 | **US:** US-029, US-030

```
PASO 1 — Acceder a Health
──────────────────────────
El Admin abre Memory Service UI → navegación → "Health".

  Sistema ejecuta:
  GET /health

PASO 2 — Resultado: todo OK
────────────────────────────
La UI muestra la pantalla Health Status (UF-08):

  ┌─────────────────────────────────────────────┐
  │  Estado: ● HEALTHY                           │
  │                                              │
  │  BD (PostgreSQL):     ● OK                   │
  │  Storage (/storage/): ● OK                   │
  │  Redis:               ● OK                   │
  │                                              │
  │  Última verificación: 2026-05-06 14:32:15    │
  │  [Refresh]                                   │
  └─────────────────────────────────────────────┘

Los tres semáforos están en verde. El Admin confirma que
el servicio está listo y procede con el deploy del Runtime.

PASO 3 — Auto-refresh
──────────────────────
La pantalla se actualiza automáticamente cada 60 segundos.
Si el Admin quiere verificar inmediatamente, click "Refresh".

  GET /health (nueva request)
  
  Se actualizan los semáforos con datos frescos.
```

**Semáforos (AMB-UX-03):**
- ● Verde = OK — componente responde normalmente
- ⚠ Amarillo = SLOW/DEGRADED — componente responde pero con latencia
- ✕ Rojo = DOWN/UNREACHABLE — componente no responde
