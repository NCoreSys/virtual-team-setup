# 2.6.3 — Error Flows

**Proyecto:** Memory Service  
**Fase:** Analysis (Phase 4)  
**Tarea VTT:** MS-023  
**Autor:** UX Designer — `a75a1dae-754a-4b6f-a3ff-db8d51f6a91b`  
**Fecha:** 2026-05-06

---

## 1. Propósito

Documentar qué ve el usuario cuando algo falla. Cada error flow define el trigger, el estado del sistema, el mensaje visible en la UI y las acciones disponibles para el usuario.

Los flujos EF-01 a EF-04 son obligatorios per ASSIGNMENT MS-023. Los flujos EF-05 a EF-09 completan la cobertura de errores relevantes en las 8 pantallas.

---

## 2. Principios de Diseño de Errores

| Principio | Aplicación |
|-----------|------------|
| **Sin jerga técnica** | El usuario ve "No se pudo cargar el contenido", no "MEM-ERR-404" ni "ENOENT" |
| **Acción clara** | Todo mensaje de error incluye al menos un botón de acción (Reintentar, Volver, Intentar con otro archivo) |
| **Color semántico** | Rojo = error bloqueante, Amarillo = warning/degradado, Azul = informativo (ALREADY_INDEXED) |
| **Sin stack traces** | Nunca exponer detalles técnicos internos en la UI (requisito de seguridad SPEC §15.4) |
| **Errores recuperables vs terminales** | Recuperable: Reintentar. Terminal: Volver + explicación |

## 3. Flujos de Error Obligatorios

### EF-01 · Error en Import desde Upload

**Trigger:** POST /upload → el procesamiento interno falla (escritura en storage, parseo o clasificación).  
**Estado del sistema:** Conversación queda en PROCESSING (BR-002 — catch NO mueve a ERROR).  
**Pantalla:** Manual Upload (UF-07)  
**UC:** UC-007 | **US:** US-006 | **BR:** BR-002, BR-006

```
FLUJO:
───────
1. Usuario sube archivo via POST /upload
2. API responde 201 con { conversationId, status: "PROCESSING" }
   (el procesamiento interno falló pero el catch delegó al cleanup — BR-002)

3. UI muestra:
   ┌─────────────────────────────────────────────────────┐
   │  ⏳ Procesando...                                    │
   │                                                      │
   │  La conversación se está importando.                 │
   │  Este proceso puede tomar unos segundos.             │
   │                                                      │
   │  [spinner girando]                                   │
   └─────────────────────────────────────────────────────┘

4. UI ejecuta auto-poll cada 30s (AMB-UX-01):
   GET /conversations/{id} → verifica statusCode

5a. Si tras 6 polls (3 min) sigue en PROCESSING:
   ┌─────────────────────────────────────────────────────┐
   │  ℹ️  El procesamiento está tardando más de lo usual  │
   │                                                      │
   │  El sistema reintentará automáticamente.             │
   │  Puedes cerrar esta ventana sin perder la            │
   │  conversación.                                       │
   │                                                      │
   │  [Ver en lista de conversaciones]                    │
   └─────────────────────────────────────────────────────┘

5b. Si el poll detecta status ERROR (cleanup agotó MAX_RETRIES):
   ┌─────────────────────────────────────────────────────┐
   │  ✕  El procesamiento falló                          │
   │                                                      │
   │  La conversación no pudo importarse después de       │
   │  múltiples intentos.                                 │
   │                                                      │
   │  [Reintentar upload]  [Volver al inicio]             │
   └─────────────────────────────────────────────────────┘
```

**Lo que pasa en background:**
- La conversación queda en PROCESSING en BD (BR-002)
- Cleanup cron la detecta en <10 min (BR-006)
- Si retryCount < 3 → resetea a PENDING → se reprocesa
- Si retryCount >= 3 → mueve a ERROR
- Si archivo no existe en /storage/ → ERROR directo (AMB-04)

### EF-02 · Storage No Disponible (Contenido no encontrado)

**Trigger:** GET /conversations/:id/content → archivo JSONL no existe en /storage/.  
**Código HTTP:** 404 con `{ error: "MEM-ERR-404", message: "Content file not found" }`  
**Pantalla:** Conversation Viewer (UF-03)  
**UC:** UC-011 flujo alternativo A | **US:** US-017, US-021

```
FLUJO:
───────
1. Usuario click en una conversación desde la lista o timeline
2. UI envía GET /conversations/{id}/content
3. API responde 404 MEM-ERR-404 (archivo no existe en /storage/)

4. UI muestra:
   ┌─────────────────────────────────────────────────────┐
   │  ✕  No se pudo cargar el contenido                  │
   │                                                      │
   │  El archivo de esta conversación no está             │
   │  disponible en el almacenamiento.                    │
   │  Esto puede ocurrir si el archivo fue eliminado      │
   │  o si hubo un problema durante la importación.       │
   │                                                      │
   │  [Volver a la lista]                                 │
   └─────────────────────────────────────────────────────┘

   Nota: el header con metadata SÍ se muestra (viene de BD).
   Solo el body con contenido de turns muestra el error.
```

**Contexto técnico:**
- La BD tiene metadata (title, turns count, tokens) pero NO el contenido completo (D-MEM-43)
- El contenido completo vive en /storage/{sourceCode}/{id}.jsonl
- Si el archivo se borró manualmente o si el import falló en la escritura → 404
- El header de metadata se renderiza normalmente porque viene de la BD

### EF-03 · Health Degradado / Down

**Trigger:** GET /health → uno o más componentes reportan fallo.  
**Código HTTP:** 200 (degraded) o 503 (down)  
**Pantalla:** Health Status (UF-08)  
**UC:** UC-019 | **US:** US-029, US-030 | **AMB-UX-03**

```
FLUJO — Degradado (un componente con problemas):
─────────────────────────────────────────────────
1. Admin accede a Health Status
2. GET /health → 200 { status: "degraded", db: "ok", storage: "ok", redis: "down" }

3. UI muestra:
   ┌─────────────────────────────────────────────────────┐
   │  ⚠  Estado: DEGRADED                                │
   │                                                      │
   │  BD (PostgreSQL):     ● OK                           │
   │  Storage (/storage/): ● OK                           │
   │  Redis:               ✕ DOWN                         │
   │                                                      │
   │  El servicio está operativo con funcionalidad         │
   │  reducida. Algunos features pueden estar lentos.      │
   │                                                      │
   │  Última verificación: 2026-05-06 14:32:15            │
   │  [Refresh]                                           │
   └─────────────────────────────────────────────────────┘


FLUJO — Down (componente crítico caído):
────────────────────────────────────────
1. Admin accede a Health Status
2. GET /health → 503 { status: "down", db: "down", storage: "ok", redis: "ok" }

3. UI muestra:
   ┌─────────────────────────────────────────────────────┐
   │  ✕  Estado: DOWN                                    │
   │                                                      │
   │  BD (PostgreSQL):     ✕ UNREACHABLE                  │
   │  Storage (/storage/): ● OK                           │
   │  Redis:               ● OK                           │
   │                                                      │
   │  El servicio no está disponible.                     │
   │  La base de datos no responde.                       │
   │                                                      │
   │  [Refresh]                                           │
   └─────────────────────────────────────────────────────┘


FLUJO — Error de conexión (servicio completamente caído):
─────────────────────────────────────────────────────────
1. Admin accede a Health Status
2. GET /health → Network error / ECONNREFUSED

3. UI muestra:
   ┌─────────────────────────────────────────────────────┐
   │  ✕  No se puede conectar con el servicio            │
   │                                                      │
   │  Memory Service no responde en el puerto 3002.       │
   │  Verifica que el contenedor esté ejecutándose.       │
   │                                                      │
   │  [Reintentar]                                        │
   └─────────────────────────────────────────────────────┘
```

**Reglas de semáforos (AMB-UX-03):**
- Estado global = el peor estado individual de los 3 componentes
- Si cualquiera es DOWN → estado global es DOWN (HTTP 503)
- Si cualquiera es SLOW pero ninguno DOWN → DEGRADED (HTTP 200)
- Si todos OK → HEALTHY (HTTP 200)

### EF-04 · Lista Vacía (Sin Resultados)

**Trigger:** GET /conversations → 0 resultados con los filtros aplicados.  
**Código HTTP:** 200 con array vacío.  
**Pantalla:** Conversations List (UF-02), Agent Timeline (UF-04)  
**UC:** UC-012, UC-014 | **US:** US-018, US-016

```
FLUJO — Conversations List sin resultados:
──────────────────────────────────────────
1. Usuario aplica filtros restrictivos (agente + proyecto + fuente + rango de fechas)
2. GET /conversations?primaryAgentId=X&projectId=Y&sourceCode=CHATGPT
3. API responde 200 con { data: [], pagination: { total: 0 } }

4. UI muestra:
   ┌─────────────────────────────────────────────────────┐
   │  [Panel de filtros activos mostrados arriba]         │
   │                                                      │
   │           📭                                         │
   │  No hay conversaciones que coincidan                 │
   │  con los filtros seleccionados.                      │
   │                                                      │
   │  Intenta con filtros menos restrictivos              │
   │  o verifica que existan conversaciones               │
   │  importadas para esta combinación.                   │
   │                                                      │
   │  [Limpiar filtros]                                   │
   └─────────────────────────────────────────────────────┘


FLUJO — Agent Timeline sin resultados:
──────────────────────────────────────
1. Usuario selecciona agente con filtro de fecha muy restrictivo
2. GET /agents/{id}/timeline?from=2026-01-01&to=2026-01-07
3. API responde 200 con array vacío

4. UI muestra:
   ┌─────────────────────────────────────────────────────┐
   │  Agent: BE (Backend Engineer)                        │
   │  Período: 01 Ene 2026 — 07 Ene 2026                │
   │                                                      │
   │           📭                                         │
   │  Este agente no tiene actividad registrada           │
   │  en el período seleccionado.                         │
   │                                                      │
   │  [Limpiar filtros de fecha]                          │
   └─────────────────────────────────────────────────────┘
```

### EF-07 · Error de Red en Cualquier Pantalla

**Trigger:** Cualquier GET falla por timeout de red, CORS o servicio caído.  
**Pantalla:** Todas (UF-01 a UF-08)

```
UI muestra (overlay o inline según pantalla):
┌─────────────────────────────────────────────────────┐
│  ✕  Error de conexión                               │
│                                                      │
│  No se pudo conectar con Memory Service.             │
│  Verifica tu conexión de red y que el servicio       │
│  esté ejecutándose.                                  │
│                                                      │
│  [Reintentar]                                        │
└─────────────────────────────────────────────────────┘
```

**Comportamiento por pantalla:**
- **Dashboard:** Error reemplaza todo el contenido. Botón Reintentar recarga GET /dashboard/stats.
- **Listas (Conversations, Timeline):** Error inline debajo de los filtros. Lista previa permanece visible si había datos cargados.
- **Conversation Viewer:** Error en el body. Header con metadata puede seguir visible si venía de la navegación anterior.
- **Cost Reports:** Error reemplaza las tablas. Filtros de fecha permanecen visibles.
- **Health:** Error específico "No se puede conectar con el servicio" (ver EF-03).

### EF-08 · Conversación No Encontrada (404 genérico)

**Trigger:** GET /conversations/:id/content → ID no existe en BD.  
**Código HTTP:** 404  
**Pantalla:** Conversation Viewer (UF-03)

```
UI muestra:
┌─────────────────────────────────────────────────────┐
│  ✕  Conversación no encontrada                      │
│                                                      │
│  No existe una conversación con este identificador.  │
│  Puede haber sido eliminada o el enlace es           │
│  incorrecto.                                         │
│                                                      │
│  [Volver a la lista]                                 │
└─────────────────────────────────────────────────────┘
```

**Diferencia con EF-02:** EF-02 es cuando la conversación existe en BD pero el archivo no está en /storage/. EF-08 es cuando el ID no existe en absoluto en la BD.

## 5. Flujo Interno No Expuesto en UI

### Context Timeout (MEM-ERR-504)

**Trigger:** GET /context → queries exceden 500ms → HTTP 504.  
**Pantalla:** Ninguna. Este endpoint es consumido por Prompt Builder (A2), no por la UI.  
**UC:** UC-009 | **US:** US-012 | **BR:** BR-011

```
NO se muestra en la UI.
El error 504 es manejado por Prompt Builder:
- PB recibe { error: "MEM-ERR-504" }
- PB ensambla el prompt sin contexto histórico
- NO retry, NO degraded, NO cache fallback (D-MEM-07, D-MEM-37)

El Admin puede detectar estos timeouts en:
- Logs del servicio (console.error)
- Métricas de latencia (si se implementa monitoring futuro)
- Dashboard: indirectamente, si el contexto vacío produce agentes menos eficientes
```
