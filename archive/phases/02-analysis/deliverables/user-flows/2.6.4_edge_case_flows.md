# 2.6.4 — Edge Case Flows

**Proyecto:** Memory Service  
**Fase:** Analysis (Phase 4)  
**Tarea VTT:** MS-023  
**Autor:** UX Designer — `a75a1dae-754a-4b6f-a3ff-db8d51f6a91b`  
**Fecha:** 2026-05-06

---

## 1. Propósito

Documentar los casos borde (edge cases) que no son errores técnicos pero requieren tratamiento específico en la UI. Son estados válidos del sistema que podrían confundir al usuario si no se manejan explícitamente.

Los edge cases EC-01 a EC-05 son obligatorios per ASSIGNMENT MS-023. Los edge cases EC-06 a EC-10 cubren situaciones adicionales derivadas del análisis de los Use Cases y Business Rules.

---

## 2. Diferencia entre Error Flow y Edge Case

| Concepto | Error Flow (2.6.3) | Edge Case (este doc) |
|----------|--------------------|-----------------------|
| **Naturaleza** | Algo falló | Estado válido pero inusual |
| **HTTP** | 4xx, 5xx o network error | 200 OK con datos inesperados |
| **Ejemplo** | Archivo no existe en /storage/ | Agente sin costo porque solo tiene VTT_CHANNEL |
| **UI** | Banner rojo + acción de recuperación | Información contextual + explicación |

## 3. Edge Cases Obligatorios

### EC-01 · Lista de Conversaciones con Filtro Demasiado Restrictivo

**Pantalla:** Conversations List (UF-02)  
**Trigger:** Usuario aplica combinación de filtros que no matchea ninguna conversación.  
**Estado del sistema:** 200 OK, array vacío — no es error.

```
ESCENARIO:
──────────
El TL filtra por:
  Agente: QA · Proyecto: Memory Service · Fuente: CHATGPT · Status: IMPORTED

No existen conversaciones ChatGPT del QA en Memory Service.
GET /conversations?primaryAgentId=X&projectId=Y&sourceCode=CHATGPT → []

UI MUESTRA:
───────────
┌─────────────────────────────────────────────────────┐
│  Filtros activos: QA · Memory Service · CHATGPT      │
│                                                      │
│           📭                                         │
│  No hay conversaciones que coincidan                 │
│  con los filtros seleccionados.                      │
│                                                      │
│  Sugerencia: prueba eliminando el filtro de fuente   │
│  o ampliando el rango de fechas.                     │
│                                                      │
│  [Limpiar todos los filtros]                         │
└─────────────────────────────────────────────────────┘

DECISIÓN UX:
────────────
- Los filtros activos permanecen visibles arriba para que el usuario
  identifique cuál es demasiado restrictivo.
- El botón "Limpiar todos los filtros" resetea a defaults (status=IMPORTED, sin otros filtros).
- NO se muestra como error — es resultado válido de una búsqueda sin matches.
```

### EC-02 · Agente Sin Conversaciones SDK (Cost Report $0.00)

**Pantalla:** Agent Cost Report (UF-06), Project Cost Report (UF-05)  
**Trigger:** El agente solo tiene conversaciones CLAUDE_WEB, CHATGPT o VTT_CHANNEL — ninguna con costo real del SDK.  
**Estado del sistema:** 200 OK con totalCostUsd = 0.00. No es error — es dato válido.  
**BR:** BR-014 (costo del SDK, no recalculado), BR-016 (solo IMPORTED)

```
ESCENARIO:
──────────
El PM consulta el cost report del agente "UX Designer".
Este agente solo tiene conversaciones importadas manualmente (CLAUDE_WEB)
y participaciones en reviews (VTT_CHANNEL).
Ninguna conversación tiene totalCostUsd del SDK.

GET /agents/{ux-uuid}/cost-report → { totalCostUsd: 0.00, totalConversations: 12 }

UI MUESTRA:
───────────
┌─────────────────────────────────────────────────────┐
│  UX Designer — Cost Report                           │
│                                                      │
│  Total Conversaciones:  12                           │
│  Total Tokens:          0                            │
│  Costo Total:           $0.00 USD                    │
│                                                      │
│  ℹ️  Este agente no tiene conversaciones con costo    │
│  real del SDK. Solo las conversaciones importadas     │
│  via Runtime (CLAUDE_SDK) incluyen datos de costo.    │
│  Las fuentes manuales (CLAUDE_WEB, CHATGPT) y los    │
│  canales VTT muestran $0.00.                         │
│                                                      │
│  Desglose por Proyecto:                              │
│  Memory Service  │  12 convs  │  $0.00               │
└─────────────────────────────────────────────────────┘

DECISIÓN UX:
────────────
- Se muestra el reporte completo con $0.00 — NO se oculta ni se muestra error.
- La nota informativa explica POR QUÉ el costo es $0.00 para que el PM no piense
  que hay un bug.
- Las conversaciones sin costo aparecen en el conteo pero con $0.00 (BR-014).
```

### EC-03 · Archivo JSONL Corrompido en /storage/

**Pantalla:** Conversation Viewer (UF-03)  
**Trigger:** GET /content → archivo existe en /storage/ pero su contenido no es JSONL válido (parseo falla).  
**Estado del sistema:** El archivo se escribió pero está corrupto o incompleto.

```
ESCENARIO:
──────────
Una conversación tiene status IMPORTED en BD y el archivo existe en /storage/,
pero el contenido del JSONL está truncado o corrupto (ej: escritura interrumpida).

GET /conversations/{id}/content → el backend intenta parsear → falla

Dos posibilidades técnicas:
a) El backend retorna 500 (error interno de parseo)
b) El backend retorna 200 con contenido parcial/raw

UI MUESTRA (opción recomendada — tratar como error de contenido):
─────────────────────────────────────────────────────────────────
┌─────────────────────────────────────────────────────┐
│  [Header con metadata — se muestra normalmente]      │
│                                                      │
│  ⚠  El contenido de esta conversación no pudo        │
│  cargarse correctamente.                             │
│                                                      │
│  El archivo puede estar dañado o incompleto.         │
│  La metadata de la conversación es válida pero       │
│  el contenido detallado no está disponible.          │
│                                                      │
│  [Volver a la lista]                                 │
└─────────────────────────────────────────────────────┘

DECISIÓN UX:
────────────
- El header con metadata (de BD) se muestra normalmente — esa información es válida.
- Solo el body con los turns muestra el error.
- Se usa ⚠ amarillo (degradado), no ✕ rojo, porque la conversación SÍ existe y la
  metadata es correcta — solo el contenido detallado falla.
- Diferencia con EF-02: EF-02 es archivo no encontrado (404), EC-03 es archivo
  encontrado pero no parseable.
```

### EC-04 · Dashboard con 0 Conversaciones (Primera Ejecución)

**Pantalla:** Dashboard (UF-01)  
**Trigger:** El sistema acaba de desplegarse. No hay ninguna conversación importada.  
**Estado del sistema:** 200 OK con todos los contadores en 0. Estado válido e inicial.

```
ESCENARIO:
──────────
Memory Service se desplegó en producción.
El Admin verificó que el health es OK.
El PM abre la UI por primera vez.

GET /dashboard/stats → {
  totalConversations: { IMPORTED: 0, PENDING: 0, PROCESSING: 0, ERROR: 0 },
  bySource: {},
  byType: {},
  activeAgents: 0,
  recentActivity: []
}

UI MUESTRA:
───────────
┌─────────────────────────────────────────────────────┐
│  MEMORY SERVICE — Dashboard                         │
│                                                      │
│  Total: 0    Agentes activos: 0                     │
│                                                      │
│           🚀                                         │
│  ¡Bienvenido a Memory Service!                      │
│                                                      │
│  Aún no hay conversaciones importadas.              │
│  Hay dos formas de empezar:                         │
│                                                      │
│  1. Espera a que Runtime importe la primera sesión   │
│     de un agente automáticamente.                    │
│                                                      │
│  2. Sube una conversación manualmente desde          │
│     un export de claude.ai o ChatGPT.                │
│                                                      │
│  [Subir conversación manualmente]                    │
└─────────────────────────────────────────────────────┘

DECISIÓN UX:
────────────
- NO es error — es onboarding. Se muestra con tono positivo (🚀) y guía al usuario.
- Las cards de métricas se muestran con 0 pero no vacías.
- Los gráficos de breakdown (bySource, byType) se ocultan — no tiene sentido mostrar
  gráficos vacíos. Aparecen cuando hay al menos 1 conversación.
- El CTA principal dirige al Upload manual.
```

### EC-05 · Upload de Archivo Duplicado → ALREADY_INDEXED

**Pantalla:** Manual Upload (UF-07)  
**Trigger:** Usuario sube un archivo que ya fue importado previamente (mismo sourceId + externalSessionId).  
**Estado del sistema:** 200 OK con status ALREADY_INDEXED. No es error — es idempotencia (BR-001).  
**AMB-UX-02**

```
ESCENARIO:
──────────
El SA subió "conversation_2026-05-06.json" hace 1 hora.
Ahora vuelve a subirlo (olvidó que ya lo había hecho o quiere verificar).

POST /upload → idempotencia detecta @@unique([sourceId, externalSessionId])
→ { status: "ALREADY_INDEXED", conversationId: "abc-123" }

UI MUESTRA:
───────────
┌─────────────────────────────────────────────────────┐
│  ℹ️  Esta conversación ya fue importada              │
│                                                      │
│  La conversación existe en el sistema desde          │
│  2026-05-06 a las 10:15.                             │
│                                                      │
│  No se creó duplicado — el sistema detectó que       │
│  es la misma conversación automáticamente.           │
│                                                      │
│  [Ver conversación existente]  [Subir otro archivo]  │
└─────────────────────────────────────────────────────┘

DECISIÓN UX (AMB-UX-02):
─────────────────────────
- Banner AZUL (info), NO rojo. Reimportar no es error del usuario.
- Se muestra cuándo se importó originalmente para dar contexto.
- Link directo al Conversation Viewer de la conversación existente.
- Botón alternativo para subir otro archivo si ese no era el que buscaba.
- La idempotencia es una feature, no un fallo — el tono del mensaje es informativo.
```

## 4. Edge Cases Complementarios

### EC-06 · Conversación en PROCESSING Visible en Lista

**Pantalla:** Conversations List (UF-02)  
**Trigger:** Usuario cambia el filtro de status a "PROCESSING" y ve conversaciones en proceso.

```
ESCENARIO:
──────────
El TL selecciona filtro Status: PROCESSING.
GET /conversations?statusCode=PROCESSING → 2 resultados

UI MUESTRA:
───────────
Cada card de conversación en PROCESSING muestra:

  ● "Import de sesión BE" · CLAUDE_SDK · ⏳ Procesando
    Hace 2 minutos · Sin contenido disponible aún

  ● "Import de sesión FE" · CLAUDE_SDK · ⏳ Procesando
    Hace 8 minutos · Sin contenido disponible aún

DECISIÓN UX:
────────────
- Las cards muestran un badge "⏳ Procesando" en lugar de métricas.
- No son clickeables al Conversation Viewer (no hay contenido disponible en /storage/).
- Si updatedAt > 10 min → mostrar badge adicional "⚠ En recuperación" (el cleanup
  cron debería estar reintentándolas).
- El status PROCESSING es transitorio — el default de la lista es IMPORTED (RF-020).
  El usuario debe seleccionar explícitamente PROCESSING para ver estos registros.
```

### EC-07 · Agente Solo con Participaciones Multi-Agente

**Pantalla:** Agent Timeline (UF-04)  
**Trigger:** Un agente nunca fue primaryAgent — solo participó en AGENT_REVIEW/CLARIFICATION.

```
ESCENARIO:
──────────
El TL busca el timeline del agente "Integration Reviewer".
Este agente solo existe como participante en ConversationParticipant,
nunca como primaryAgentId en Conversation.

GET /agents/{ir-uuid}/timeline → resultados solo tipo AGENT_REVIEW

UI MUESTRA:
───────────
Timeline con todas las entries mostrando badge "Participante":

  2026-05-06
  │
  ├─ 14:00  ● AGENT_REVIEW · VTT_CHANNEL
  │         Badge: Participante
  │         "Review contratos de integración"
  │
  ├─ 10:30  ● AGENT_CLARIFICATION · VTT_CHANNEL
  │         Badge: Participante
  │         "Clarificación sobre SERVICE_KEY"

No se muestra ningún badge "Ejecutor" porque no hay TASK_EXECUTION.

DECISIÓN UX:
────────────
- Es un timeline válido — no mostrar aviso especial.
- Los filtros de tipo funcionan normalmente (si filtra por TASK_EXECUTION → EC-01 lista vacía).
- La ausencia de entries "Ejecutor" es dato, no error.
```

### EC-08 · Cost Report con Mezcla de Fuentes (Con y Sin Costo)

**Pantalla:** Project Cost Report (UF-05)  
**Trigger:** Proyecto tiene conversaciones SDK (con costo) y conversaciones manuales (sin costo).

```
ESCENARIO:
──────────
Project Memory Service tiene:
  - 45 conversaciones CLAUDE_SDK → totalCostUsd = $4.56
  - 18 conversaciones CLAUDE_WEB → totalCostUsd = null ($0.00)
  - 8 conversaciones CHATGPT → totalCostUsd = null ($0.00)
  - 26 conversaciones VTT_CHANNEL → totalCostUsd = null ($0.00)

Total: 97 conversaciones, $4.56 USD.

UI MUESTRA:
───────────
  Total Conversaciones: 97
  Costo Total: $4.56 USD

  Desglose por Fuente:
  Fuente       │ Convs │ Costo USD │ Nota
  ─────────────┼───────┼───────────┼──────────────
  CLAUDE_SDK   │  45   │ $4.56     │
  CLAUDE_WEB   │  18   │ $0.00     │ Sin costo SDK
  CHATGPT      │   8   │ $0.00     │ Sin costo SDK
  VTT_CHANNEL  │  26   │ $0.00     │ Sin costo SDK

  ℹ️  97 conversaciones totales. Solo las 45 importadas via
  Runtime (CLAUDE_SDK) incluyen datos de costo real.

DECISIÓN UX:
────────────
- Las conversaciones sin costo SDK se incluyen en el conteo pero con $0.00 (BR-014).
- La columna "Nota" indica cuáles fuentes no tienen costo.
- El total refleja solo el costo real SDK.
- NO se excluyen las fuentes sin costo del reporte — el PM necesita ver el volumen
  completo de conversaciones aunque no todas tengan costo.
```

### EC-09 · JSONL Muy Grande Causa Carga Lenta en Viewer

**Pantalla:** Conversation Viewer (UF-03)  
**Trigger:** GET /content de una conversación con muchos turns (>100) o tool_use blocks muy extensos.  
**D-MEM-43:** Contenido se lee de /storage/ en runtime — archivos grandes tardan más.

```
ESCENARIO:
──────────
Una sesión CLAUDE_SDK del agente BE tiene 85 turns con múltiples tool_use
blocks de código. El archivo JSONL pesa ~2MB.

GET /conversations/{id}/content → respuesta tarda 3-5 segundos.

UI MUESTRA (durante la carga):
──────────────────────────────
┌─────────────────────────────────────────────────────┐
│  [Header con metadata — se carga inmediato de BD]    │
│                                                      │
│  Title: "Implementar módulo de clasificación"        │
│  85 turns · 234,567 tokens · $2.34                   │
│                                                      │
│  ─────────────────────────────────────────────       │
│                                                      │
│  ▓▓▓░░░░░░░  Cargando contenido...                  │
│                                                      │
│  [skeleton placeholders para turns]                   │
└─────────────────────────────────────────────────────┘

Después de 3-5 segundos → se renderizan los 85 turns.

DECISIÓN UX:
────────────
- El header con metadata se carga inmediatamente (dato de BD, rápido).
- El body con turns usa un skeleton loader mientras espera el JSONL de /storage/.
- Si la carga excede 10 segundos, mostrar mensaje:
  "El contenido es extenso y está tardando en cargarse. Por favor espera."
- NO implementar paginación de turns en R1 — se muestra todo.
- Considerar lazy rendering: renderizar los primeros 20 turns y cargar el resto
  on-scroll (decisión de implementación FE, no de UX spec).
```

### EC-10 · Upload Doble Rápido (Mismo Archivo Dos Veces en Segundos)

**Pantalla:** Manual Upload (UF-07)  
**Trigger:** Usuario hace double-click en "Subir" o sube el mismo archivo inmediatamente después de la primera subida.

```
ESCENARIO:
──────────
El SA click "Subir conversación" y inmediatamente click de nuevo (double-click).
Dos POST /upload salen en paralelo con el mismo archivo.

Request A → create → éxito → PENDING → PROCESSING → IMPORTED
Request B → create → P2002 (race condition) → ALREADY_INDEXED

UI MUESTRA:
───────────
La UI recibe la primera respuesta (Request A: IMPORTED) y muestra éxito.
Si la segunda respuesta llega después, la UI la ignora (el resultado
ya se mostró).

DECISIÓN UX:
────────────
- Deshabilitar el botón "Subir" después del primer click (prevención de double-submit).
- Mostrar spinner inmediatamente para indicar que la acción está en curso.
- Si por alguna razón llegan dos respuestas, la UI usa solo la primera.
- El sistema backend maneja esto correctamente via P2002 → ALREADY_INDEXED (UC-021).
- La prevención es de UX (botón disabled), no de backend.
```
