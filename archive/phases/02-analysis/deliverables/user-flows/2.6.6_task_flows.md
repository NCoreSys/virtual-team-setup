---

## 1. Propósito

Los Task Flows documentan la secuencia exacta de interacciones que un usuario realiza para completar una tarea concreta. A diferencia de los Happy Paths (2.6.2) que narran escenarios de principio a fin, y de los Journey Maps (2.6.5) que mapean la experiencia emocional, los Task Flows son instrucciones paso a paso con los clics, datos ingresados y respuestas del sistema en cada acción.

---

## 2. Convención

Cada task flow usa este formato:

```
[N] ACCIÓN del usuario
    → SISTEMA responde
    → UI muestra / cambia
```

## 3. Task Flows del TL

### TF-01 · Verificar qué hizo un agente en una tarea específica

**Actor:** TL  
**Pregunta:** "¿Qué hizo el agente DB en la tarea MEM-048?"  
**Pantallas:** UF-02 → UF-03  
**Duración estimada:** 30 segundos

```
[1] Click en navegación → "Conversations"
    → GET /conversations (default: status=IMPORTED, limit=20)
    → UI muestra Conversations List con las últimas 20 conversaciones

[2] En panel de filtros, seleccionar Agente: "DB"
    → Filtro aplicado visualmente (chip o tag)

[3] En panel de filtros, escribir Tarea: "MEM-048"
    → Filtro aplicado visualmente

[4] Click botón "Aplicar filtros"
    → GET /conversations?primaryAgentId={db-uuid}&taskId={mem048-uuid}
    → UI muestra N resultados filtrados

[5] Leer contentPreview de cada resultado para identificar la sesión relevante
    → Cada card muestra: title, sourceCode, startedAt, turnCount, 
      tokens, topics[], y contentPreview (500 chars)

[6] Click en la conversación deseada
    → GET /conversations/{id}/content
    → UI navega a Conversation Viewer
    → Header: metadata completa
    → Body: turns USER → ASSISTANT con tool_use expandibles

[7] Leer el contenido. Opcionalmente expandir tool_use blocks para ver código.

[8] Click "← Volver"
    → UI regresa a Conversations List con filtros preservados
```

### TF-02 · Auditar la actividad semanal de un agente

**Actor:** TL  
**Pregunta:** "¿Cuántas sesiones tuvo BE esta semana y en qué trabajó?"  
**Pantallas:** UF-04  
**Duración estimada:** 20 segundos

```
[1] Click en navegación → "Timeline"
    → UI muestra Agent Timeline con selector de agente

[2] Seleccionar agente: "BE (Backend Engineer)" del dropdown
    → GET /agents/{be-uuid}/timeline?limit=20
    → UI muestra timeline cronológico

[3] En filtros, seleccionar rango de fechas:
    Desde: "2026-05-01"  Hasta: "2026-05-06"
    → Click "Aplicar"
    → GET /agents/{be-uuid}/timeline?from=2026-05-01&to=2026-05-06
    → UI actualiza timeline con entries solo de esa semana

[4] Contar entries visibles y leer titles/topics para entender el trabajo
    → Cada entry muestra: hora, tipo (TASK_EXECUTION/REVIEW),
      sourceCode, title, proyecto, tarea, turnCount, costUsd, topics[]
    → Badge "Ejecutor" o "Participante" según tipo

[5] Si necesita ver detalle de alguna sesión:
    → Click en entry → navega a Conversation Viewer (UF-03)
    → "← Volver" regresa al timeline filtrado
```

### TF-03 · Revisar una sesión de review multi-agente

**Actor:** TL  
**Pregunta:** "¿Qué se discutió en el review del schema de BD?"  
**Pantallas:** UF-04 → UF-03  
**Duración estimada:** 45 segundos

```
[1] En Agent Timeline (UF-04), localizar entry tipo AGENT_REVIEW
    → Identificable por: badge "Participante", sourceCode "VTT_CHANNEL",
      tipo "AGENT_REVIEW", indicador "N participantes"

[2] Click en la entry
    → GET /conversations/{review-id}/content
    → Archivo leído desde /storage/_reviews/{review-id}.jsonl (BR-004)
    → UI navega a Conversation Viewer (UF-03)

[3] Viewer detecta tipo AGENT_REVIEW y renderiza como thread:
    → Header: title, tipo, participantes listados, N mensajes
    → Body: mensajes cronológicos con:
      - Badge del participante (agente que envió)
      - Timestamp
      - messageType (REVIEW_REQUEST, REVIEW_RESPONSE, COMMENT, etc.)
      - Priority si existe
      - Contenido del mensaje

[4] Leer el thread de arriba a abajo para seguir la discusión

[5] Click "← Volver"
    → Regresa a Agent Timeline
```

### TF-05 · Investigar una conversación en estado ERROR

**Actor:** TL  
**Pregunta:** "¿Por qué hay conversaciones en ERROR en el dashboard?"  
**Pantallas:** UF-01 → UF-02  
**Duración estimada:** 30 segundos

```
[1] En Dashboard (UF-01), observar card "Total Conversaciones"
    → ERROR: 9

[2] Click en navegación → "Conversations"
    → UI muestra Conversations List

[3] En filtros, cambiar Status de "IMPORTED" a "ERROR"
    → Click "Aplicar"
    → GET /conversations?statusCode=ERROR
    → UI muestra las 9 conversaciones en ERROR

[4] Revisar cada card:
    → Cada conversación en ERROR muestra:
      title, sourceCode, startedAt, errorMessage
      (ej: "Max retries exceeded after 3 attempts"
       o "Storage file missing after processing")

[5] Identificar patrón:
    → Si todas dicen "Storage file missing" → problema de disco
    → Si dicen "Max retries exceeded" → error transitorio recurrente
    → Comunicar hallazgo al Admin/DO
```

### TF-10 · Verificar estado general del sistema

**Actor:** PM  
**Pregunta:** "¿Cómo está el sistema? ¿Hay errores pendientes?"  
**Pantallas:** UF-01  
**Duración estimada:** 10 segundos

```
[1] Abrir Memory Service UI (URL http://{host}:3003)
    → GET /dashboard/stats
    → UI muestra Dashboard

[2] Revisar cards de métricas:
    → Total Conversaciones: N (por status: IMPORTED / PENDING / PROCESSING / ERROR)
    → Agentes Activos (últimos 30 días): N
    → Por Fuente: breakdown de 5 fuentes
    → Por Tipo: breakdown de 3 tipos

[3] Si ERROR > 0 → investigar (ver TF-05)
    Si PROCESSING > 0 y son recientes → normal, esperar cleanup
    Si todo IMPORTED → sistema operando correctamente

[4] Revisar "Actividad Reciente" para confirmar que hay imports frescos
    → Si la última actividad es de hace >1 hora y el Runtime está activo
      → posible problema de integración
```

## 5. Task Flows del Admin

### TF-11 · Verificar health antes de un deploy

**Actor:** Admin / DO  
**Pregunta:** "¿Puedo hacer deploy del Runtime sin problemas?"  
**Pantallas:** UF-08  
**Duración estimada:** 10 segundos

```
[1] Click en navegación → "Health"
    → GET /health
    → UI muestra Health Status

[2] Verificar los 3 semáforos:
    → BD (PostgreSQL): ● OK
    → Storage (/storage/): ● OK
    → Redis: ● OK

[3] Si todos ● verdes → proceder con deploy
    Si alguno ⚠ amarillo → evaluar riesgo antes de deploy
    Si alguno ✕ rojo → NO hacer deploy, resolver primero

[4] Click "Refresh" para confirmar que el estado es estable
    → GET /health (nueva request)
    → Verificar que sigue igual
```

---

### TF-12 · Diagnosticar health degradado

**Actor:** Admin / DO  
**Pregunta:** "Redis está caído, ¿qué impacto tiene?"  
**Pantallas:** UF-08  
**Duración estimada:** 15 segundos

```
[1] En Health Status (UF-08), ver:
    → Estado: ⚠ DEGRADED
    → BD: ● OK
    → Storage: ● OK
    → Redis: ✕ DOWN

[2] Evaluar impacto según componente:
    
    Si Redis DOWN:
    → Impacto: rate limiting y cache deshabilitados
    → El servicio sigue operativo pero sin cache
    → Acción: verificar contenedor Redis, reiniciar si necesario
    
    Si Storage DOWN:
    → Impacto: GET /content falla, imports nuevos fallan en escritura
    → Acción: verificar espacio en disco, bind mount, permisos
    
    Si BD DOWN:
    → Impacto: todo el servicio inoperativo
    → Acción: verificar PostgreSQL, conexiones, espacio en disco

[3] Click "Refresh" periódicamente para verificar si se recuperó
    → Auto-refresh cada 60s también actualiza
```
